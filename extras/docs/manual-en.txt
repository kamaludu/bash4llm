================================================================================
BASH4LLM(1)                      Bash4LLM Manual                    BASH4LLM(1)
================================================================================

NAME
    bash4llm - Resilient and secure Bash-first wrapper for querying external 
    LLM providers (v2.4.0)

SYNOPSIS
    bash4llm [OPTIONS] [PROMPT]
    bash4llm -f FILE [OPTIONS]
    echo "prompt" | bash4llm [OPTIONS]
    bash4llm --batch BATCH_FILE [OPTIONS]

DESCRIPTION
    bash4llm is a native shell-environment command-line utility designed for 
    flexible integration, automation, and orchestration of interactions with 
    large language models (LLMs) provided by external services such as Groq, 
    Gemini, and HuggingFace.

    The runtime operates under restrictive execution constraints (set -euo 
    pipefail) to ensure the immediate termination of pipelines when encountering 
    invalid commands or uninitialized variables. The architecture is entirely 
    designed to isolate system operations from remote execution, prohibiting 
    any dynamic interpretation or local execution of generated code (zero eval 
    policy) and excluding the use of shared system directories like /tmp to 
    mitigate symlink attacks (symlink attack mitigation).

OPTIONS
    Script options are grouped by functional category. Parameters can be passed 
    in any arbitrary order, except for the end-of-options marker.

  Section 1: Input and Prompt Management
    -f FILE
        Adds the specified file as a context source for processing. Can be 
        specified multiple times to concatenate multiple input files 
        sequentially. Individual files are separated by structured visual 
        demarcators to allow the model to clearly distinguish sources.

    --json-input FILE
        Sends a structured JSON payload, useful for directly defining a list of 
        custom messages with specific roles. If set to '-', the script reads 
        from standard input (stdin). Using this option excludes and clears the 
        generic prompt variable resolution.

    --template NAME
        Indicates the text template to apply to the user prompt, sourced from 
        the user template directory. The template file must contain the 
        placeholder "{{CONTENT}}" which is replaced at runtime with the 
        assembled user input text.

    --batch FILE
        Activates batch processing mode. The script parses the specified file 
        and executes a distinct API call for each non-empty line, excluding 
        comments (lines starting with a hash symbol). An error in a single 
        prompt does not halt the overall batch sequence.

    PROMPT (Positional Arguments)
        All positional arguments not captured by specific flags are treated as 
        text fragments. If an argument corresponds to a regular, readable file 
        path on disk, the script expands it by reading and incorporating its 
        content; otherwise, it interprets it as literal text to append to the 
        main prompt.

  Section 2: Model and Generation Parameters
    -m, --model MODEL
        Specifies the name of the LLM model to query. This setting takes 
        absolute precedence over locally saved default models.

    --system TEXT
        Defines or overrides the system prompt, which instructs or conditions 
        the behavior, tone, and operational context of the language model.

    --temperature VALUE, --ture VALUE
        Sets the decimal sampling temperature (usually between 0.0 and 2.0). 
        Both flags link and synchronize bidirectionally to the internal 
        variables "TURE" and "TEMPERATURE", unifying their internal resolution 
        before transmission.

    --max NUMBER
        Sets the maximum limit of tokens to generate in the model's response 
        (internal variable "MAX_TOKENS").

  Section 3: Thread and Conversation Management
    --thread THREAD_ID
        Enables multi-turn chat context persistence by associating the history 
        with a unique alphanumeric identifier. The ID is formally validated to 
        exclude directory traversal or unauthorized characters. The ID must 
        match the regex ^[A-Za-z0-9._-]{1,128}$.

    --thread-window [WINDOW]
        Specifies the maximum number of previous historical messages to retrieve 
        from the thread history file to compose the context window. If the 
        flag is specified without arguments, the default value is set to 10. 
        To optimize performance and avoid saturating the model's context, 
        values less than or equal to 20 are recommended; setting values greater 
        than 20 causes the runtime to emit a diagnostic warning.

    --init-thread (Requires --thread <id>)
        Allows external components (such as a CGI graphical interface) or 
        automation scripts to pre-register a new thread in the system before 
        sending the first prompt. This option performs the following local 
        operations:
        - Creates an empty history file (<id>.ndjson) with restrictive 
          permissions.
        - Generates the thread metadata file (<id>.json) with a placeholder 
          title and zero message count.
        - Updates the global thread index (index.json) under an exclusive lock 
          to prevent race conditions.
        - Terminates execution immediately (exit 0) without making network 
          connections or requiring API keys.

  Section 4: Output and Formatting
    --json
        Disables text extraction and displays the raw JSON response payload 
        from the provider exactly as received from the external API.

    --pretty
        Performs structured formatting (pretty-print) on the JSON response 
        before displaying it on standard output.

    --text
        Selects standard text display mode (default). The script extracts only 
        the textual message body produced by the model, removing the surrounding 
        JSON structures.

    --raw
        Returns only the response text, excluding any additional trailing 
        newline characters.

    --out PATH
        Saves the final response to the specified path. If the path points to 
        an existing directory, the script generates a unique file named with 
        a chronological timestamp inside it; if it points to a file, it 
        overwrites it transactionally and securely.

    --threshold NUMBER
        Sets the threshold limit, expressed in text characters (default 1000 
        characters), above which the response produced by the LLM is 
        automatically and permanently saved in the interaction history directory.

    --save
        Forces unconditional saving of the textual result to the history 
        directory, ignoring the threshold limit check (FORCE_SAVE_MODE="save").

    --nosave
        Absolutely disables automatic archiving of the result in the 
        interaction history (FORCE_SAVE_MODE="nosave").

  Section 5: Connection Flows and Network
    --stream
        Enables real-time token reception (data streaming). Requires the active 
        provider to support the SSE (Server-Sent Events) connection interface. 
        This option is mutually exclusive and incompatible with structured 
        --json and --pretty formats; if used together, the script blocks 
        execution.

    --no-stream
        Explicitly inhibits and disables token streaming mode, restoring 
        synchronous, blocking reception of the entire response.

    --dry-run
        Simulates the transaction without making real network connections. The 
        script compiles the final JSON request payload structure, displays its 
        first few lines (head) on standard error for diagnostic purposes, and 
        terminates execution with a success status.

  Section 6: Module Selection and Installation
    --provider [NAME]
        If specified with a name (excluding strings starting with a hyphen), 
        selects and sets the active provider, persistently saving the preference 
        in the user configuration file. If the passed value is the literal 
        string "list", or if the flag is specified without any argument in an 
        interactive terminal (TTY), the script opens an interactive selection 
        menu listing all installed providers and allowing the user to select 
        the default.

    --install-extras [SOURCE]
        Installs and extends the runtime by copying modules and resources 
        from the extensions directory (e.g., external providers, session 
        engine scripts, thread resources, tests, documentation) into the 
        persistent user extensions folder. If not specified, the script 
        attempts to locate the sources under the "extras" origin folder 
        relative to the script's location. The module performs strict security 
        checks to prevent copying symbolic links or installing into 
        unauthorized nested paths.

  Section 7: Model Management and Quick Configuration
    --refresh-models, --refresh-model
        Forces an immediate connection with the active provider's API to 
        synchronize the local list of available models, saving it atomically 
        and securely under an exclusive lock in the models file.

    --list-models
        Displays a formatted, readable list of text models currently available 
        and ready for use in the local system.

    --list-models-raw
        Returns the raw list of local models in machine format (one normalized 
        identifier per line) for automation purposes.

    --list-providers
        Displays the list of provider modules (both built-in and extended via 
        plugins) installed in the system.

    --list-providers-raw
        Returns the list of installed providers in machine format, without 
        formatting or text headers.

    --set-default MODEL
        Sets and permanently archives the specified model as the default choice 
        for the currently active provider in the system.

  Section 8: Diagnostics and Maintenance
    --quiet
        Runtime silencer. Inhibits all informational prints, tracking messages, 
        and non-critical warnings on standard error, limiting output to only 
        the formal results of the query or blocking execution errors.

    --debug
        Enables advanced diagnostics mode for developers. Writes detailed log 
        messages to standard error and, if configured, to the log destination. 
        Additionally, it inhibits the automatic removal of resources allocated 
        in the instance's temporary directory when the script terminates.

    --show-config
        Generates a structured tabular view of current runtime parameters 
        (software versions, release dates, selected provider, active model, 
        absolute paths of all operational directories in the file system, 
        debug flag states, and synchronization file sizes).

    --diagnostics
        Starts a complete self-check and security requirements inspection cycle. 
        Verifies the physical existence of user folders, the resolution of 
        write permissions, the syntactic validity of imported provider modules, 
        and the presence of the correct API key for the active provider.

    --version
        Prints the script's identifier name, current software version, release 
        date, and terminates execution.

    -h, --help
        Displays the quick start interface guide and terminates.

    --
        End-of-options marker. Any argument positioned after this delimiter is 
        fully interpreted as the input prompt text, excluding subsequent flag 
        processing.

ENVIRONMENT VARIABLES
    The behavior of the bash4llm runtime is governed and can be conditioned by 
    the following environment variables.

  Section 1: API Authentication Credentials
    GROQ_API_KEY
        Default API key for authenticating network calls to the built-in Groq 
        provider.

    PROVIDER_API_ENV_[PROVIDER]
        Indirect mapping for external providers. Allows defining the name of 
        an alternative environment variable intended to contain the API key for 
        a given custom provider (e.g., setting PROVIDER_API_ENV_gemini="CUSTOM_KEY", 
        the script will look for the API key inside the CUSTOM_KEY variable).

  Section 2: Paths and File System Structure
    BASH4LLM_DIR
        Persistent root directory for bash4llm resources and data. If omitted, 
        it is dynamically resolved pointing to the "bash4llm.d" directory 
        located under the script's source folder.

    BASH4LLM_ROOT
        Alternative path used to derive the root directory if BASH4LLM_DIR is 
        not explicitly declared in the environment.

    BASH4LLM_CONFIG_DIR
        Directory intended to store persistent user settings 
        (default: $BASH4LLM_DIR/config).

    BASH4LLM_MODELS_DIR
        Directory containing the list of configured and downloaded models 
        (default: $BASH4LLM_DIR/models).

    BASH4LLM_TEMPLATES_DIR
        Directory containing user template files 
        (default: $BASH4LLM_DIR/templates).

    BASH4LLM_HISTORY_DIR
        Directory intended to accumulate the response history and interaction 
        logs (default: $BASH4LLM_DIR/history).

    BASH4LLM_TMPDIR
        Protected root directory for allocating temporary script data 
        (default: $BASH4LLM_DIR/tmp).

    BASH4LLM_EXTRAS_DIR
        Active directory for accessing and importing user extensions 
        (default: $BASH4LLM_DIR/extras).

    PROVIDERS_DIR
        Directory containing installed external provider modules 
        (default: $BASH4LLM_EXTRAS_DIR/providers).

    THREAD_CACHE_DIR
        Directory intended to host the raw temporary cache of thread messages 
        and contexts (default: $BASH4LLM_CONFIG_DIR/thread_cache).

  Section 3: Resource Configuration and Synchronization
    MODELS_FILE
        Absolute or relative path of the local file in which to store the list 
        of models retrieved from the API (default: $BASH4LLM_MODELS_DIR/models.txt).

    MAX_MODELS
        Numeric limit of models manageable locally in the models file. If the 
        provider returns a greater number of models, the excess is truncated to 
        avoid memory saturation (default: 200).

    PROVIDER_FILE
        Path of the file that persists the name of the last active provider set 
        by the user (located under the configuration directory).

    MODELS_LOCK
        Exclusive synchronization file (lock) to regulate access and updates to 
        the local models file (default: $BASH4LLM_MODELS_DIR/models.lock).

    HISTORY_LOCK
        Exclusive lock file to prevent physical corruption due to parallel or 
        concurrent access to the historical directory and threads 
        (default: $BASH4LLM_HISTORY_DIR/history.lock).

    TMP_LOCK
        Exclusive lock file associated with the temporary directory to regulate 
        the creation of instance subfolders (default: $BASH4LLM_TMPDIR/tmp.lock).

    BASH4LLM_LOCK_TIMEOUT_TMP
        Maximum wait timeout expressed in seconds to acquire the lock on 
        temporary files (default: 10 seconds).

    BASH4LLM_LOCK_TIMEOUT_MODELS
        Maximum timeout expressed in seconds to acquire the lock on the models 
        file (default: 10 seconds).

    BASH4LLM_LOCK_TIMEOUT_HISTORY
        Maximum timeout expressed in seconds to acquire the lock to access 
        history and threads (default: 10 seconds).

  Section 4: History Maintenance (Rotation)
    BASH4LLM_ROTATE_HISTORY
        Boolean flag (1 or 0) to enable automatic periodic purging and 
        maintenance of history logs after queries (default: 0).

    BASH4LLM_HISTORY_MAX_FILES
        Maximum numeric threshold of files storable in the response history 
        directory (default: 100). Once the threshold is exceeded, older files 
        are removed.

    BASH4LLM_HISTORY_MAX_BYTES
        Cumulative limit threshold expressed in bytes of the total space occupied 
        by history files (default: 104857600 bytes, equivalent to 100MB).

    BASH4LLM_HISTORY_KEEP_DAYS
        Time retention limit expressed in days for log and history files 
        (default: 90 days). Files older than the limit are deleted.

  Section 5: Parameters and Runtime Execution Control
    ALLOW_API_CALLS
        External transaction enabling flag. If set to a value other than 1 
        (e.g., 0), and if the script is not operating in dry-run simulation 
        mode, the presence of a valid API key generates a fatal error, blocking 
        network transmission for precautionary purposes.

    BASH4LLM_SKIP_NETWORK
        Inhibits and excludes any network call. If set to 1, forces the API 
        call to produce a dummy diagnostic response simulating dry-run.

    BASH4LLM_ENFORCE_NO_NETWORK_IF_QUIET
        If set to 1, combined with quiet mode enabling (QUIET=1), imposes 
        immediate blocking of all external connections for secure automation 
        purposes in the absence of a TTY.

    MAX_STAGE_BYTES
        Dimensional security threshold expressed in bytes applied to the transfer 
        and staging of external resources encoded in Base64 
        (default: 10485760 bytes, equivalent to 10MB). Files larger than this 
        are blocked before encoding to avoid resource saturation.

    LAST_CHECK_LINES
        Number of standard lines to scan backward in the thread NDJSON history 
        file during record extraction (default: 50 lines).

    DEBUG
        Diagnostic alignment variable. If set to 1, activates internal debug 
        features (inherits configurations from BASH4LLM_DEBUG).

    BASH4LLM_LOG
        Optional path of a file in which to append the unified operational logs 
        of the script with UTC timestamps in a structured chronological way.

    BASH4LLM_SOURCE_ONLY
        Development guard variable. If set to 1 before importing the script, 
        terminates main execution immediately after bootstrap. Allows loading 
        and testing only environment variables, definitions, and functions of 
        bash4llm in an active shell or external unit tests.

EXIT CODES
    The runtime adopts a set of fixed numeric exit codes to isolate failure 
    causes at the process level:

    10 (BASH4LLM_ERR_NO_API_KEY)
        The API key required by the active provider is missing from the 
        environment and the script is running in a non-interactive session 
        (absence of TTY), preventing manual input.

    11 (BASH4LLM_ERR_BAD_MODEL)
        The selected model is invalid, unsupported by the active provider, or 
        has been excluded due to core validation filters (non-textual 
        multimodal models).

    12 (BASH4LLM_ERR_CURL_FAILED)
        The network call made via curl utility failed due to connection issues, 
        DNS, server instability, or maximum timeout expiration.

    13 (BASH4LLM_ERR_INVALID_JSON)
        The JSON configuration file, models file, or response payload returned 
        by the provider's API is corrupted or lacks a valid syntactic structure.

    14 (BASH4LLM_ERR_NO_PROMPT)
        No text prompt or JSON file was provided as input for processing (except 
        in interactive chat mode or batch processing).

    15 (BASH4LLM_ERR_TMP)
        Unable to allocate or configure the temporary instance directory or 
        irreversible anomalies in acquiring exclusive lock files.

    16 (BASH4LLM_ERR_API)
        The provider's API server rejected the request or returned an invalid 
        HTTP status code (e.g., error 400, 401, 429, 500), or an anomalous 
        empty generation occurred (empty completion).

OPERATIONAL MODES
    bash4llm implements five main operational modes, selectable via CLI flags or 
    determined automatically by the state of input flows.

  1. Single Request (Default)
    The runtime acquires and assembles the prompt following the exclusive 
    priority chain. Resolves active model and provider, validates the API key, 
    and executes a single synchronous or streaming call. Upon completion, 
    extracts the text response, displays it based on output format, saves to 
    history if over the threshold limit, and logs in the active thread file.

  2. Token Streaming (--stream)
    If enabled, HTTP connection is established disabling network buffering. The 
    runtime parses line-by-line the Server-Sent Events (SSE) data stream. 
    Extracts textual fragments (delta.content) validating JSON syntax, sending 
    them instantly to stdout. In parallel, accumulates the raw stream to 
    transactionally reconstruct the full response and save the summary JSON 
    packet in the response file upon termination.

  3. Batch Processing (--batch)
    Sequentially scans lines of a prompt file. For each prompt, executes payload 
    compilation, request forwarding, response extraction, and saving. Each 
    execution is isolated: API or network failures in one call do not interrupt 
    subsequent prompt processing.

  4. Interactive Chat (--chat)
    Only available in interactive terminals (TTY). Opens a continuous read loop: 
    acquires prompt, compiles payload integrating thread history, executes 
    query, and displays assistant response. Updates real-time.

  5. Interactive Provider Selection
    Console lists supported providers, awaiting input. Once validated, stores 
    choice, removes temporary local files associated with models of the previous 
    provider, and terminates.

INTERNAL IMPLEMENTATION DETAILS
    This section describes the mechanisms, algorithms, and logics governing the 
    core of bash4llm.

  1. Prompt Assembly Algorithm (Priority Chain)
    Assembly of $CONTENT scanned sequentially: Structured JSON ($JSON_INPUT) -> 
    files via -f ($FILE_INPUTS) -> templates ($TEMPLATE) -> stdin 
    ($STDIN_CONTENT) -> positional arguments ($ARGS).

  2. Thread Management and Session Engine Integration
    Thread multi-turn context persistence operates on two levels:
      - Extended Architecture (Session Engine): At boot, the runtime checks for 
        the presence of the "session-engine.sh" extension script. If detected 
        and deemed compliant, $SE_AVAILABLE is set to 1. The runtime delegates 
        the context window construction and message writing to the external 
        engine routines (session_engine_build_window and session_engine_append).
      - Built-in Fallback (Legacy): If Session Engine is unavailable, the script 
        falls back on native NDJSON files. thread_read_window extracts trailing 
        lines to compose a coherent JSON array, while thread_append records user 
        messages, locking permissions to 600. A marking mechanism based on 
        SHA-256 hashes prevents duplicate entries.

  3. Model Synchronization and Core Whitelisting
    Conformance handled by validate_model_core. Filters non-textual multimodal 
    models out.

  4. Thread Cache (Thread Cache)
    Avoids redundant queries. Stores thread response caches under THREAD_CACHE_DIR 
    with TTL timestamp check on thread_cache_get and eviction via 
    thread_cache_invalidate.

  5. Multimodal Manifest and Base64 Staging
    Staging utilities manifest_create and manifest_add_part. Security size 
    validation $MAX_STAGE_BYTES (10MB).

  6. Concurrent Synchronization and File Locking (flock)
    Lock exclusive mechanism regulated by lock_exec with timeouts to avoid write 
    corruptions.

  7. Network Gate and Network Policy
    Centralized HTTP gate enforce_network_policy. Blocks connections under 
    dry-run, skip network, or quiet lock parameters.

  8. Base64 Encoding Architectural Detection
    Runtime probe _detect_base64_opts checking syntax differences between 
    GNU/Linux (-w0 and -d) and BSD/macOS (no wrapping and -D).

LEGACY COMPATIBILITY WRAPPERS
    To ensure integration with legacy scripts or third-party workflows, the 
    runtime maintains and exposes the following direct compatibility wrappers:

    buildpayloadgroq
        Direct alias of buildpayload_groq. Compiles the Groq JSON payload.

    call_api_streaming_groq_legacy
        Direct alias of call_api_streaming_groq. Starts token streaming.

    refreshmodelsgroq
        Direct alias of refresh_models_groq. Refreshes models.

    validatemodelgroq
        Direct alias of validate_model_groq. Validates models.

    autoselectmodelgroq
        Direct alias of auto_select_model_groq. Performs auto-selection.

TROUBLESHOOTING AND DIAGNOSTICS
    This section describes known anomalous behaviors and procedures to resolve 
    them.

  Anomalous Empty Responses (Empty Completion)
    External API may return HTTP 200 OK but complete generation without 
    producing text, sending "stop" finish_reason and near-zero tokens.
    bash4llm detects this automatically via detect_empty_edge_case, extracting:
      - Request ID (if present)
      - Generation Finish Reason
      - Returned Completion Tokens count
    Upon detection, logs a single structured record to standard error or log 
    and terminates with exit code 16.

  Connection Failures or Lock Timeouts
    If the script returns exit code 15 with timeout messages, ensure no other 
    background instances of bash4llm are locking resources. If needed, extend 
    lock timeouts by setting BASH4LLM_LOCK_TIMEOUT_... environment variables 
    to larger values (e.g., 30).
