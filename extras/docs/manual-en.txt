4. CLI options (complete, grouped)
--------------------------------------
This section lists **all** options and flags recognized by the script parser. Descriptions are strictly based on the behavior implemented in the script.

Note: the order and form are exactly those handled by the parser.

### Section: Input and prompt
- `-f <file>`  
  Adds a file as input. Can be repeated for multiple files. Files are read and concatenated with separators when appropriate.

- `--json-input <file>`  
  Uses a raw JSON payload from a file. If the value is `-`, the script reads JSON from stdin and uses it as JSON_INPUT. When provided, `CONTENT` is ignored and the JSON payload is used directly.

- `--template <name>`  
  Uses a template present in `groqbash.d/templates`. The template must contain the placeholder `{{CONTENT}}` which will be replaced with the assembled content.

- `--batch <file>`  
  Batch mode: processes each non-empty line of the file as a separate prompt.

- Positional ARGS (array ARGS)  
  Positional arguments collected as prompt elements or expanded into CONTENT.

### Section: Model and generation parameters
- `-m, --model <name>`  
  Selects the model (has highest precedence).

- `--system <text>`  
  Sets the system prompt (provider-specific field).

- `--ture <n>`  
  Alias for temperature (note: the script internally uses the variable TURE). Must be numeric.

- `--temperature <n>`  
  Alias for temperature; maps to the same internal variable (TURE/TEMPERATURE). Must be numeric.

- `--max <n>`  
  Sets the maximum number of tokens (MAX_TOKENS). Must be numeric.

### Section: Sessions and conversations
- `--session <session_id>`  
  Enables sessioning with the provided ID. The script validates the ID with `session_validate_id` (alphanumeric, dots, underscores, dashes, max length). When active, the script prepares BUILD_MESSAGES_FILE and uses session_read_window / session_append to read and write session history.

- `--session-window [<N>]`  
  Specifies the window (number of messages) to read for the session. If provided without an argument, the parser sets `SESSION_WINDOW` to `10`. If provided with a value, it uses the numeric value. The script recommends values <= 20 and emits a warning if > 20.

### Section: Output and formatting
- `--json`  
  Raw JSON output (prints the RESP file as-is).

- `--pretty`  
  Pretty-prints JSON (if RESP is JSON).

- `--text`  
  Text output (default).

- `--raw`  
  Raw text output without an added newline.

- `--out <path>`  
  Saves output to a file or directory. If `OUT_PATH` is a directory, the script creates a timestamped file inside.

- `--threshold <bytes>`  
  Byte threshold for automatic saving to history (THRESHOLD). Default 1000.

- `--save`  
  Forces saving textual output to history (FORCE_SAVE_MODE="save").

- `--nosave`  
  Disables saving output (FORCE_SAVE_MODE="nosave").

### Section: Streaming and call mode
- `--stream`  
  Enables SSE streaming. Requires the provider to implement `call_api_streaming_<prov>()`. Incompatible with `--json` and `--pretty` (the script checks and rejects the combination).

- `--no-stream`  
  Explicitly disables streaming.

### Section: Interactive modes and provider
- `--provider <name>`  
  If provided with an argument (non-empty and not starting with `-`), sets `PROVIDER_CLI` and persists the choice in `groqbash.d/config/provider`. If the argument is the literal string `list`, the code enters the interactive selection flow (see parser behavior).

- `--provider` (without argument)  
  Enters interactive provider selection mode (TTY required). Sets `PROVIDER_INTERACTIVE`.

- `--install-extras [<src>]`  
  Installs extras from the specified source path (optional) into the runtime directory `GROQBASH_EXTRAS_DIR`. Also supports the form `--install-extras=<src>`. Validates source, properties and copies files atomically.

### Section: Model and list management actions
- `--refresh-models`  
  Triggers refresh of provider models (invokes `refresh_models_<prov>()` after provider is loaded and API key ensured).

- `--refresh-model`  
  Alias accepted by the script for `--refresh-models`.

- `--list-models`  
  Prints the local whitelist of models in a human-readable form.

- `--list-models-raw`  
  Prints the list of models in machine-readable form (one normalized model per line).

- `--list-providers`  
  Prints available providers (embedded + installed) in a human-readable form.

- `--list-providers-raw`  
  Prints available providers in machine-readable form (one provider per line).

- `--set-default <model>`  
  Persists the default model for the active provider in `groqbash.d/config/model.<provider>`.

### Section: Debug, logging and behavior
- `--dry-run`  
  Prints the payload preview and skips the HTTP call (DRY_RUN). Shows the payload head.

- `--quiet`  
  Minimizes runtime messages.

- `--debug`  
  Enables verbose logging and preserves `RUN_TMPDIR` (does not clean it at the end).

- `--show-config`  
  Prints the active configuration (provider, model, paths, flags) and exits.

- `--diagnostics`  
  Runs diagnostic checks (permissions, provider, model) and exits.

- `--version`  
  Prints script name, version and date and exits.

- `-h, --help`  
  Shows this help and exits.

### Section: End of options
- `--`  
  End-of-options marker: everything that follows is treated as positional arguments (ARGS).

5. Operating modes
---------------------
This section describes the operating modes implemented in the script and the associated behavior.

### Mode: Single request (default)
- Flow: assemble content (ARGS, FILE_INPUTS, STDIN, TEMPLATE, JSON_INPUT), select model, build payload via provider, invoke API (streaming or non-streaming), extract response, finalize output, conditionally save history, optionally append to session.
- Typical usage: `groqbash "prompt"`.

### Mode: Streaming (`--stream`)
- Requirement: provider must implement `call_api_streaming_<prov>()`.
- Behavior: streaming is executed via curl HTTP/1.1; the raw stream is saved to `RUN_TMPDIR/resp.raw`. The script extracts lines starting with `data:` and filters valid JSON, reconstructs `resp.chunks.json` and `resp.text.txt`, and writes `RESP` atomically.
- Incompatibility: `--stream` cannot be used with `--json` or `--pretty`.

### Mode: Batch (`--batch <file>`)
- For each non-empty line in the file:
  - Sets `CONTENT` to the line.
  - If `--session` is active, prepares BUILD_MESSAGES_FILE for the session.
  - Builds payload and invokes provider.
  - Prints separator `----` between items.
- Does not abort the entire batch on a single error (the code continues with logged errors).

### Mode: Interactive chat (`--chat`)
- Requires TTY.
- Interactive loop: reads user lines, builds `CONTENT`, prepares BUILD_MESSAGES_FILE if `--session` is active, invokes provider (streaming or non-streaming), prints response.
- Conversation state is kept in memory for the current session; `session_append` can be used to persist.

### Mode: Interactive provider selection (`--provider` without argument)
- Requires TTY.
- Shows a list of available providers (embedded + installed), allows numeric selection, persists choice in `groqbash.d/config/provider`.

8. Sessions (full behavior)
------------------------------------
The script implements a session system based on NDJSON files and a windowing mechanism.

**Session ID validation**
- `session_validate_id` verifies the ID is non-empty and matches the pattern `[A-Za-z0-9._-]{1,128}`.

**Reading the message window**
- `session_read_window <session_id> <N> <out_file>`:
  - Reads the session NDJSON (`groqbash.d/history/sessions/<session_id>.ndjson`) and produces a JSON file with `{"messages":[...]}` containing the last N entries.
  - Uses a lock on the session file to avoid partial reads during concurrent appends.
  - Normalizes roles (`user`, `assistant`, `system`) and builds messages compatible with the provider.
  - If the file does not exist or is unreadable, it may produce an empty file or fail silently (the CORE has fallbacks).

**Idempotent append**
- `session_append <session_id> <role> <content> <meta_json>`:
  - Creates a compact NDJSON record with timestamp, role, content and meta.
  - Uses markers based on `message_id` for cross-process idempotency; if `message_id` is present and the marker exists, the append is skipped.
  - Acquires an exclusive lock on the session file for atomic append.
  - Saves the file with restrictive permissions (600).
  - Avoids appending empty user messages.

**BUILD_MESSAGES_FILE**
- When `--session` is active, the CORE prepares `BUILD_MESSAGES_FILE` in `RUN_TMPDIR` and exports it for the provider; the provider can use this file to build the payload.

**Fallback**
- If `session_read_window` does not produce messages, the CORE attempts to reconstruct the window with `tail` on the NDJSON or, as a last resort, creates a payload with only the current user message.

ADDED: Session cache (integration)
- **Location:** inserted after "Sessions (full behavior)".
- **Purpose:** cache session-derived payloads/responses to avoid recomputation.
- **API:** `session_cache_key(sid, params)`, `session_cache_get(key)`, `session_cache_set(key, ttl, payload)`, `session_cache_invalidate(key|sid)`.
- **Format:** cache files stored under `${GROQBASH_CONFIG_DIR%/}/session_cache` with perms 600. File format: **first line = expiry_epoch (integer)**; subsequent lines = payload JSON.
- **Behavior:** `session_cache_get` removes expired files and returns 0 on hit; `session_cache_set` writes atomically; `session_cache_invalidate` removes entries when session or provider changes. TTL enforced by expiry_epoch.

9. History and rotation
----------------------
**Saving output**
- `save_to_history` saves textual outputs in `groqbash.d/history` with names like `YYYYMMDD-HHMMSS-groq-output-$$.txt`.
- Uses atomic writes and locks to ensure consistency.
- Respects `FORCE_SAVE_MODE` (`save`/`nosave`) and `THRESHOLD` to decide automatic saving.

**Rotate history**
- `rotate_history` (enabled with `GROQBASH_ROTATE_HISTORY=1`) applies:
  - Removal of files older than `GROQBASH_HISTORY_KEEP_DAYS`.
  - Reduction of total space to `GROQBASH_HISTORY_MAX_BYTES`.
  - Reduction of number of files to `GROQBASH_HISTORY_MAX_FILES`.
- All operations are performed under `HISTORY_LOCK`.

---  
ADDED: Manifest multimodal (integration)
- **Location:** inserted after "History and rotation".
- **Purpose:** manage multimodal manifests and staged parts safely.
- **Primitives:** `manifest_create(manifest_path)`, `manifest_add_part(manifest, srcfile, metadata)`, `manifest_read(manifest_path)`.
- **Staging:** parts are staged via `stage_b64(srcfile, dst.b64)`; each part stored as a `.b64` artifact.
- **manifest.b64:** a base64-encoded representation of the manifest is written alongside the JSON manifest for transport/archival.
- **Atomicity:** manifest updates occur under a manifest lock using `lock_exec` and `atomic_write`; failures must leave consistent state.

10. Streaming (technical details)
-------------------------------
**Requirements**
- Provider must implement `call_api_streaming_<prov>()`.

**Behavior**
- Streaming is executed with `curl --http1.1` (the script uses the `--http1.1` option).
- The raw stream is saved to `RUN_TMPDIR/resp.raw`.
- The script extracts lines that start with `data:` and removes the prefix.
- It filters only lines that are valid JSON; saves valid JSON lines to `resp.valid.jsons`.
- Builds `resp.chunks.json` as an array of JSON chunks and `resp.text.txt` as the concatenation of extracted text fragments.
- Writes `RESP` atomically using `resp.chunks.json` if valid chunks are present; otherwise, if `RESP_RAW` is valid JSON, it copies it to `RESP`.
- During streaming, extracted text fragments are emitted to stdout as they arrive.

**Temporary files**
- `resp.raw`, `resp.lines`, `resp.valid.jsons`, `resp.chunks.json`, `resp.text.txt` are created in `RUN_TMPDIR` and removed when appropriate (unless debug mode is active).

11. Retry and API edge-case handling
----------------------------------
**Retry**
- `perform_request_once` implements a retry loop up to `MAX_RETRIES` (default 3).
- After each failure, the code evaluates the error type:
  - If a network error (`GROQBASHERRCURL_FAILED`), it retries with incremental backoff.
  - If an API error (`GROQBASHERRAPI`), it does not retry and fails immediately.

**Response extraction**
- `extract_text_from_resp` attempts to extract text from `RESP` using:
  - `.choices[] | .message?.content // .delta?.content` and joins with newline.
  - Fallback: if `RESP` is not JSON, it cats the file.

**Edge case: "empty completion"**
- `detect_empty_edge_case` analyzes `RESP` to detect the empty completion pattern:
  - Checks `choices[0].message.content` or `choices[0].delta.content` (empty),
  - `choices[0].finish_reason == "stop"`,
  - `usage.completion_tokens == 1` (or equivalent).
- If detected, it sets global diagnostic variables:
  - `GROQBASH_EDGE_EMPTY=1`
  - `GROQBASH_EDGE_REQ_ID` (if present in `RESP` as `.x_groq?.id` or `.id`)
  - `GROQBASH_EDGE_FINISH_REASON`
  - `GROQBASH_EDGE_COMPLETION_TOKENS`
- The CORE emits **one structured log** with these details inside `perform_request_once` and then fails with the appropriate API exit code.

12. Security and filesystem constraints
----------------------------------
**General principles**
- All runtime state is under `GROQBASH_DIR` (default: `SCRIPTDIR/groqbash.d`).
- No use of the system `/tmp`: all tmp files are under `GROQBASH_TMPDIR` and `RUN_TMPDIR`.
- Umask and restrictive permissions: directories 700, files 600 where applicable.
- No `eval` and no execution of model-generated content.
- All critical writes (models, manifest, history) are atomic and protected by locks (flock).

**Provider modules**
- Must not be symlinks.
- Must pass `bash -n` before being sourced.
- Ownership and permissions controlled: not world-writable; preferably owned by the current user.
- The CORE verifies the presence of required functions and rejects incomplete providers.

**Extras repository vs runtime**
- `./extras/` in the repo root is only a source for `--install-extras`.
- At runtime, the only trusted paths for extras/providers are `GROQBASH_EXTRAS_DIR` and `PROVIDERS_DIR` under `GROQBASH_DIR`.
- The script prevents installs that would create nested copies or use symlinks as source.

**Locking**
- Locks are colocated near resources:
  - `MODELS_LOCK` (e.g., `groqbash.d/models/models.lock`)
  - `HISTORY_LOCK` (e.g., `groqbash.d/history/history.lock`)
  - `TMP_LOCK` (e.g., `groqbash.d/tmp/tmp.lock`)
- `lock_exec` uses `flock -x -w <timeout>` and fails with code 124 on timeout.

---  
ADDED: Network policy (enforce_network_policy)
- **Location:** suggested insertion under "Security and filesystem constraints".
- **Purpose:** single centralized gate for all network access.
- **Behavior:** `enforce_network_policy()` returns success only if network calls are allowed. It respects `DRY_RUN` (blocks real network calls), `GROQBASH_SKIP_NETWORK` (explicit block), and `GROQBASH_ENFORCE_NO_NETWORK_IF_QUIET` combined with `QUIET` (block when quiet mode enforces no network). All provider call sites **must** call `enforce_network_policy()` before performing curl; when blocked, providers must produce a RESP diagnostic and return non-zero. This function is the canonical place to implement CI/automation safe-guards.

15. Troubleshooting
-------------------
- **Script exits immediately**: verify that all mandatory commands are in PATH (see Requirements).
- **Provider not found**: install provider script in `groqbash.d/extras/providers/<prov>.sh` or use `--install-extras`.
- **API calls fail in CI**: ensure the API key is exported (`GROQ_API_KEY` or the variable specified by `PROVIDER_API_ENV_<prov>`).
- **History not saved**: check `GROQBASH_ROTATE_HISTORY`, `THRESHOLD` and permissions on `groqbash.d/history`.
- **Provider module issues**: verify the file is not a symlink, passes `bash -n` and is not world-writable.
- **Lock timeout**: if receiving lock timeout errors, increase `GROQBASH_LOCK_TIMEOUT_*` or investigate concurrent processes holding locks.

16. Ambiguities found in the code
---------------------------------
The script is the source of truth; however, the parser and some naming choices introduce ambiguities which are neutrally documented here:

1. **Option `--ture` vs `--temperature`**  
   - The script accepts both forms. Internally it uses the variable `TURE` as the primary name and then synchronizes with `TEMPERATURE`. This dual naming exists in the code; both options are supported and map to the same value.

2. **`--json-input` with value `-`**  
   - The code supports `--json-input -` as a signal to read JSON from stdin. This behavior is implemented but requires care: when `JSON_INPUT` is `-`, the script captures stdin into a temporary file and uses it as the JSON source.

3. **`--session-window` without argument**  
   - The parser allows `--session-window` without a value and in that case sets `SESSION_WINDOW="10"`. This behavior is explicit in the parser and can be a source of confusion if the user expects the option to always require an argument.

4. **`--provider list` vs `--provider` without argument**  
   - If the user invokes `--provider list` (the word `list` as an argument), the code treats the string `list` as a request for interactive selection. Additionally, `--provider` without an argument activates interactive selection. Both forms lead to the same interactive flow, but the presence of the word `list` as an argument is a special case handled by the parser.

5. **`--install-extras` form with and without argument**  
   - The script supports `--install-extras <src>`, `--install-extras=<src>` and also `--install-extras` without an argument (in this case it uses `SCRIPTDIR/extras` as the source). The behavior is implemented but the presence of multiple forms should be considered when automating usage.

These ambiguities are reported for clarity and to guide correct usage of the script; they are not code errors but parsing choices the user should be aware of.

---------------------------------------
- The script is designed to be auditable: all sensitive operations (I/O, locks, writes) are centralized in atomic primitives and utility functions.
- External providers are loaded via `source` after syntax and permission checks; the CORE communicates with them exclusively through the minimal interface functions (`buildpayload_<prov>`, `call_api_<prov>`, and optional ones).
- For non-interactive integrations, ensure the correct API key is exported and set `ALLOW_API_CALLS` if necessary (the code enforces a check if `GROQ_API_KEY` is present but `ALLOW_API_CALLS` is not 1 and `DRY_RUN` is not 1).
- All runtime state must remain under `GROQBASH_DIR` (default `SCRIPTDIR/groqbash.d`).
