================================================================================
BASH4LLM(1)                  Manuale di Bash4LLM                    BASH4LLM(1)
================================================================================

NOME
    bash4llm - Wrapper Bash sicuro e resiliente per interrogare provider 
    LLM esterni (v2.6.0)

SINTASSI
    bash4llm [OPZIONI] [PROMPT]
    bash4llm -f FILE [OPZIONI]
    echo "prompt" | bash4llm [OPZIONI]
    bash4llm --batch FILE_BATCH [OPZIONI]
    bash4llm --chat [OPZIONI]
    bash4llm --vault

DESCRIZIONE
    bash4llm è un'utility a riga di comando per ambienti shell nativi, progettata 
    per l'integrazione flessibile, l'automazione e l'orchestrazione delle 
    interazioni con modelli linguistici di grandi dimensioni (LLM) forniti da 
    servizi esterni come Groq, Gemini e HuggingFace.

    Il runtime opera sotto vincoli di esecuzione restrittivi (set -euo pipefail) 
    per garantire l'interruzione immediata delle pipeline in presenza di comandi 
    non validi o variabili non inizializzate. L'architettura è interamente 
    progettata per isolare le operazioni di sistema dall'esecuzione remota, 
    proibendo qualsiasi interpretazione dinamica o esecuzione locale di codice 
    generato (politica "zero eval") ed escludendo l'uso di directory di sistema 
    condivise come /tmp per mitigare gli attacchi basati su link simbolici 
    (mitigazione attacchi symlink).

OPZIONI
    Le opzioni dello script sono raggruppate per categoria funzionale. I parametri 
    possono essere passati in qualsiasi ordine arbitrario, ad eccezione del 
    marcatore di fine opzioni.

  Sezione 1: Gestione Input e Prompt
    -f FILE
        Aggiunge il file specificato come sorgente di contesto per l'elaborazione. 
        Può essere specificato più volte per concatenare sequenzialmente più file 
        di input. I singoli file sono separati da demarcatori visivi strutturati 
        per consentire al modello di distinguere chiaramente le fonti.

    --json-input FILE
        Invia un payload JSON strutturato, utile per definire direttamente una 
        lista di messaggi personalizzati con ruoli specifici. Se impostato su '-', 
        lo script legge dallo standard input (stdin). L'uso di questa opzione 
        esclude e azzera la risoluzione della variabile prompt generica.

    --template NOME
        Indica il modello di testo da applicare al prompt dell'utente, prelevato 
        dalla directory dei template utente. Il file template deve contenere 
        il segnaposto "{{CONTENT}}" che viene sostituito a runtime con il testo 
        di input dell'utente assemblato.

    --batch FILE
        Attiva la modalità di elaborazione a lotti (batch). Lo script analizza il 
        file specificato ed esegue una chiamata API distinta per ogni riga non 
        vuota, escludendo i commenti (righe che iniziano con un cancelletto). Un 
        errore in un singolo prompt non interrompe l'intera sequenza batch.

    PROMPT (Argomenti Posizionali)
        Tutti gli argomenti posizionali non catturati da flag specifici sono 
        trattati come frammenti di testo. Se un argomento corrisponde al percorso 
        di un file regolare e leggibile su disco, lo script lo espande leggendone 
        e incorporandone il contenuto; altrimenti, lo interpreta come testo 
        letterale da appendere al prompt principale.

  Sezione 2: Parametri del Modello e della Generazione
    -m, --model MODELLO
        Specifica il nome del modello LLM da interrogare. Questa impostazione ha 
        precedenza assoluta sui modelli predefiniti salvati localmente.

    --system TESTO
        Definisce o sovrascrive il prompt di sistema (system prompt), che 
        istruisce o condiziona il comportamento, il tono e il contesto operativo 
        del modello linguistico.

    --temperature VALORE, --ture VALORE
        Imposta la temperatura di campionamento decimale (solitamente compresa 
        tra 0.0 e 2.0). Entrambi i flag si collegano e si sincronizzano in modo 
        bidirezionale alle variabili interne "TURE" e "TEMPERATURE", unificando 
        la loro risoluzione interna prima della trasmissione.

    --max NUMERO
        Imposta il limite massimo di token da generare nella risposta del 
        modello (variabile interna "MAX_TOKENS").

  Sezione 3: Gestione dei Thread e delle Conversazioni
    --thread THREAD_ID
        Abilita la persistenza del contesto di chat multi-turno associando la 
        cronologia a un identificativo univoco. L'ID viene formalmente validato 
        e anonimizzato crittograficamente tramite SHA-256 ($SAFE_THREAD_ID) per 
        prevenire la perdita di dati personali (PII) su disco. L'ID deve 
        corrispondere alla regex ^[A-Za-z0-9._-]{1,128}$.

    --thread-window [FINESTRA]
        Specifica il numero massimo di messaggi storici precedenti da recuperare 
        dal file della cronologia del thread per comporre la finestra di contesto. 
        Se il flag è specificato senza argomenti, il valore predefinito è 10. 
        Per ottimizzare le prestazioni ed evitare di saturare il contesto del 
        modello, si raccomandano valori minori o uguali a 20; impostare valori 
        superiori a 20 fa sì che il runtime emetta un avviso diagnostico.

    --init-thread (Richiede --thread <id>)
        Consente a componenti esterni (come un'interfaccia grafica CGI) o a 
        script di automazione di pre-registrare un nuovo thread nel sistema prima 
        di inviare il primo prompt. Questa opzione esegue la registrazione locale 
        del thread e termina immediatamente l'esecuzione (exit 0) senza effettuare 
        chiamate API.

    --delete-thread THREAD_ID
        Elimina definitivamente i file di cronologia del thread attivo (.ndjson), 
        i file di lock, i file di metadati (.json) e rimuove la voce dell'ID 
        dall'indice globale dei thread sotto lock esclusivo.

    --rename-thread THREAD_ID
        Aggiorna il titolo visibile all'utente per il thread specificato all'interno 
        del suo file di metadati. Deve essere usato in combinazione con --title <testo>.

    --title TESTO
        Specifica la nuova stringa di titolo utente durante l'esecuzione della 
        rinomina di un thread tramite --rename-thread.

  Sezione 4: Output e Formattazione
    --json
        Disabilita l'estrazione del testo e visualizza il payload JSON grezzo della 
        risposta del provider esattamente come ricevuto dall'API esterna.

    --no-color
        Disabilita l'uso dei colori ANSI e degli stili di testo sia sullo standard 
        output che sullo standard error, garantendo log puliti nelle sessioni 
        non interattive.

    --pretty
        Esegue la formattazione strutturata (pretty-print) sulla risposta JSON 
        prima di visualizzarla sullo standard output.

    --text
        Seleziona la modalità di visualizzazione del testo standard (predefinita). 
        Lo script estrae solo il corpo del messaggio testuale prodotto dal modello.

    --raw
        Restituisce solo il testo della risposta, escludendo qualsiasi carattere 
        di nuova riga finale aggiuntivo.

    --out PERCORSO
        Salva la risposta finale nel percorso specificato. Se il percorso indica 
        una directory esistente, lo script genera al suo interno un file univoco 
        nominato con un timestamp cronologico; se indica un file, lo sovrascrive 
        in modo transazionale e sicuro.

    --threshold NUMERO
        Imposta il limite di soglia, espresso in caratteri di testo (predefinito 
        1000 caratteri), oltre il quale la risposta prodotta dall'LLM viene 
        salvata automaticamente e permanentemente nella directory della cronologia.

    --save
        Forza il salvataggio incondizionato del risultato testuale nella directory 
        della cronologia, ignorando il controllo del limite di soglia 
        (FORCE_SAVE_MODE="save").

    --nosave
        Disabilita assolutamente l'archiviazione automatica del risultato nella 
        cronologia delle interazioni (FORCE_SAVE_MODE="nosave").

  Sezione 5: Flussi di Connessione e Rete
    --stream
        Abilita la ricezione dei token in tempo reale (streaming dati). Richiede 
        che il provider attivo supporti l'interfaccia di connessione SSE 
        (Server-Sent Events). Questa opzione è mutualmente esclusiva con i formati 
        strutturati --json e --pretty.

    --no-stream
        Inibisce e disabilita esplicitamente la modalità di streaming dei token, 
        ripristinando la ricezione sincrona e bloccante dell'intera risposta.

    --dry-run
        Simula la transazione senza effettuare connessioni di rete reali. Lo 
        script compila la struttura del payload della richiesta JSON finale, ne 
        visualizza le prime righe sullo standard error a scopo diagnostico e 
        termina l'esecuzione con uno stato di successo.

  Sezione 6: Selezione Moduli, Sicurezza e Installazione
    --provider [NOME], --providers [NOME]
        Se specificato con un nome, seleziona e imposta il provider attivo, 
        salvando la preferenza in modo persistente nella directory di configurazione. 
        Se il valore passato è "list", o se il flag è specificato senza alcun 
        argomento in un terminale interattivo (TTY), apre un menu di selezione 
        interattivo.

    --vault
        Avvia lo strumento console crittografico interattivo per gestire le chiavi 
        API cifrate all'interno di keys.enc, supportando la creazione, modifica, 
        eliminazione delle chiavi, cambio della password master e recupero 
        offline d'emergenza.

    --install-extras [SORGENTE], --install-extra [SORGENTE]
        Installa ed estende il runtime copiando moduli e risorse dalla directory 
        delle estensioni nella cartella utente persistente, eseguendo rigorosi 
        controlli di integrità tramite manifest.sha256.

  Sezione 7: Gestione Modelli e Configurazione Rapida
    --refresh-models, --refresh-model
        Forza una connessione immediata con l'API del provider attivo per 
        sincronizzare l'elenco locale dei modelli disponibili, salvandolo in modo 
        atomico e sicuro sotto lock esclusivo nel file dei modelli.

    --list-models, --list-model
        Visualizza un elenco formattato e leggibile dei modelli di testo 
        attualmente disponibili e pronti all'uso nel sistema locale.

    --list-models-raw
        Restituisce l'elenco grezzo dei modelli locali in formato macchina (un 
        identificatore normalizzato per riga) per scopi di automazione.

    --list-providers, --list-provider
        Visualizza l'elenco dei moduli provider installati nel sistema.

    --list-providers-raw
        Restituisce l'elenco dei provider installati in formato macchina, senza 
        formattazione o intestazioni di testo.

    --set-default MODELLO
        Imposta e archivia permanentemente il modello specificato come scelta 
        predefinita per il provider attualmente attivo nel sistema.

  Sezione 8: Interfaccia Utente Interattiva, Diagnostica e Ispezione
    --chat, --tui
        Avvia la sessione REPL interattiva TUI a schermo intero per conversazioni 
        continue.

    --quiet
        Silenziatore del runtime. Inibisce tutte le stampe informative e gli avvisi 
        non critici sullo standard error.

    --debug
        Abilita la modalità diagnostica avanzata per gli sviluppatori. Scrive 
        messaggi di log dettagliati sullo standard error e preserva i file di 
        runtime temporanei.

    --show-config
        Genera una vista tabellare strutturata dei parametri di runtime attuali 
        (versioni del software, provider selezionato, modello attivo, percorsi 
        operativi e stati dei file di sincronizzazione).

    --check-config
        Esegue un audit statico di sicurezza sul file di configurazione utente, 
        verificando i permessi del filesystem per vulnerabilità di scrittura 
        group/world e analizzando le chiavi delle variabili rispetto alle 
        specifiche canoniche.

    --diagnostics, --diagnostic
        Avvia un ciclo completo di ispezione e autocontrollo verificando le 
        cartelle fisiche, gli accessi in scrittura, l'integrità dei moduli provider, 
        le chiavi API e testando gli handshake delle connessioni TLS tramite OpenSSL.

    --explain-error CODICE_O_ALIAS
        Spiega i codici numerici di errore in uscita (es. 15, 17) o gli alias 
        testuali (es. BASH4LLM_ERR_SEC) utilizzando le definizioni documentate e 
        fornendo consigli operativi di risoluzione.

    --print-config-dir
        Stampa il percorso canonico della directory di configurazione sullo 
        standard output.

    --print-provider-file
        Stampa il percorso canonico del file di persistenza del provider attivo 
        sullo standard output.

    --print-model-file PROVIDER
        Stampa il percorso canonico del file del modello predefinito per il provider 
        specificato sullo standard output.

    --version
        Stampa il nome identificativo dello script, la versione del software, la 
        data di rilascio ed esce.

    -h, --help
        Visualizza la guida rapida dell'interfaccia e termina.

    --
        Marcatore di fine opzioni. Qualsiasi argomento posizionato dopo questo 
        delimitatore viene interpretato come testo del prompt di input.

VARIABILI D'AMBIENTE
    Il comportamento del runtime bash4llm è governato e può essere condizionato 
    dalle seguenti variabili d'ambiente.

  Sezione 1: Credenziali API e Sicurezza del Vault
    GROQ_API_KEY
        Chiave API predefinita per l'autenticazione delle chiamate di rete al 
        provider integrato Groq.

    PROVIDER_API_ENV_[PROVIDER]
        Mappatura indiretta per provider esterni. Definisce il nome di una 
        variabile d'ambiente alternativa destinata a contenere la chiave API 
        per un dato provider personalizzato.

    BASH4LLM_VAULT_ENABLED
        Flag booleano (1 o 0) per controllare il sistema di vault crittografico 
        delle chiavi. Impostare a 0 per disabilitare permanentemente o 
        temporaneamente le ricerche nel vault (predefinito: 1).

    _B4L_RT_CTX
        Token di contesto di sessione offuscato mantenuto nella memoria della 
        shell attiva per sbloccare la chiave Master del Vault ed evitare richieste 
        di password tra più esecuzioni.

    BASH4LLM_OPENSSL_ACTIVE
        Flag di stato interno (1 o 0) che indica se l'estensione helper di 
        sicurezza OpenSSL è caricata e attiva.

  Sezione 2: Percorsi e Directory Isolate del Filesystem
    BASH4LLM_DIR
        Directory radice persistente per le risorse e i dati di bash4llm. L'impostazione 
        predefinita è "bash4llm.d" relativa alla posizione dello script.

    BASH4LLM_ROOT
        Percorso alternativo utilizzato per derivare la directory radice se 
        BASH4LLM_DIR non è esplicitamente dichiarata.

    BASH4LLM_CONFIG_DIR
        Directory contenente le impostazioni utente persistenti 
        (predefinito: $BASH4LLM_DIR/config).

    BASH4LLM_MODELS_DIR
        Directory contenente i modelli configurati e memorizzati nella cache 
        (predefinito: $BASH4LLM_DIR/models).

    BASH4LLM_TEMPLATES_DIR
        Directory contenente i file di modello utente 
        (predefinito: $BASH4LLM_DIR/templates).

    BASH4LLM_HISTORY_DIR
        Directory contenente la cronologia delle risposte e i log dei thread 
        (predefinito: $BASH4LLM_DIR/history).

    BASH4LLM_TMPDIR
        Directory radice protetta per i dati temporanei dello script 
        (predefinito: $BASH4LLM_DIR/tmp).

    BASH4LLM_RUN_DIR
        Directory che ospita i file di processo di runtime 
        (predefinito: $BASH4LLM_DIR/var/run).

    BASH4LLM_LOCKS_DIR
        Directory isolata che memorizza i file di lock 
        (predefinito: $BASH4LLM_RUN_DIR/locks).

    BASH4LLM_RATES_DIR
        Directory isolata che traccia i timestamp delle transazioni per il 
        rate-limiting (predefinito: $BASH4LLM_TMPDIR/rates).

    BASH4LLM_EXTRAS_DIR
        Directory attiva per accedere alle estensioni utente 
        (predefinito: $BASH4LLM_DIR/extras).

    PROVIDERS_DIR
        Directory contenente i moduli dei provider esterni 
        (predefinito: $BASH4LLM_EXTRAS_DIR/providers).

    THREAD_CACHE_DIR
        Directory che ospita la cache temporanea dei messaggi dei thread 
        (predefinito: $BASH4LLM_CONFIG_DIR/thread_cache).

  Sezione 3: Configurazione Risorse e Sincronizzazione
    MODELS_FILE
        Percorso del file locale che memorizza i modelli recuperati 
        (predefinito: $BASH4LLM_MODELS_DIR/<provider>.txt).

    MAX_MODELS
        Limite numerico di modelli gestibili localmente (predefinito: 200).

    PROVIDER_FILE
        Percorso del file che persiste il nome del provider attivo.

    MODELS_LOCK
        Lock esclusivo che regola gli aggiornamenti ai file dei modelli 
        (predefinito: $BASH4LLM_LOCKS_DIR/models.lock).

    HISTORY_LOCK
        Lock esclusivo che previene la corruzione durante le operazioni sulla 
        cronologia e sui thread (predefinito: $BASH4LLM_HISTORY_DIR/history.lock).

    TMP_LOCK
        Lock esclusivo che regola l'allocazione della directory temporanea 
        (predefinito: $BASH4LLM_LOCKS_DIR/tmp.lock).

    BASH4LLM_LOCK_TIMEOUT_TMP
        Timeout in secondi per acquisire i lock della directory temporanea 
        (predefinito: 10).

    BASH4LLM_LOCK_TIMEOUT_MODELS
        Timeout in secondi per acquisire i lock dei file dei modelli 
        (predefinito: 10).

    BASH4LLM_LOCK_TIMEOUT_HISTORY
        Timeout in secondi per acquisire i lock della cronologia e dei thread 
        (predefinito: 10).

  Sezione 4: Manutenzione Cronologia (Rotazione)
    BASH4LLM_ROTATE_HISTORY
        Flag booleano (1 o 0) che abilita la rotazione periodica dei log della 
        cronologia (predefinito: 0).

    BASH4LLM_HISTORY_MAX_FILES
        Soglia massima di file nella directory della cronologia (predefinito: 100).

    BASH4LLM_HISTORY_MAX_BYTES
        Limite cumulativo di byte per i file della cronologia 
        (predefinito: 104857600, equivalente a 100MB).

    BASH4LLM_HISTORY_KEEP_DAYS
        Limite di conservazione temporale in giorni per i file della cronologia 
        (predefinito: 90 giorni).

  Sezione 5: Parametri, Rate Limiting e Politiche di Sicurezza
    BASH4LLM_RATE_LIMIT
        Limite numerico di richieste API consentite per thread all'interno di 
        una finestra scorrevole di 30 secondi (predefinito: illimitato).

    BASH4LLM_AUTH_TOKEN
        Token di autorizzazione segreto che ignora i controlli di rate-limiting 
        locali.

    FALLBACK_PAYLOAD
        Payload di riserva codificato in Base64 generato dagli hook di post-esecuzione 
        in caso di errori API.

    BASH4LLM_IGNORE_SEC_CHECKS
        Impostare a 1 per ignorare i controlli rigorosi di proprietà e permessi 
        dei file POSIX (utile per WSL, Cygwin o filesystem non standard).

    ALLOW_API_CALLS
        Flag per l'abilitazione delle transazioni esterne. Se impostato a 0, 
        blocca le chiamate di rete.

    BASH4LLM_SKIP_NETWORK
        Inibisce le chiamate di rete e forza l'esecuzione simulata in dry-run.

    BASH4LLM_ENFORCE_NO_NETWORK_IF_QUIET
        Se impostato a 1, blocca le connessioni esterne ogni volta che QUIET=1 è attivo.

    MAX_STAGE_BYTES
        Limite massimo in byte per lo staging in Base64 
        (predefinito: 10485760, equivalente a 10MB).

    DEBUG
        Flag diagnostico. Impostare a 1 per abilitare il debug dettagliato su stderr.

    BASH4LLM_LOG
        Percorso del file per la registrazione cronologica strutturata dei log.

    BASH4LLM_SOURCE_ONLY
        Guardia di sviluppo. Impostare a 1 per uscire immediatamente dopo il 
        bootstrap a scopo di test.

CODICI DI USCITA
    Il runtime adotta un insieme fisso di codici di uscita numerici:

    10 (BASH4LLM_ERR_NO_API_KEY)
        La chiave API richiesta è assente dall'ambiente e dal Vault in una 
        sessione non interattiva.

    11 (BASH4LLM_ERR_BAD_MODEL)
        Il modello selezionato non è valido, non è supportato o è multimodale 
        non testuale.

    12 (BASH4LLM_ERR_CURL_FAILED)
        La richiesta di rete tramite curl è fallita per problemi di connessione 
        o timeout.

    13 (BASH4LLM_ERR_INVALID_JSON)
        La configurazione JSON o il payload della risposta API è corrotto o 
        sintatticamente non valido.

    14 (BASH4LLM_ERR_NO_PROMPT)
        Non è stato fornito alcun testo di prompt o payload JSON.

    15 (BASH4LLM_ERR_TMP)
        Impossibile allocare la directory temporanea sicura o fallimento 
        nell'acquisizione del lock.

    16 (BASH4LLM_ERR_API)
        L'API del provider ha rifiutato la richiesta, ha restituito uno stato di 
        errore HTTP o ha generato una risposta vuota (empty completion).

    17 (BASH4LLM_ERR_SEC)
        Violazione della politica di sicurezza: permessi file non sicuri scritti 
        da group/world, link simbolico rilevato su un percorso critico o 
        fallimento del controllo di integrità crittografica SHA-256 del modulo 
        rispetto a manifest.sha256.

MODALITÀ OPERATIVE
    bash4llm implementa cinque modalità operative principali:

  1. Richiesta Singola (Predefinita)
    Acquisisce il prompt, risolve provider e modello, valida le chiavi API, 
    applica il rate limiting e gli hook di pre-esecuzione, esegue la query e 
    restituisce la risposta.

  2. Streaming dei Token (--stream)
    Stabilisce una connessione senza buffering, analizza l'SSE riga per riga, 
    invia immediatamente i delta di testo e salva la risposta JSON aggregata.

  3. Elaborazione a Lotti (--batch)
    Scansiona il file dei prompt riga per riga, eseguendo chiamate API isolate 
    per ogni prompt.

  4. Chat Interattiva (--chat, --tui)
    Avvia un REPL TUI interattivo a schermo intero per conversazioni multi-turno.

  5. Selezione Interattiva del Provider (--provider)
    Apre un menu di selezione interattivo per scegliere e salvare in modo 
    persistente il provider attivo.

DETTAGLI IMPLEMENTATIVI INTERNI
    Questa sezione descrive gli algoritmi principali e le logiche di sicurezza.

  1. Integrità Crittografica dei Moduli e Verifica del Manifest
    Provider esterni, hook ed estensioni vengono sottoposti a una verifica 
    obbligatoria a due livelli tramite verify_module_integrity prima del caricamento: 
    validazione della sicurezza del percorso (proprietà, nessuna scrittura group/world) 
    e corrispondenza dell'hash SHA-256 rispetto a extras/manifest.sha256. Eventuali 
    discrepanze interrompono l'esecuzione con codice di uscita 17.

  2. Anonimizzazione PII del Thread e Locking
    Gli ID grezzi dei thread vengono cifrati tramite hash SHA-256 mediante 
    anonymize_thread_id producendo $SAFE_THREAD_ID. Tutti i file su disco 
    (.ndjson, .json, lock) utilizzano questo hash anonimizzato, prevenendo la 
    perdita di dati personali sensibili sul filesystem.

  3. Limitatore di Frequenza Locale a Finestra Scorrevole
    check_local_rate_limit traccia i file dei timestamp di richiesta sotto 
    $BASH4LLM_RATES_DIR/$SAFE_THREAD_ID. Le richieste che superano 
    $BASH4LLM_RATE_LIMIT all'interno di una finestra di 30 secondi vengono 
    bloccate con codice di uscita 17.

  4. Architettura del Vault Crittografico delle Chiavi
    Quando OpenSSL è attivo, le chiavi API del provider vengono cifrate utilizzando 
    AES-256-CBC protetto da Chiave Master con PBKDF2 (100.000 iterazioni) e sale. 
    Il token di contesto di sessione $_B4L_RT_CTX mantiene le chiavi sbloccate del 
    vault nella RAM per evitare le richieste di password.

  5. Catena di Priorità per l'Assemblaggio del Prompt
    Assemblaggio di $CONTENT: JSON strutturato ($JSON_INPUT) -> file tramite -f -> 
    template ($TEMPLATE) -> standard input ($STDIN_CONTENT) -> argomenti posizionali.

RISOLUZIONE PROBLEMI E DIAGNOSTICA
    Risposte Vuote Anomale (Empty Completion)
        Rilevate automaticamente tramite detect_empty_edge_case quando viene 
        restituito HTTP 200 con zero token generati. Registra la diagnostica 
        ed esce con codice 16.

    Fallimenti di Connessione o Timeout dei Lock
        Se si verifica il codice di uscita 15 con messaggi di timeout, verificare 
        che non vi siano istanze in background che mantengono i lock, oppure 
        aumentare i valori di BASH4LLM_LOCK_TIMEOUT_....
