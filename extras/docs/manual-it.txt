4. Opzioni CLI (complete, raggruppate)
--------------------------------------
Questa sezione elenca **tutte** le opzioni e i flag che il parser dello script riconosce. Le descrizioni sono strettamente basate sul comportamento implementato nello script.

Nota: l’ordine e la forma sono esattamente quelli gestiti dal parser.

### Sezione: Input e prompt
- `-f <file>`  
  Aggiunge un file come input. Può essere ripetuto per più file. I file vengono letti e concatenati con separatori quando appropriato.

- `--json-input <file>`  
  Usa un payload JSON grezzo da file. Se il valore è `-`, lo script legge JSON da stdin e lo usa come JSON_INPUT. Quando fornito, `CONTENT` è ignorato e il payload JSON viene usato direttamente.

- `--template <name>`  
  Usa un template presente in `groqbash.d/templates`. Il template deve contenere il placeholder `{{CONTENT}}` che verrà sostituito con il contenuto assemblato.

- `--batch <file>`  
  Modalità batch: processa ogni riga non vuota del file come prompt separato.

- Positional ARGS (array ARGS)  
  Argomenti posizionali raccolti come prompt o come elementi da espandere in CONTENT.

### Sezione: Modello e parametri di generazione
- `-m, --model <name>`  
  Seleziona il modello (ha la massima precedenza).

- `--system <text>`  
  Imposta il system prompt (campo provider-specific).

- `--ture <n>`  
  Alias per temperatura (nota: lo script usa internamente la variabile TURE). Deve essere numerico.

- `--temperature <n>`  
  Alias per temperatura; mappa alla stessa variabile interna (TURE/TEMPERATURE). Deve essere numerico.

- `--max <n>`  
  Imposta il numero massimo di token (MAX_TOKENS). Deve essere numerico.

### Sezione: Sessioni e conversazioni
- `--session <session_id>`  
  Abilita sessioning con l’ID fornito. Lo script valida l’ID con `session_validate_id` (pattern alfanumerico, punti, underscore, trattini, lunghezza massima). Quando attivo, lo script prepara BUILD_MESSAGES_FILE e usa session_read_window / session_append per leggere e scrivere la cronologia della sessione.

- `--session-window [<N>]`  
  Specifica la finestra (numero di messaggi) da leggere per la sessione. Se fornito senza argomento, il parser imposta `SESSION_WINDOW` a `10`. Se fornito con valore, usa il valore numerico. Lo script raccomanda valori <= 20 e emette un warning se > 20.

### Sezione: Output e formato
- `--json`  
  Output raw JSON (stampa il file RESP così com’è).

- `--pretty`  
  Pretty-print JSON (se RESP è JSON).

- `--text`  
  Output testuale (default).

- `--raw`  
  Output raw text senza newline aggiuntiva.

- `--out <path>`  
  Salva l’output in file o directory. Se `OUT_PATH` è una directory, lo script crea un file timestamped all’interno.

- `--threshold <bytes>`  
  Soglia in byte per il salvataggio automatico in history (THRESHOLD). Default 1000.

- `--save`  
  Forza il salvataggio dell’output testuale in history (FORCE_SAVE_MODE="save").

- `--nosave`  
  Disabilita il salvataggio dell’output (FORCE_SAVE_MODE="nosave").

### Sezione: Streaming e modalità di chiamata
- `--stream`  
  Abilita streaming SSE. Richiede che il provider implementi `call_api_streaming_<prov>()`. Incompatibile con `--json` e `--pretty` (lo script verifica e rifiuta la combinazione).

- `--no-stream`  
  Disabilita streaming esplicitamente.

### Sezione: Modalità interattive e provider
- `--provider <name>`  
  Se fornito con argomento (non vuoto e non iniziante con `-`), imposta `PROVIDER_CLI` e persiste la scelta in `groqbash.d/config/provider`. Se l’argomento è la stringa letterale `list`, il codice entra nel flusso di selezione interattiva (vedi comportamento nel parser).

- `--provider` (senza argomento)  
  Entra in modalità interattiva di selezione provider (TTY richiesto). Imposta `PROVIDER_INTERACTIVE`.

- `--install-extras [<src>]`  
  Installa gli extras dal percorso sorgente specificato (opzionale) nella directory runtime `GROQBASH_EXTRAS_DIR`. Supporta anche la forma `--install-extras=<src>`. Valida sorgente, proprietà e copia i file in modo atomico.

### Sezione: Azioni di gestione modelli e liste
- `--refresh-models`  
  Richiama il refresh dei modelli del provider (invoca `refresh_models_<prov>()` dopo che il provider è caricato e la chiave API è assicurata).

- `--refresh-model`  
  Alias accettato dallo script per `--refresh-models`.

- `--list-models`  
  Stampa la whitelist locale dei modelli in forma leggibile.

- `--list-models-raw`  
  Stampa la lista dei modelli in forma machine-readable (una riga per modello, normalizzati).

- `--list-providers`  
  Stampa i provider disponibili (embedded + installati) in forma leggibile.

- `--list-providers-raw`  
  Stampa i provider disponibili in forma machine-readable (una riga per provider).

- `--set-default <model>`  
  Persiste il modello di default per il provider attivo in `groqbash.d/config/model.<provider>`.

### Sezione: Debug, logging e comportamento
- `--dry-run`  
  Stampa l’anteprima del payload e salta la chiamata HTTP (DRY_RUN). Mostra il payload head.

- `--quiet`  
  Minimizza i messaggi runtime.

- `--debug`  
  Abilita logging verboso e preserva `RUN_TMPDIR` (non lo pulisce alla fine).

- `--show-config`  
  Stampa la configurazione attiva (provider, model, path, flags) e termina.

- `--diagnostics`  
  Esegue controlli diagnostici (permessi, provider, modello) e termina.

- `--version`  
  Stampa nome script, versione e data e termina.

- `-h, --help`  
  Mostra questo help e termina.

### Sezione: Fine opzioni
- `--`  
  End-of-options marker: tutto ciò che segue è trattato come argomento posizionale (ARGS).

5. Modalità operative
---------------------
Questa sezione descrive le modalità operative implementate nello script e il comportamento associato.

### Modalità: Single request (default)
- Flusso: assemblaggio del contenuto (ARGS, FILE_INPUTS, STDIN, TEMPLATE, JSON_INPUT), selezione modello, build payload tramite provider, invocazione API (streaming o non-streaming), estrazione risposta, finalizzazione output, salvataggio history condizionato, eventuale append sessione.
- Uso tipico: `groqbash "prompt"`.

### Modalità: Streaming (`--stream`)
- Requisito: provider deve implementare `call_api_streaming_<prov>()`.
- Comportamento: lo streaming viene eseguito via curl HTTP/1.1; il flusso raw è salvato in `RUN_TMPDIR/resp.raw`. Lo script estrae linee `data:` e filtra JSON validi, ricostruisce `resp.chunks.json` e `resp.text.txt`, e scrive `RESP` in modo atomico.
- Incompatibilità: `--stream` non può essere usato con `--json` o `--pretty`.

### Modalità: Batch (`--batch <file>`)
- Per ogni riga non vuota del file:
  - Imposta `CONTENT` alla riga.
  - Se `--session` è attivo, prepara BUILD_MESSAGES_FILE per la sessione.
  - Costruisce payload e invoca provider.
  - Stampa separatore `----` tra elementi.
- Non interrompe l’intero batch su singolo errore (il codice continua con log degli errori).

### Modalità: Interactive chat (`--chat`)
- Richiede TTY.
- Loop interattivo: legge linee utente, costruisce `CONTENT`, prepara BUILD_MESSAGES_FILE se `--session` è attivo, invoca provider (streaming o non-streaming), stampa risposta.
- Stato conversazione mantenuto in memoria per la sessione corrente; session_append può essere usato per persistere.

### Modalità: Interactive provider selection (`--provider` senza argomento)
- Richiede TTY.
- Mostra elenco provider disponibili (embedded + installati), consente selezione numerica, persiste scelta in `groqbash.d/config/provider`.

8. Sessioni (comportamento completo)
------------------------------------
Lo script implementa un sistema di sessioni basato su file NDJSON e su un meccanismo di windowing.

**Validazione ID sessione**
- `session_validate_id` verifica che l’ID sia non vuoto e rispetti il pattern `[A-Za-z0-9._-]{1,128}`.

**Lettura finestra messaggi**
- `session_read_window <session_id> <N> <out_file>`:
  - Legge l’NDJSON della sessione (`groqbash.d/history/sessions/<session_id>.ndjson`) e produce un file JSON con `{"messages":[...]}` contenente le ultime N entries.
  - Usa lock sul file di sessione per evitare letture parziali durante append concorrenti.
  - Normalizza ruoli (`user`, `assistant`, `system`) e costruisce messaggi compatibili con il provider.
  - Se il file non esiste o non è leggibile, può produrre un file vuoto o fallire silenziosamente (il CORE ha fallback).

**Append idempotente**
- `session_append <session_id> <role> <content> <meta_json>`:
  - Crea un record NDJSON compatto con timestamp, role, content e meta.
  - Usa marker basati su `message_id` per idempotenza cross-processo; se `message_id` è presente e marker esiste, l’append viene saltato.
  - Acquisisce lock esclusivo sul file di sessione per append atomico.
  - Salva file con permessi restrittivi (600).
  - Evita di appendare messaggi utente vuoti.

**BUILD_MESSAGES_FILE**
- Quando `--session` è attivo, il CORE prepara `BUILD_MESSAGES_FILE` in `RUN_TMPDIR` e lo esporta per il provider; il provider può usare questo file per costruire il payload.

**Fallback**
- Se `session_read_window` non produce messaggi, il CORE tenta di ricostruire la finestra con `tail` su NDJSON o, come ultima risorsa, crea un payload con il solo messaggio utente corrente.

9. History e rotazione
----------------------
**Salvataggio output**
- `save_to_history` salva output testuali in `groqbash.d/history` con nome `YYYYMMDD-HHMMSS-groq-output-$$.txt`.
- Usa scritture atomiche e lock per garantire consistenza.
- Rispetta `FORCE_SAVE_MODE` (`save`/`nosave`) e `THRESHOLD` per decidere il salvataggio automatico.

**Rotate history**
- `rotate_history` (attivabile con `GROQBASH_ROTATE_HISTORY=1`) applica:
  - Rimozione file più vecchi di `GROQBASH_HISTORY_KEEP_DAYS`.
  - Riduzione dello spazio totale a `GROQBASH_HISTORY_MAX_BYTES`.
  - Riduzione del numero di file a `GROQBASH_HISTORY_MAX_FILES`.
- Tutte le operazioni sono eseguite sotto lock `HISTORY_LOCK`.

10. Streaming (dettagli tecnici)
-------------------------------
**Requisiti**
- Provider deve implementare `call_api_streaming_<prov>()`.

**Comportamento**
- Lo streaming viene eseguito con `curl --http1.1` (opzione `--http1.1` usata nello script).
- Il flusso raw è salvato in `RUN_TMPDIR/resp.raw`.
- Lo script estrae linee che iniziano con `data:` e rimuove il prefisso.
- Filtra solo le linee che sono JSON validi; salva le JSON valide in `resp.valid.jsons`.
- Costruisce `resp.chunks.json` come array di chunk JSON e `resp.text.txt` come concatenazione dei frammenti di testo estratti.
- Scrive `RESP` in modo atomico con `resp.chunks.json` se sono presenti chunk validi; altrimenti, se `RESP_RAW` è JSON valido, lo copia in `RESP`.
- Durante lo streaming, i frammenti di testo estratti vengono emessi su stdout man mano che arrivano.

**File temporanei**
- `resp.raw`, `resp.lines`, `resp.valid.jsons`, `resp.chunks.json`, `resp.text.txt` sono creati in `RUN_TMPDIR` e rimossi quando appropriato (salvo debug mode).

11. Retry e gestione edge case API
----------------------------------
**Retry**
- `perform_request_once` implementa un ciclo di retry fino a `MAX_RETRIES` (default 3).
- Dopo ogni fallimento, il codice valuta il tipo di errore:
  - Se errore di rete (`GROQBASHERRCURL_FAILED`), tenta retry con backoff incrementale.
  - Se errore API (`GROQBASHERRAPI`), non retrya e fallisce immediatamente.

**Estrazione risposta**
- `extract_text_from_resp` tenta di estrarre testo da `RESP` usando:
  - `.choices[] | .message?.content // .delta?.content` e join con newline.
  - Fallback: se `RESP` non è JSON, `cat` del file.

**Edge case: "empty completion"**
- `detect_empty_edge_case` analizza `RESP` per rilevare il pattern di completamento vuoto:
  - Controlla `choices[0].message.content` o `choices[0].delta.content` (vuoto),
  - `choices[0].finish_reason == "stop"`,
  - `usage.completion_tokens == 1` (o valore equivalente).
- Se rilevato, imposta variabili diagnostiche globali:
  - `GROQBASH_EDGE_EMPTY=1`
  - `GROQBASH_EDGE_REQ_ID` (se presente in `RESP` come `.x_groq?.id` o `.id`)
  - `GROQBASH_EDGE_FINISH_REASON`
  - `GROQBASH_EDGE_COMPLETION_TOKENS`
- Il CORE emette **una sola volta** un log strutturato con questi dettagli in `perform_request_once` e poi fallisce con codice API appropriato.

12. Sicurezza e vincoli filesystem
----------------------------------
**Principi generali**
- Tutto lo stato runtime è sotto `GROQBASH_DIR` (default: `SCRIPTDIR/groqbash.d`).
- Nessun uso di `/tmp` di sistema: tutti i tmp sono sotto `GROQBASH_TMPDIR` e `RUN_TMPDIR`.
- Umask e permessi restrittivi: directory 700, file 600 dove applicabile.
- Nessun `eval` e nessuna esecuzione di contenuto generato dal modello.
- Tutte le scritture critiche (models, manifest, history) sono atomiche e protette da lock (flock).

**Provider modules**
- Non devono essere symlink.
- Devono passare `bash -n` prima di essere sourcati.
- Ownership e permessi controllati: non world-writable; preferibilmente owned dall’utente corrente.
- Il CORE verifica la presenza delle funzioni richieste e rifiuta provider incompleti.

**Extras repository vs runtime**
- `./extras/` nella root del repo è solo sorgente per `--install-extras`.
- A runtime, gli unici percorsi attendibili per extras/provider sono `GROQBASH_EXTRAS_DIR` e `PROVIDERS_DIR` sotto `GROQBASH_DIR`.
- Lo script impedisce installazioni che provochino copie annidate o che usino symlink come sorgente.

**Locking**
- Lock collocati vicino alle risorse:
  - `MODELS_LOCK` (es. `groqbash.d/models/models.lock`)
  - `HISTORY_LOCK` (es. `groqbash.d/history/history.lock`)
  - `TMP_LOCK` (es. `groqbash.d/tmp/tmp.lock`)
- `lock_exec` usa `flock -x -w <timeout>` e fallisce con codice 124 su timeout.

13. Esempi
----------
Esempi di invocazione (forme esatte riconosciute dal parser):

- Singolo prompt (text output):
  - `groqbash "Write a short summary of the following text"`

- Da file:
  - `groqbash -f prompt.txt`

- Pipe:
  - `echo "Summarize this" | groqbash --model my-text-model`

- Streaming (provider deve supportare streaming):
  - `groqbash --stream -m my-model "Generate code"`

- Dry-run (mostra payload e non chiama l’API):
  - `groqbash --dry-run "Show me the payload"`

- Install extras da sorgente:
  - `groqbash --install-extras /path/to/repo/extras`
  - oppure: `groqbash --install-extras=/path/to/repo/extras`

- Refresh modelli:
  - `groqbash --refresh-models`

- Provider interattivo:
  - `groqbash --provider` (TTY richiesto)

- Provider via argomento:
  - `groqbash --provider groq`

- Provider "list" (trattato come richiesta di selezione interattiva):
  - `groqbash --provider list`

- Session example:
  - `groqbash --session mysession --session-window 20 "Continue conversation"`

15. Troubleshooting
-------------------
- **Script esce immediatamente**: verificare che tutti i comandi obbligatori siano in PATH (vedi Requisiti).
- **Provider non trovato**: installare provider script in `groqbash.d/extras/providers/<prov>.sh` o usare `--install-extras`.
- **API calls falliscono in CI**: assicurarsi di esportare la chiave API (`GROQ_API_KEY` o la variabile specificata da `PROVIDER_API_ENV_<prov>`).
- **History non salvata**: controllare `GROQBASH_ROTATE_HISTORY`, `THRESHOLD` e permessi su `groqbash.d/history`.
- **Problemi con provider module**: verificare che il file non sia symlink, che passi `bash -n` e che non sia world-writable.
- **Lock timeout**: se si ricevono errori di lock timeout, aumentare `GROQBASH_LOCK_TIMEOUT_*` o investigare processi concorrenti che tengono lock.

16. Ambiguità rilevate nel codice
---------------------------------
Lo script è la fonte di verità; tuttavia, il parser e alcune scelte di naming introducono ambiguità che vengono qui segnalate in modo neutro e documentato:

1. **Opzione `--ture` vs `--temperature`**  
   - Lo script accetta entrambe le forme. Internamente usa la variabile `TURE` come nome principale e poi sincronizza con `TEMPERATURE`. Questo duplice naming è presente nel codice; entrambe le opzioni sono supportate e mappano allo stesso valore.

2. **`--json-input` con valore `-`**  
   - Il codice supporta `--json-input -` come segnale per leggere JSON da stdin. Questo comportamento è implementato ma richiede attenzione: quando `JSON_INPUT` è `-`, lo script cattura stdin in un file temporaneo e lo usa come sorgente JSON.

3. **`--session-window` senza argomento**  
   - Il parser consente `--session-window` senza valore e in quel caso imposta `SESSION_WINDOW="10"`. Questo comportamento è esplicitato nel parser e può essere fonte di confusione se l’utente si aspetta che l’opzione richieda sempre un argomento.

4. **`--provider list` vs `--provider` senza argomento**  
   - Se l’utente invoca `--provider list` (la parola `list` come argomento), il codice tratta la stringa `list` come richiesta di selezione interattiva. Inoltre, `--provider` senza argomento attiva la selezione interattiva. Entrambe le forme portano allo stesso flusso interattivo, ma la presenza della parola `list` come argomento è un caso speciale gestito dal parser.

5. **`--install-extras` forma con e senza argomento**  
   - Lo script supporta sia `--install-extras <src>` che `--install-extras=<src>` e anche `--install-extras` senza argomento (in questo caso usa `SCRIPTDIR/extras` come sorgente). Il comportamento è implementato ma la presenza di più forme è da considerare quando si automatizza l’uso.

Queste ambiguità sono riportate per chiarezza e per guidare l’uso corretto dello script; non sono errori del codice ma scelte di parsing che l’utente deve conoscere.

17. Note finali per auditor/integratori
---------------------------------------
- Lo script è progettato per essere auditabile: tutte le operazioni sensibili (I/O, lock, scritture) sono centralizzate in primitive atomiche e in funzioni di utilità.
- Provider esterni sono caricati tramite `source` dopo controlli di sintassi e permessi; il CORE comunica con loro esclusivamente tramite le funzioni dell’interfaccia minima (`buildpayload_<prov>`, `call_api_<prov>`, e le opzionali).
- Per integrazioni non-interattive, assicurarsi di esportare la chiave API corretta e di impostare `ALLOW_API_CALLS` se necessario (il codice impone un controllo se `GROQ_API_KEY` è presente ma `ALLOW_API_CALLS` non è 1 e `DRY_RUN` non è 1).
- Tutto lo stato runtime deve rimanere sotto `GROQBASH_DIR` (default `SCRIPTDIR/groqbash.d`).
