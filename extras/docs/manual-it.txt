4. Opzioni CLI (complete, raggruppate)
--------------------------------------
Questa sezione elenca **tutte** le opzioni e i flag riconosciuti dal parser dello script. Le descrizioni si basano strettamente sul comportamento implementato nello script.

Nota: l'ordine e la forma sono esattamente quelli gestiti dal parser.

### Sezione: Input e prompt
- `-f <file>`  
  Aggiunge un file come input. Può essere ripetuto per più file. I file vengono letti e concatenati con separatori quando appropriato.

- `--json-input <file>`  
  Usa un payload JSON grezzo da un file. Se il valore è `-`, lo script legge JSON da stdin e lo usa come JSON_INPUT. Quando fornito, `CONTENT` è ignorato e viene usato il payload JSON direttamente.

- `--template <name>`  
  Usa un template presente in `groqbash.d/templates`. Il template deve contenere il placeholder `{{CONTENT}}` che verrà sostituito con il contenuto assemblato.

- `--batch <file>`  
  Modalità batch: processa ogni riga non vuota del file come prompt separato.

- ARGS posizionali (array ARGS)  
  Argomenti posizionali raccolti come elementi del prompt o espansi in CONTENT.

### Sezione: Parametri modello e generazione
- `-m, --model <name>`  
  Seleziona il modello (ha la precedenza più alta).

- `--system <text>`  
  Imposta il system prompt (campo specifico del provider).

- `--ture <n>`  
  Alias per la temperatura (nota: lo script usa internamente la variabile TURE). Deve essere numerico.

- `--temperature <n>`  
  Alias per la temperatura; mappa alla stessa variabile interna (TURE/TEMPERATURE). Deve essere numerico.

- `--max <n>`  
  Imposta il numero massimo di token (MAX_TOKENS). Deve essere numerico.

### Sezione: Sessioni e conversazioni
- `--session <session_id>`  
  Abilita le sessioni con l'ID fornito. Lo script valida l'ID con `session_validate_id` (alfanumerico, punti, underscore, trattini, lunghezza massima). Quando attiva, lo script prepara BUILD_MESSAGES_FILE e usa session_read_window / session_append per leggere e scrivere la cronologia della sessione.

- `--session-window [<N>]`  
  Specifica la finestra (numero di messaggi) da leggere per la sessione. Se fornito senza argomento, il parser imposta `SESSION_WINDOW` a `10`. Se fornito con un valore, usa il valore numerico. Lo script raccomanda valori <= 20 e emette un avviso se > 20.

### Sezione: Output e formattazione
- `--json`  
  Output JSON grezzo (stampa il file RESP così com'è).

- `--pretty`  
  Pretty-print del JSON (se RESP è JSON).

- `--text`  
  Output testuale (predefinito).

- `--raw`  
  Testo grezzo senza newline aggiuntivo.

- `--out <path>`  
  Salva l'output in un file o directory. Se `OUT_PATH` è una directory, lo script crea un file con timestamp all'interno.

- `--threshold <bytes>`  
  Soglia in byte per il salvataggio automatico nella cronologia (THRESHOLD). Default 1000.

- `--save`  
  Forza il salvataggio dell'output testuale nella cronologia (FORCE_SAVE_MODE="save").

- `--nosave`  
  Disabilita il salvataggio dell'output (FORCE_SAVE_MODE="nosave").

### Sezione: Streaming e modalità di chiamata
- `--stream`  
  Abilita lo streaming SSE. Richiede che il provider implementi `call_api_streaming_<prov>()`. Incompatibile con `--json` e `--pretty` (lo script verifica e rifiuta la combinazione).

- `--no-stream`  
  Disabilita esplicitamente lo streaming.

### Sezione: Modalità interattive e provider
- `--provider <name>`  
  Se fornito con un argomento (non vuoto e non iniziando con `-`), imposta `PROVIDER_CLI` e persiste la scelta in `groqbash.d/config/provider`. Se l'argomento è la stringa letterale `list`, il codice entra nel flusso di selezione interattiva (vedi comportamento del parser).

- `--provider` (senza argomento)  
  Entra nella modalità di selezione provider interattiva (richiede TTY). Imposta `PROVIDER_INTERACTIVE`.

- `--install-extras [<src>]`  
  Installa extras dalla sorgente specificata (opzionale) nella directory runtime `GROQBASH_EXTRAS_DIR`. Supporta anche la forma `--install-extras=<src>`. Valida la sorgente, le proprietà e copia i file in modo atomico.

### Sezione: Azioni di gestione modelli e liste
- `--refresh-models`  
  Avvia l'aggiornamento dei modelli del provider (invoca `refresh_models_<prov>()` dopo che il provider è caricato e la chiave API è assicurata).

- `--refresh-model`  
  Alias accettato dallo script per `--refresh-models`.

- `--list-models`  
  Stampa la whitelist locale dei modelli in forma leggibile.

- `--list-models-raw`  
  Stampa la lista dei modelli in forma macchina (un modello normalizzato per riga).

- `--list-providers`  
  Stampa i provider disponibili (embedded + installati) in forma leggibile.

- `--list-providers-raw`  
  Stampa i provider disponibili in forma macchina (un provider per riga).

- `--set-default <model>`  
  Persiste il modello di default per il provider attivo in `groqbash.d/config/model.<provider>`.

### Sezione: Debug, logging e comportamento
- `--dry-run`  
  Stampa l'anteprima del payload e salta la chiamata HTTP (DRY_RUN). Mostra l'head del payload.

- `--quiet`  
  Minimizza i messaggi runtime.

- `--debug`  
  Abilita logging verboso e preserva `RUN_TMPDIR` (non lo pulisce alla fine).

- `--show-config`  
  Stampa la configurazione attiva (provider, modello, percorsi, flag) ed esce.

- `--diagnostics`  
  Esegue controlli diagnostici (permessi, provider, modello) ed esce.

- `--version`  
  Stampa nome script, versione e data ed esce.

- `-h, --help`  
  Mostra questa guida ed esce.

### Sezione: Fine opzioni
- `--`  
  Marcatore fine opzioni: tutto ciò che segue è trattato come argomenti posizionali (ARGS).

5. Modalità operative
---------------------
Questa sezione descrive le modalità operative implementate nello script e il comportamento associato.

### Modalità: Richiesta singola (default)
- Flusso: assemblare il contenuto (ARGS, FILE_INPUTS, STDIN, TEMPLATE, JSON_INPUT), selezionare il modello, costruire il payload tramite il provider, invocare l'API (streaming o non-streaming), estrarre la risposta, finalizzare l'output, salvare condizionalmente nella cronologia, opzionalmente appendere alla sessione.
- Uso tipico: `groqbash "prompt"`.

### Modalità: Streaming (`--stream`)
- Requisito: il provider deve implementare `call_api_streaming_<prov>()`.
- Comportamento: lo streaming è eseguito via curl HTTP/1.1; lo stream grezzo è salvato in `RUN_TMPDIR/resp.raw`. Lo script estrae le righe che iniziano con `data:` e filtra i JSON validi, ricostruisce `resp.chunks.json` e `resp.text.txt`, e scrive `RESP` in modo atomico.
- Incompatibilità: `--stream` non può essere usato con `--json` o `--pretty`.

### Modalità: Batch (`--batch <file>`)
- Per ogni riga non vuota nel file:
  - Imposta `CONTENT` sulla riga.
  - Se `--session` è attivo, prepara BUILD_MESSAGES_FILE per la sessione.
  - Costruisce il payload e invoca il provider.
  - Stampa il separatore `----` tra gli elementi.
- Non interrompe l'intero batch su un singolo errore (il codice continua con errori loggati).

### Modalità: Chat interattiva (`--chat`)
- Richiede TTY.
- Loop interattivo: legge le righe dell'utente, costruisce `CONTENT`, prepara BUILD_MESSAGES_FILE se `--session` è attivo, invoca il provider (streaming o non-streaming), stampa la risposta.
- Lo stato della conversazione è mantenuto in memoria per la sessione corrente; `session_append` può essere usato per persistere.

### Modalità: Selezione provider interattiva (`--provider` senza argomento)
- Richiede TTY.
- Mostra una lista dei provider disponibili (embedded + installati), permette la selezione numerica, persiste la scelta in `groqbash.d/config/provider`.

8. Sessioni (comportamento completo)
------------------------------------
Lo script implementa un sistema di sessioni basato su file NDJSON e un meccanismo di windowing.

**Validazione ID sessione**
- `session_validate_id` verifica che l'ID non sia vuoto e corrisponda al pattern `[A-Za-z0-9._-]{1,128}`.

**Lettura della finestra di messaggi**
- `session_read_window <session_id> <N> <out_file>`:
  - Legge il NDJSON di sessione (`groqbash.d/history/sessions/<session_id>.ndjson`) e produce un file JSON con `{"messages":[...]}` contenente gli ultimi N elementi.
  - Usa un lock sul file di sessione per evitare letture parziali durante append concorrenti.
  - Normalizza i ruoli (`user`, `assistant`, `system`) e costruisce messaggi compatibili con il provider.
  - Se il file non esiste o è illeggibile, può produrre un file vuoto o fallire silenziosamente (il CORE ha fallback).

**Append idempotente**
- `session_append <session_id> <role> <content> <meta_json>`:
  - Crea un record NDJSON compatto con timestamp, role, content e meta.
  - Usa marker basati su `message_id` per idempotenza cross-processo; se `message_id` è presente e il marker esiste, l'append viene saltato.
  - Acquisisce un lock esclusivo sul file di sessione per append atomico.
  - Salva il file con permessi restrittivi (600).
  - Evita di appendere messaggi utente vuoti.

**BUILD_MESSAGES_FILE**
- Quando `--session` è attivo, il CORE prepara `BUILD_MESSAGES_FILE` in `RUN_TMPDIR` ed lo esporta per il provider; il provider può usare questo file per costruire il payload.

**Fallback**
- Se `session_read_window` non produce messaggi, il CORE tenta di ricostruire la finestra con `tail` sul NDJSON o, come ultima risorsa, crea un payload contenente solo il messaggio utente corrente.

ADDED: Session cache (integrazione)
- **Posizione:** inserito dopo "Sessions (full behavior)".
- **Scopo:** cache di payload/risposte derivate dalla sessione per evitare ricalcoli.
- **API:** `session_cache_key(sid, params)`, `session_cache_get(key)`, `session_cache_set(key, ttl, payload)`, `session_cache_invalidate(key|sid)`.
- **Formato:** file di cache memorizzati sotto `${GROQBASH_CONFIG_DIR%/}/session_cache` con permessi 600. Formato file: **prima riga = expiry_epoch (intero)**; righe successive = payload JSON.
- **Comportamento:** `session_cache_get` rimuove i file scaduti e ritorna 0 su hit; `session_cache_set` scrive in modo atomico; `session_cache_invalidate` rimuove le voci quando sessione o provider cambiano. TTL applicato tramite expiry_epoch.

9. Cronologia e rotazione
----------------------
**Salvataggio output**
- `save_to_history` salva output testuali in `groqbash.d/history` con nomi come `YYYYMMDD-HHMMSS-groq-output-$$.txt`.
- Usa scritture atomiche e lock per garantire consistenza.
- Rispetta `FORCE_SAVE_MODE` (`save`/`nosave`) e `THRESHOLD` per decidere il salvataggio automatico.

**Rotate history**
- `rotate_history` (abilitato con `GROQBASH_ROTATE_HISTORY=1`) applica:
  - Rimozione dei file più vecchi di `GROQBASH_HISTORY_KEEP_DAYS`.
  - Riduzione dello spazio totale a `GROQBASH_HISTORY_MAX_BYTES`.
  - Riduzione del numero di file a `GROQBASH_HISTORY_MAX_FILES`.
- Tutte le operazioni sono eseguite sotto `HISTORY_LOCK`.

---  
ADDED: Manifest multimodale (integrazione)
- **Posizione:** inserito dopo "History and rotation".
- **Scopo:** gestire manifest multimodali e parti staged in modo sicuro.
- **Primitive:** `manifest_create(manifest_path)`, `manifest_add_part(manifest, srcfile, metadata)`, `manifest_read(manifest_path)`.
- **Staging:** le parti sono staged tramite `stage_b64(srcfile, dst.b64)`; ogni parte è memorizzata come artefatto `.b64`.
- **manifest.b64:** una rappresentazione base64 del manifest è scritta accanto al manifest JSON per trasporto/archiviazione.
- **Atomicità:** gli aggiornamenti del manifest avvengono sotto un lock del manifest usando `lock_exec` e `atomic_write`; i fallimenti devono lasciare lo stato consistente.

10. Streaming (dettagli tecnici)
-------------------------------
**Requisiti**
- Il provider deve implementare `call_api_streaming_<prov>()`.

**Comportamento**
- Lo streaming è eseguito con `curl --http1.1` (lo script usa l'opzione `--http1.1`).
- Lo stream grezzo è salvato in `RUN_TMPDIR/resp.raw`.
- Lo script estrae le righe che iniziano con `data:` e rimuove il prefisso.
- Filtra solo le righe che sono JSON validi; salva le righe JSON valide in `resp.valid.jsons`.
- Costruisce `resp.chunks.json` come array di chunk JSON e `resp.text.txt` come concatenazione dei frammenti di testo estratti.
- Scrive `RESP` in modo atomico usando `resp.chunks.json` se sono presenti chunk validi; altrimenti, se `RESP_RAW` è JSON valido, lo copia in `RESP`.
- Durante lo streaming, i frammenti di testo estratti sono emessi su stdout man mano che arrivano.

**File temporanei**
- `resp.raw`, `resp.lines`, `resp.valid.jsons`, `resp.chunks.json`, `resp.text.txt` sono creati in `RUN_TMPDIR` e rimossi quando appropriato (a meno che la modalità debug non li preservi).

11. Retry e gestione dei casi limite API
----------------------------------
**Retry**
- `perform_request_once` implementa un loop di retry fino a `MAX_RETRIES` (default 3).
- Dopo ogni fallimento, il codice valuta il tipo di errore:
  - Se errore di rete (`GROQBASHERRCURL_FAILED`), ritenta con backoff incrementale.
  - Se errore API (`GROQBASHERRAPI`), non ritenta e fallisce immediatamente.

**Estrazione risposta**
- `extract_text_from_resp` tenta di estrarre testo da `RESP` usando:
  - `.choices[] | .message?.content // .delta?.content` e unisce con newline.
  - Fallback: se `RESP` non è JSON, fa cat del file.

**Caso limite: "empty completion"**
- `detect_empty_edge_case` analizza `RESP` per rilevare il pattern di completamento vuoto:
  - Controlla `choices[0].message.content` o `choices[0].delta.content` (vuoto),
  - `choices[0].finish_reason == "stop"`,
  - `usage.completion_tokens == 1` (o equivalente).
- Se rilevato, imposta variabili diagnostiche globali:
  - `GROQBASH_EDGE_EMPTY=1`
  - `GROQBASH_EDGE_REQ_ID` (se presente in `RESP` come `.x_groq?.id` o `.id`)
  - `GROQBASH_EDGE_FINISH_REASON`
  - `GROQBASH_EDGE_COMPLETION_TOKENS`
- Il CORE emette **un unico log strutturato** con questi dettagli dentro `perform_request_once` e poi fallisce con il codice di uscita API appropriato.

12. Vincoli di sicurezza e filesystem
----------------------------------
**Principi generali**
- Tutto lo stato runtime è sotto `GROQBASH_DIR` (default: `SCRIPTDIR/groqbash.d`).
- Nessun uso del `/tmp` di sistema: tutti i tmp sono sotto `GROQBASH_TMPDIR` e `RUN_TMPDIR`.
- Umask e permessi restrittivi: directory 700, file 600 dove applicabile.
- Nessun `eval` e nessuna esecuzione di contenuto generato dal modello.
- Tutte le scritture critiche (models, manifest, history) sono atomiche e protette da lock (flock).

**Moduli provider**
- Non devono essere symlink.
- Devono passare `bash -n` prima di essere sourcati.
- Proprietà e permessi controllati: non world-writable; preferibilmente di proprietà dell'utente corrente.
- Il CORE verifica la presenza delle funzioni richieste e rifiuta provider incompleti.

**Extras repository vs runtime**
- `./extras/` nella root del repo è solo sorgente per `--install-extras`.
- A runtime, i percorsi attendibili per extras/providers sono `GROQBASH_EXTRAS_DIR` e `PROVIDERS_DIR` sotto `GROQBASH_DIR`.
- Lo script previene installazioni che creerebbero copie annidate o userebbero symlink come sorgente.

**Locking**
- I lock sono collocati vicino alle risorse:
  - `MODELS_LOCK` (es. `groqbash.d/models/models.lock`)
  - `HISTORY_LOCK` (es. `groqbash.d/history/history.lock`)
  - `TMP_LOCK` (es. `groqbash.d/tmp/tmp.lock`)
- `lock_exec` usa `flock -x -w <timeout>` e fallisce con codice 124 su timeout.

---  
ADDED: Network policy (enforce_network_policy)
- **Posizione:** suggerita inserzione sotto "Security and filesystem constraints".
- **Scopo:** gate centralizzato unico per tutto l'accesso di rete.
- **Comportamento:** `enforce_network_policy()` ritorna successo solo se le chiamate di rete sono permesse. Rispetta `DRY_RUN` (blocca le chiamate di rete reali), `GROQBASH_SKIP_NETWORK` (blocco esplicito) e `GROQBASH_ENFORCE_NO_NETWORK_IF_QUIET` combinato con `QUIET` (blocca quando la modalità quiet impone nessuna rete). Tutti i punti di chiamata provider **devono** chiamare `enforce_network_policy()` prima di eseguire curl; quando bloccato, i provider devono produrre un RESP diagnostico e ritornare non-zero. Questa funzione è il luogo canonico per implementare salvaguardie per CI/automation.

13. Esempi
----------
Esempi di invocazione (forme esatte riconosciute dal parser):

- Prompt singolo (output testuale):
  - `groqbash "Write a short summary of the following text"`

- Da file:
  - `groqbash -f prompt.txt`

- Pipe:
  - `echo "Summarize this" | groqbash --model my-text-model`

- Streaming (il provider deve supportare streaming):
  - `groqbash --stream -m my-model "Generate code"`

- Dry-run (mostra il payload e non chiama l'API):
  - `groqbash --dry-run "Show me the payload"`

- Installa extras da sorgente:
  - `groqbash --install-extras /path/to/repo/extras`
  - oppure: `groqbash --install-extras=/path/to/repo/extras`

- Refresh modelli:
  - `groqbash --refresh-models`

- Provider interattivo:
  - `groqbash --provider` (richiede TTY)

- Provider via argomento:
  - `groqbash --provider groq`

- Provider "list" (trattato come richiesta di selezione interattiva):
  - `groqbash --provider list`

- Esempio sessione:
  - `groqbash --session mysession --session-window 20 "Continue conversation"`

14. Codici di uscita
--------------------
Codici di uscita canonici usati dallo script (numeri fissi):

- `10` — `GROQBASH_ERR_NO_API_KEY` — Mancanza chiave API per il provider (non-interattivo).
- `11` — `GROQBASH_ERR_BAD_MODEL` — Nessun modello valido selezionato / modello non valido.
- `12` — `GROQBASH_ERR_CURL_FAILED` — Errore di rete / curl.
- `13` — `GROQBASH_ERR_INVALID_JSON` — JSON non valido o errore di parsing.
- `14` — `GROQBASH_ERR_NO_PROMPT` — Nessun prompt fornito.
- `15` — `GROQBASH_ERR_TMP` — Errore tmpdir / file temporaneo.
- `16` — `GROQBASH_ERR_API` — Errore HTTP/API dal provider.

15. Risoluzione problemi
-------------------
- **Lo script esce immediatamente**: verificare che tutti i comandi obbligatori siano in PATH (vedi Requisiti).
- **Provider non trovato**: installare lo script provider in `groqbash.d/extras/providers/<prov>.sh` o usare `--install-extras`.
- **Chiamate API falliscono in CI**: assicurarsi che la chiave API sia esportata (`GROQ_API_KEY` o la variabile specificata da `PROVIDER_API_ENV_<prov>`).
- **Cronologia non salvata**: controllare `GROQBASH_ROTATE_HISTORY`, `THRESHOLD` e i permessi su `groqbash.d/history`.
- **Problemi modulo provider**: verificare che il file non sia symlink, passi `bash -n` e non sia world-writable.
- **Timeout di lock**: se si ricevono errori di timeout lock, aumentare `GROQBASH_LOCK_TIMEOUT_*` o investigare processi concorrenti che tengono i lock.

16. Ambiguità trovate nel codice
---------------------------------
Lo script è la fonte di verità; tuttavia il parser e alcune scelte di naming introducono ambiguità che sono documentate in modo neutrale qui:

1. **Opzione `--ture` vs `--temperature`**  
   - Lo script accetta entrambe le forme. Internamente usa la variabile `TURE` come nome primario e poi la sincronizza con `TEMPERATURE`. Questa doppia denominazione esiste nel codice; entrambe le opzioni sono supportate e mappano allo stesso valore.

2. **`--json-input` con valore `-`**  
   - Il codice supporta `--json-input -` come segnale per leggere JSON da stdin. Questo comportamento è implementato ma richiede attenzione: quando `JSON_INPUT` è `-`, lo script cattura stdin in un file temporaneo e lo usa come sorgente JSON.

3. **`--session-window` senza argomento**  
   - Il parser permette `--session-window` senza valore e in quel caso imposta `SESSION_WINDOW="10"`. Questo comportamento è esplicito nel parser e può essere fonte di confusione se l'utente si aspetta che l'opzione richieda sempre un argomento.

4. **`--provider list` vs `--provider` senza argomento**  
   - Se l'utente invoca `--provider list` (la parola `list` come argomento), il codice tratta la stringa `list` come richiesta di selezione interattiva. Inoltre, `--provider` senza argomento attiva la selezione interattiva. Entrambe le forme portano allo stesso flusso interattivo, ma la presenza della parola `list` come argomento è un caso speciale gestito dal parser.

5. **Forma `--install-extras` con e senza argomento**  
   - Lo script supporta `--install-extras <src>`, `--install-extras=<src>` e anche `--install-extras` senza argomento (in questo caso usa `SCRIPTDIR/extras` come sorgente). Il comportamento è implementato ma la presenza di più forme dovrebbe essere considerata quando si automatizza l'uso.

Queste ambiguità sono riportate per chiarezza e per guidare l'uso corretto dello script; non sono errori di codice ma scelte di parsing di cui l'utente dovrebbe essere consapevole.

17. Note finali per auditor/integratori
---------------------------------------
- Lo script è progettato per essere auditabile: tutte le operazioni sensibili (I/O, lock, scritture) sono centralizzate in primitive atomiche e funzioni di utilità.
- I provider esterni sono caricati via `source` dopo controlli di sintassi e permessi; il CORE comunica con essi esclusivamente tramite le funzioni di interfaccia minime (`buildpayload_<prov>`, `call_api_<prov>`, e opzionali).
- Per integrazioni non-interattive, assicurarsi che la chiave API corretta sia esportata e impostare `ALLOW_API_CALLS` se necessario (il codice applica un controllo se `GROQ_API_KEY` è presente ma `ALLOW_API_CALLS` non è 1 e `DRY_RUN` non è 1).
- Tutto lo stato runtime deve rimanere sotto `GROQBASH_DIR` (default `SCRIPTDIR/groqbash.d`).
