SPECIFICA DELL'ARCHITETTURA E DELLA LOGICA DI BASH4LLM⁺ (v2.5.0)

Questo documento descrive in modo completo e senza perdita di informazioni la
logica, le variabili, le funzioni e i blocchi strutturali del sistema Bash4LLM⁺,
suddivisi secondo le cinque macro-sezioni originarie allineate alla versione
stabile 2.5.0.

1. PRECORE_BOOT

VARIABILI

  - SCRIPT_NAME: Nome identificativo del programma (impostato a "bash4llm").
  - SCRIPT_VERSION: Versione corrente del software (impostata a "2.5.0").
  - SCRIPT_DATE: Data di rilascio dello script (impostata a "2026-07-18").
  - BASH4LLM_PLAT_ANDROID: Flag di ambiente (valore 1 o 0) impostato su
    bootstrap indicante se il runtime è eseguito sotto ambiente Android/Termux.
  - BASH4LLM_PLAT_MACOS: Flag di ambiente (valore 1 o 0) impostato su bootstrap
    indicante se il runtime è eseguito sotto sistema macOS/Darwin.
  - BASH4LLM_PLAT_WSL: Flag di ambiente (valore 1 o 0) impostato su bootstrap
    indicante se il runtime è eseguito sotto Windows Subsystem for Linux (WSL).
  - BASH4LLM_PLAT_CYGWIN: Flag di ambiente (valore 1 o 0) impostato su bootstrap
    indicante se l'esecuzione avviene sotto ambienti Windows compatibili Cygwin,
    MSYS2 o MinGW.
  - BASH4LLM_PLAT_BSD: Flag di ambiente (valore 1 o 0) impostato su bootstrap
    indicante se l'esecuzione avviene sotto sistemi FreeBSD, OpenBSD, NetBSD o
    DragonFly.
  - BASH4LLM_PLAT_LINUX: Flag di ambiente (valore 1 o 0) impostato su bootstrap
    indicante se l'esecuzione avviene sotto ambiente Linux generico o WSL.
  - BASH4LLM_MODEL_CACHE: Array associativo interno adibito ad archiviare in
    memoria i nomi dei modelli normalizzati per velocizzare le esecuzioni
    successive.
  - BASH4LLM_ERR_NO_API_KEY: Costante numerica di errore (valore 10) associata
    alla mancanza di una chiave API.
  - BASH4LLM_ERR_BAD_MODEL: Costante numerica di errore (valore 11) associata
    alla richiesta di un modello non valido o non supportato.
  - BASH4LLM_ERR_CURL_FAILED: Costante numerica di errore (valore 12) associata
    al fallimento della chiamata di rete curl.
  - BASH4LLM_ERR_NO_PROMPT: Costante numerica di errore (valore 14) associata
    alla mancanza di un prompt di input.
  - BASH4LLM_ERR_TMP: Costante numerica di errore (valore 15) associata a
    problemi del file-system temporaneo o dei file di lock.
  - BASH4LLM_ERR_API: Costante numerica di errore (valore 16) associata a
    risposte d'errore o codici HTTP non validi restituiti dal provider API.
  - BASH4LLM_ERR_SEC: Costante numerica di errore (valore 17) associata a
    violazioni dei permessi sul file-system o ad altre falle di sicurezza.
  - BASH4LLMERR_NO_API_KEY: Mappatura alias diretta del valore di
    BASH4LLM_ERR_NO_API_KEY con notazione ad underscore.
  - BASH4LLMERR_BAD_MODEL: Mappatura alias diretta del valore di
    BASH4LLM_ERR_BAD_MODEL con notazione ad underscore.
  - BASH4LLMERR_CURL_FAILED: Mappatura alias diretta del valore di
    BASH4LLM_ERR_CURL_FAILED con notazione ad underscore.
  - BASH4LLMERR_NO_PROMPT: Mappatura alias diretta del valore di
    BASH4LLM_ERR_NO_PROMPT con notazione ad underscore.
  - BASH4LLMERR_TMP: Mappatura alias diretta del valore di BASH4LLM_ERR_TMP con
    notazione ad underscore.
  - BASH4LLMERR_API: Mappatura alias diretta del valore di BASH4LLM_ERR_API con
    notazione ad underscore.
  - BASH4LLMERR_SEC: Mappatura alias diretta del valore di BASH4LLM_ERR_SEC con
    notazione ad underscore.
  - DEBUG: Flag di configurazione per l'output di tracciamento dello sviluppo.
    Eredita il valore di ambiente ${DEBUG} o ${BASH4LLM_DEBUG}, o viene
    impostato su 0 (disattivo).
  - BASH4LLM_LOG: Percorso assoluto o relativo del file in cui registrare i log
    di tracciamento. Eredita ${BASH4LLM_LOG} (vuoto se disattivato).
  - SCRIPTDIR: Percorso assoluto della directory contenente lo script corrente,
    risolto dinamicamente tramite la funzione resolve_script_dir.
  - BASH4LLM_DIR: Directory principale di installazione dei dati e delle
    configurazioni di Bash4LLM⁺. Dichiarata dall'ambiente o derivata da
    SCRIPTDIR/bash4llm.d.
  - BASH4LLM_CONFIG_DIR: Directory dei file di configurazione utente
    persistenti. Impostata a BASH4LLM_CONFIG_DIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/config.
  - BASH4LLM_MODELS_DIR: Directory dei file relativi ai modelli supportati.
    Impostata a BASH4LLM_MODELS_DIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/models.
  - BASH4LLM_TEMPLATES_DIR: Directory contenente i file dei modelli di template.
    Impostata a BASH4LLM_TEMPLATES_DIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/templates.
  - BASH4LLM_HISTORY_DIR: Directory contenente lo storico e i log delle
    interazioni dello script. Impostata a BASH4LLM_HISTORY_DIR di ambiente o
    posizionata sotto $BASH4LLM_DIR/history.
  - BASH4LLM_TMPDIR: Directory radice per l'allocazione dei dati temporanei di
    Bash4LLM⁺. Impostata a BASH4LLM_TMPDIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/tmp.
  - CANONICAL_EXTRAS_DIR: Percorso della directory canonica delle estensioni,
    calcolato rimuovendo la barra finale da BASH4LLM_DIR e annettendo /extras.
  - LEGACY_EXTRAS_DIR: Percorso storico di fallback delle estensioni, relativo
    alla cartella dello script (SCRIPTDIR/extras).
  - BASH4LLM_EXTRAS_DIR: Directory attiva per l'accesso alle risorse aggiuntive
    dello script (assegnata a CANONICAL_EXTRAS_DIR).
  - PROVIDERS_DIR: Percorso assoluto o relativo dei moduli esterni dei provider
    (impostato su PROVIDERS_DIR di ambiente o, in alternativa, localizzato sotto
    la directory delle estensioni alla voce /providers).
  - THREAD_DIR: Directory dedicata alla registrazione dei log storici NDJSON dei
    thread (posizionata in $BASH4LLM_HISTORY_DIR/threads).
  - BASH4LLM_VAULT_ENABLED: Variabile di controllo per l'abilitazione (1) o
    disattivazione (0) del Key Vault OpenSSL. Prelevata dal file config o
    impostata di default a 1.
  - BASH4LLM_OPENSSL_ACTIVE: Flag booleano interno indicante se
    openssl-helper.sh è caricato con successo nel runtime (valore 1 o 0).
  - MODELS_FILE: File di testo locale in cui memorizzare la lista dei modelli
    recuperati, sincronizzato tramite sync_models_file_path con pattern
    $BASH4LLM_MODELS_DIR/.txt.
  - MAX_MODELS: Limite numerico massimo di modelli importabili o gestibili
    localmente. Eredita ${MAX_MODELS} o adotta il valore di default di 200.
  - PROVIDER_FILE: Percorso del file che persiste l'ultimo provider configurato
    dall'utente, determinato tramite canonical_provider_file.
  - MODELS_LOCK: Percorso del file di lock di sincronizzazione per l'accesso ai
    modelli, impostato su MODELS_LOCK di ambiente o localizzato sotto
    $BASH4LLM_MODELS_DIR/models.lock.
  - HISTORY_LOCK: Percorso del file di lock per l'accesso concorrente ai log di
    storico e sessione, impostato su HISTORY_LOCK di ambiente o localizzato
    sotto $BASH4LLM_HISTORY_DIR/history.lock.
  - TMP_LOCK: Percorso del file di lock esclusivo per la gestione sicura della
    directory temporanea dello script, impostato su TMP_LOCK di ambiente o
    localizzato sotto $BASH4LLM_TMPDIR/tmp.lock.
  - BASH4LLM_LOCK_TIMEOUT_TMP: Timeout massimo di attesa espresso in secondi per
    acquisire il lock temporaneo (eredita BASH4LLM_LOCK_TIMEOUT_TMP o
    imposta 10).
  - BASH4LLM_LOCK_TIMEOUT_MODELS: Timeout in secondi per acquisire il lock sui
    modelli (eredita BASH4LLM_LOCK_TIMEOUT_MODELS o imposta 10).
  - BASH4LLM_LOCK_TIMEOUT_HISTORY: Timeout in secondi per acquisire il lock
    sullo storico dei messaggi e dei thread (eredita
    BASH4LLM_LOCK_TIMEOUT_HISTORY o imposta 10).

FUNZIONI

  - resolve_script_dir
      - Scopo: Identifica la cartella reale in cui risiede lo script in
        esecuzione, tracciando in modo sicuro i collegamenti simbolici sul
        file-system.
      - Input: Nessuno (analizza la variabile speciale ${BASH_SOURCE[0]}).
      - Output: Restituisce su standard output il percorso assoluto normalizzato
        della directory.
      - Side-effects: Nessuno.
      - Dipendenze: Comandi readlink, dirname, cd, pwd, printf.
  - safe_mkdir
      - Scopo: Crea in sicurezza una directory con i permessi specificati,
        verificando che non coincida con un collegamento simbolico (mitigazione
        symlink attack).
      - Input: $1 (directory), $2 (opzionale: permessi ottali, default 700).
      - Output: Stato 0 o arresto immediato con codice d'uscita 15
        (BASH4LLM_ERR_TMP) se fallisce.
      - Side-effects: Verifica symlink, creazione dir, chmod.
      - Dipendenze: mkdir, chmod, printf.
  - check_required_arg
      - Scopo: Convalida la presenza del parametro CLI obbligatorio richiesto da
        un'opzione, verificando la prossimità del limite posizionale degli
        argomenti.
      - Input: $1 (opzione indagata), $2 (conteggio argomenti residui).
      - Output: Stato 0 o arresto con codice 15.
      - Dipendenze: log_error.
  - canonical_config_dir
      - Scopo: Normalizza il percorso della directory di configurazione
        rimuovendo eventuali barre finali ridondanti.
      - Input: Nessuno (analizza $BASH4LLM_CONFIG_DIR).
      - Output: Restituisce il percorso canonico su standard output.
      - Side-effects: Nessuno.
      - Dipendenze: printf.
  - canonical_provider_file
      - Scopo: Genera il percorso canonico del file di persistenza del provider
        selezionato.
      - Input: Nessuno.
      - Output: Restituisce su standard output il percorso del file /provider
        sotto la directory canonica di configurazione.
      - Side-effects: Richiama internamente canonical_config_dir.
      - Dipendenze: printf, canonical_config_dir.
  - canonical_model_file
      - Scopo: Genera il percorso del file in cui memorizzare il modello
        predefinito associato a un determinato provider.
      - Input: $1 (nome identificativo del provider).
      - Output: Restituisce su standard output il percorso del file formato
        /model. posizionato nella directory canonica di configurazione.
      - Side-effects: Richiama canonical_config_dir.
      - Dipendenze: printf, canonical_config_dir.
  - canonical_provider_url_file
      - Scopo: Genera il percorso canonico del file contenente l'endpoint o URL
        API specifico del provider attivo.
      - Input: Nessuno.
      - Output: Restituisce su standard output il percorso del file
        /provider-url posizionato nella directory canonica di configurazione.
      - Side-effects: Richiama o simula la logica di canonical_config_dir.
      - Dipendenze: type, canonical_config_dir, printf.
  - trim_space
      - Scopo: Rimuove spazi e tabulazioni iniziali e finali tramite costrutti
        di scomposizione nativi della shell (senza fork).
      - Input: $1 (stringa).
      - Output: Stringa pulita su stdout.
      - Dipendenze: printf.
  - sync_models_file_path
      - Scopo: Sincronizza il percorso globale di MODELS_FILE basandosi sul
        provider attivo, forzando una sanitizzazione protettiva contro directory
        traversal che esclude caratteri non alfanumerici (eccetto trattino e
        trattino basso).
      - Input: $1 (opzionale: provider).
      - Output: Esporta la variabile MODELS_FILE in ambiente.
      - Dipendenze: export.
  - validate_file_input
      - Scopo: Esegue la convalida di sicurezza di un file di input per scopi
        transazionali, verificando l'assenza di byte nulli, DEL o controlli C0
        incompatibili (tranne tab, CR, LF).
      - Input: $1 (percorso file).
      - Output: Codici stato: 0 (valido), 1 (percorso vuoto), 2 (non
        leggibile), 3 (0 byte), 4 (violazioni binarie).
      - Dipendenze: tr, head, file_size.
  - _normalize_model_name
      - Scopo: Esegue la normalizzazione dei nomi dei modelli rimuovendo spazi,
        ritorni a capo ed i prefissi models/ e . Per ottimizzare il runtime,
        conserva i risultati in un array associativo di cache in memoria.
      - Input: $1 (nome modello).
      - Output: Nome normalizzato su stdout.
      - Dipendenze: normalize_model_.
  - ensure_api_key_for_provider
      - Scopo: Convalida la presenza di una chiave API valida in ambiente per il
        provider selezionato. Tenta di recuperare la chiave dal Key Vault
        crittografico OpenSSL se attivo ($BASH4LLM_OPENSSL_ACTIVE=1) tramite
        decrittazione in memoria. In sessioni non interattive esce con errore se
        assente. In sessioni interattive (TTY), richiede l'inserimento della
        chiave all'utente dallo standard input, ripulisce la stringa rimuovendo
        comandi ridondanti e la esporta nell'ambiente visualizzando istruzioni
        di salvataggio.
      - Input: $1 (nome identificativo del provider).
      - Output: Ritorna stato 0 se la chiave è presente o acquisita
        correttamente, altrimenti restituisce il codice di errore
        BASH4LLM_ERR_NO_API_KEY. Esporta la chiave API nella variabile
        d'ambiente corretta (es. GROQ_API_KEY o variabile personalizzata
        indiretta).
      - Side-effects: Esporta variabili d'ambiente globali; effettua letture
        dallo standard input, decrittazioni tramite openssl e scritture su
        stderr.
      - Dipendenze: provider_api_env_var_name, vault_exists, vault_load_keys,
        jq, printf, export.
  - enforce_network_policy
      - Scopo: Valuta se lo stato corrente del runtime permette l'esecuzione di
        richieste di rete reali o se queste devono essere bloccate.
      - Input: Nessuno (analizza i flag $DRY_RUN, $BASH4LLM_SKIP_NETWORK,
        $BASH4LLM_ENFORCE_NO_NETWORK_IF_QUIET, $QUIET).
      - Output: Ritorna lo stato 1 se le connessioni esterne sono bloccate (o se
        si è in simulazione dry-run), 0 se le chiamate di rete sono autorizzate.
      - Side-effects: Scrittura di log di debug se attiva la modalità di
        tracciamento.
      - Dipendenze: is_truthy, log_info.
  - _extract_notes_section
      - Scopo: Estrae sezioni testuali letterali (senza esecuzione di codice)
        dal file di documentazione core-notes.sh.
      - Input: $1 (titolo sezione target).
      - Output: Righe estratte su stdout o stato 1 se assente.
      - Dipendenze: awk.
  - log_prefix
      - Scopo: Restituisce il prefisso di intestazione unificato per i log dello
        script.
      - Input: Nessuno.
      - Output: Restituisce su standard output la stringa formattata bash4llm:
        <SCRIPT_NAME>: .
      - Dipendenze: printf.
  - log_info
      - Scopo: Traccia e visualizza i messaggi informativi.
      - Input: $1 (codice categoria/modulo, default "INFO"), $2 (corpo del
        messaggio).
      - Output: Scrive il messaggio formattato su stderr se $DEBUG è attivo. Se
        $BASH4LLM_LOG è valorizzato, appende il record con timestamp UTC e
        categoria nel file di destinazione.
      - Side-effects: Scrittura su console e file-system.
      - Dipendenze: log_prefix, date, printf.
  - log_warn
      - Scopo: Traccia e visualizza a schermo i messaggi di avvertimento non
        critici.
      - Input: $1 (codice modulo, default "WARN"), $2 (corpo del messaggio).
      - Output: Scrive il messaggio su stderr. Se abilitato il log, appende il
        tracciamento nel file $BASH4LLM_LOG.
      - Side-effects: Scrittura su stderr e file-system.
      - Dipendenze: log_prefix, date, printf.
  - log_error
      - Scopo: Registra ed evidenzia gli errori critici.
      - Input: $1 (codice modulo, default "ERROR"), $2 (corpo del messaggio).
      - Output: Scrive l'errore su stderr. Se abilitato il log, appende il
        tracciamento nel file $BASH4LLM_LOG.
      - Side-effects: Scrittura su stderr e file-system.
      - Dipendenze: log_prefix, date, printf.
  - log_info_user
      - Scopo: Registra messaggi informativi di esecuzione destinati all'utente
        (non silenziati da QUIET) su stderr e file log.
      - Input: $1 (modulo), $2 (messaggio).
      - Dipendenze: log_prefix, date, printf.
  - dbg
      - Scopo: Visualizza messaggi di diagnostica rapida su stderr se il debug è
        attivo.
      - Input: Argomenti testuali liberi da stampare ($*).
      - Output: Righe di testo inviate a stderr.
      - Dipendenze: printf.
  - ensure_config_dir
      - Scopo: Assicura la presenza e la sicurezza della cartella di
        configurazione utente. Crea la cartella se mancante via safe_mkdir,
        imposta i permessi restrittivi 700 e convalida l'effettiva scrivibilità
        scrivendo e rimuovendo immediatamente un file di test.
      - Input: Nessuno.
      - Output: Ritorna stato 0 se la directory è scrivibile e pronta, 1 in caso
        di errore.
      - Side-effects: Creazione directory, test di scrittura sul disco,
        alterazione permessi.
      - Dipendenze: safe_mkdir, chmod, rm, log_error.
  - write_provider_url_if_missing
      - Scopo: Memorizza in modo permanente l'URL delle API per un determinato
        provider all'interno del file dedicato, a patto che il file non sia già
        popolato. Per evitare scritture corrotte, la scrittura avviene in modo
        transazionale su un file temporaneo, poi spostato o copiato nella
        destinazione finale ed impostato con permessi 600.
      - Input: $1 (nome identificativo del provider), $2 (stringa dell'URL).
      - Output: Ritorna stato 0 in caso di successo, 1 se falliscono le
        operazioni su file-system.
      - Side-effects: Creazione directory, file temporanei, scrittura atomica,
        modifica permessi del file.
      - Dipendenze: canonical_provider_url_file, dirname, safe_mkdir, mktemp,
        printf, mv, cp, rm, chmod.
  - resolve_provider_url
      - Scopo: Identifica ed esporta l'URL di connessione per le API. Controlla
        prioritariamente se l'utente ha impostato variabili d'ambiente esplicite
        (BASH4LLM_API_URL o BASH4LLM_PROVIDER_URL). In assenza di esse, legge la
        prima riga del file canonico dell'URL del provider. Se il file non
        esiste e il provider attivo è "groq", esporta l'endpoint nativo di
        default di Groq.
      - Input: $1 (nome provider, default $PROVIDER).
      - Output: Ritorna stato 0 se l'URL viene risolto ed esportato nella
        variabile globale $BASH4LLM_PROVIDER_URL, 1 in caso di fallimento.
      - Side-effects: Esporta la variabile d'ambiente $BASH4LLM_PROVIDER_URL.
      - Dipendenze: canonical_provider_url_file, sed, awk, export.
  - provider_api_env_var_name
      - Scopo: Calcola il nome standardizzato della variabile d'ambiente
        deputata a contenere la chiave API di un determinato provider (es:
        "gemini" -> "GEMINI_API_KEY") sfruttando l'espansione nativa della
        shell.
      - Input: $1 (nome del provider).
      - Output: Restituisce su standard output il nome normalizzato (tutto in
        maiuscolo, convertendo i caratteri speciali non alfanumerici in trattini
        bassi _ e annettendo il suffisso _API_KEY).
      - Dipendenze: printf.
  - is_valid_json_string
      - Scopo: Valida se la stringa in input contiene una struttura
        sintatticamente corretta in formato JSON.
      - Input: $1 (stringa da analizzare).
      - Output: Ritorna stato 0 se il JSON è valido, 1 altrimenti.
      - Dipendenze: printf, jq.
  - b64encode
      - Scopo: Esegue la codifica di flussi in formato Base64 leggendo da
        standard input ed unificando l'output su un'unica riga priva di
        caratteri di a capo, integrando openssl enc.
      - Input: Flusso dati in ingresso su stdin.
      - Output: Stringa codificata inviata a stdout. Ritorna stato 0 o 1 in caso
        di errore.
      - Dipendenze: openssl, base64, tr.
  - b64decode
      - Scopo: Esegue la decodifica in formato Base64 di flussi di dati letti da
        standard input, avvalendosi dei flag ottimali rilevati all'avvio e
        correggendo le anomalie di troncamento per LibreSSL/BSD tramite folding
        righe a 64 caratteri via awk.
      - Input: Stringa codificata Base64 da stdin.
      - Output: Flusso decodificato su stdout.
      - Dipendenze: awk, openssl, base64 (configurato con $B64_DECODE_OPT).
  - is_truthy
      - Scopo: Valuta se il parametro testuale in ingresso rappresenta un valore
        di veridicità affermativo logico.
      - Input: $1 (stringa da valutare).
      - Output: Ritorna stato 0 (vero) se il parametro corrisponde a 1, true,
        TRUE, True, yes, YES o Yes. In tutti gli altri casi ritorna 1 (falso).
  - file_size
      - Scopo: Restituisce la dimensione esatta in byte di un file in modo
        compatibile sia con sistemi Linux sia macOS/BSD sfruttando i flag di
        piattaforma.
      - Input: $1 (percorso del file).
      - Output: Restituisce su standard output il valore numerico dei byte (o 0
        se il file è mancante o vuoto).
      - Dipendenze: stat, printf.
  - is_valid_json_file
      - Scopo: Valida se the file indicato esiste, contiene dati ed è
        strutturato come JSON conforme.
      - Input: $1 (percorso del file).
      - Output: Ritorna stato 0 se valido, 1 altrimenti.
      - Dipendenze: jq.
  - stage_b64
      - Scopo: Acquisisce un payload testuale (da file o standard input),
        verifica che la sua dimensione non ecceda la soglia massima consentita
        ($MAX_STAGE_BYTES, default 10MB), ne esegue in sicurezza la codifica
        Base64 scrivendola in un file temporaneo protetto, e infine sposta in
        modo atomico (tramite lock della directory) il file temporaneo nel
        percorso finale di staging con permessi 600.
      - Input: $1 (opzionale: percorso del file payload di origine; se omesso
        legge da stdin), $2 (percorso del file di destinazione finale codificato
        in Base64). Se viene passato un singolo argomento, questo viene trattato
        come destinazione e l'input viene letto da stdin.
      - Output: Ritorna stato 0 se il processo transazionale si conclude
        correttamente, 1 in caso di fallimento o di superamento dei limiti di
        dimensione.
      - Side-effects: Scrittura e cancellazione di file temporanei intermedi sul
        disco, modifica permessi, acquisizione concorrente di lock.
      - Dipendenze: dirname, safe_mkdir, _tmpf, cat, rm, file_size, base64, tr,
        chmod, lock_exec, mv, log_error, log_info.
  - lock_exec
      - Scopo: Esegue un comando arbitrario garantendo mutua esclusione e
        protezione concorrente su file-system tramite lock gestito con flock o
        tramite directory lock provvisorio in base a piattaforma, umask ed mtime
        orfani, bloccando attacchi basati su symlink. Se non riesce ad acquisire
        il lock entro il limite di tempo impostato, fallisce in modo controllato
        senza eseguire il comando.
      - Input: $1 (percorso del file di lock), $2 (valore numerico del timeout
        in secondi, default 10), seguiti dal delimitatore -- e dal comando con i
        relativi argomenti da avviare.
      - Output: Restituisce lo stato di uscita del comando eseguito, 124 in caso
        di scadenza del timeout senza acquisizione del lock, o 15 in caso di
        symlink rilevati.
      - Side-effects: Creazione directory del lock, blocco concorrente del
        file-system, rimozione lockdir orfana per mtime.
      - Dipendenze: dirname, safe_mkdir, flock, printf, mkdir, rmdir, log_error.
  - _mktemp_in_dir
      - Scopo: Crea in sicurezza un file temporaneo univoco all'interno della
        cartella specificata.
      - Input: $1 (cartella radice), $2 (opzionale: prefisso del nome, default
        "groq").
      - Output: Restituisce su standard output il percorso completo del file
        temporaneo creato, altrimenti restituisce il codice BASH4LLM_ERR_TMP se
        fallisce.
      - Dipendenze: _tmpf, log_error.
  - show_payload_head
      - Scopo: Estrae e visualizza a schermo le prime righe del file di payload
        inviato alle API a scopo diagnostico di debug. Se il file di origine
        risulta codificato in Base64 (estensione .b64), esegue la decodifica
        istantanea prima dell'estrazione delle righe.
      - Input: $1 (percorso file payload, default $PAYLOAD), $2 (numero massimo
        di righe, default 200).
      - Output: Stampa le righe decodificate su stderr.
      - Side-effects: Lettura, eventuale decodifica di flussi.
      - Dipendenze: printf, grep, b64decode, head.
  - atomic_write
      - Scopo: Scrive o sovrascrive un file in modo atomico, transazionale e
        sicuro. Riceve un flusso di dati da stdin, lo scrive in un file
        temporaneo posizionato all'interno della stessa directory di
        destinazione, imposta i permessi di sicurezza a 600, acquisisce un lock
        esclusivo di protezione concorrente (se lock_exec è disponibile) e
        sposta in modo definitivo il file temporaneo sovrascrivendo l'originale.
        Questa sequenza impedisce letture di file parziali o corrotti in caso di
        interruzioni impreviste o accessi paralleli.
      - Input: $1 (percorso del file di destinazione definitivo), $2 (timeout
        del lock, default 10).
      - Output: Ritorna stato 0 se completato, altrimenti restituisce il
        relativo codice di errore di fallimento o di lock.
      - Side-effects: Creazione directory genitrice se mancante via safe_mkdir,
        allocazione e rimozione di file temporanei intermedi, lock,
        sovrascrittura, alterazione permessi.
      - Dipendenze: dirname, safe_mkdir, mktemp, cat, rm, chmod, lock_exec, mv,
        log_error.
  - extract_text_from_resp
      - Scopo: Estrae in modo flessibile e resiliente il testo di risposta
        principale dai JSON restituiti dalle API del provider LLM. Esegue una
        copia isolata del file di risposta, verifica che il file sia un JSON
        strutturato valido e non sia marcato come file diagnostico. Tenta
        l'estrazione scansionando diverse varianti di percorsi e formati
        standardizzati: strutture chat completions OpenAI (delta o messaggi in
        .choices[].message.content o .choices[].delta.content), formati di
        completamento testuale piatti (.choices[].text), formati personalizzati
        (.output_text o .data[].text) o, in ultima istanza, estrae e concatena
        ricorsivamente tutti i valori scalari di tipo stringa presenti nel JSON.
      - Input: Nessuno (analizza il file puntato da $RESP).
      - Output: Restituisce su standard output il testo decodificato se
        individuato (ritornando stato 0). Ritorna 1 se non viene trovato
        testo, 2 se il file analizzato contiene dati diagnostici o è corrotto.
      - Side-effects: Allocazione e rimozione di file di copia temporanei sul
        disco per isolamento.
      - Dipendenze: _tmpf, cp, is_valid_json_file, jq, log_warn, log_error, rm.
  - ensure_run_tmpdir

        ($RUN_TMPDIR) dedicato in modo esclusivo alla singola istanza o

      - Input: $1 (opzionale: --print per stampare esclusivamente il percorso su

      - Output: Ritorna stato 0 ed esporta le variabili $RUN_TMPDIR, $PAYLOAD,

      - Output: Rimozione dei file dal disco, rilascio lock e ripristino echo

      - Dipendenze: stty, release_thread_lock, rm, log_info.
  - b64_atomic_write
      - Scopo: Esegue la codifica Base64 del flusso di dati letto da standard

      - Input: $1 (file di destinazione), $2 (timeout del lock, default 10).
      - Output: Ritorna stato 0 o il codice di errore BASH4LLM_ERR_TMP.
      - Side-effects: Creazione directory, allocazione file temporaneo,

      - Dipendenze: dirname, mkdir, _tmpf, b64encode, chmod, lock_exec, mv, rm,

  - b64_atomic_read
      - Scopo: Legge un file cifrato o codificato in Base64 in modo sicuro, ne

      - Input: $1 (percorso del file codificato Base64).
      - Output: Restituisce il contenuto decodificato su standard output.
      - Dipendenze: b64decode.
  - ui_state_write
      - Scopo: Scrive in modo atomico lo stato corrente dell'interfaccia utente

      - Input: $1 (nome del file di stato), $2 (stringa del contenuto JSON).
      - Output: File di stato creato o aggiornato. Ritorna stato 0 se

      - Side-effects: Creazione della directory degli stati se non presente,

      - Dipendenze: safe_mkdir, chmod, atomic_write, wc, log_info, log_warn.
  - run_static_config_check
      - Scopo: Linter di conformità statica della configurazione. Convalida che

      - Input: Nessuno.
      - Output: Stato 0 o esce con codice 17 (BASH4LLM_ERR_SEC).
      - Side-effects: Analisi permessi file e parsing variabili d'ambiente.
      - Dipendenze: stat, awk, sed, tr, grep, printf.
  - explain_error_code

      - Input: $1 (codice d'errore o alias).

  - load_provider_module
      - Scopo: Carica ed importa in modo sicuro e isolato i moduli dei provider

        l'inclusione per prevenire attacchi di tipo "Race Condition" o "File

        provvisorio del modulo all'interno di una sotto-shell, cattura le

      - Input: $1 (nome identificativo del provider).
      - Output: Ritorna stato 0 se il caricamento ed i controlli di sicurezza si

      - Scopo: Rileva a runtime le opzioni compatibili con l'eseguibile base64

      - Output: Esporta le variabili $B64_WRAP_OPT (es. -w0 o vuoto) e

      - Dipendenze: printf, base64, grep.
  - list_files_sorted_by_mtime
      - Scopo: Elenca tutti i file regolari all'interno di una directory

      - Input: $1 (directory da analizzare).
      - Output: Lista di coppie formato timestamp|percorso_file separate da

      - Dipendenze: find, stat, sort.

      - Scopo: Esegue l'inversione dell'ordine delle righe di un file

BLOCCHI DI CODICE

  - PRECORE_BOOT_SETUP_SHELL
      - Fase: Inizializzazione primaria assoluta dello script.
      - Azione: Imposta i parametri operativi restrittivi della shell Bash (set
        -euo pipefail) solo se eseguito direttamente per evitare shell
        pollution. Disabilita core dumps (ulimit -c 0), esporta le proprietà di
        identità del software (v2.5.0 e data 18 Luglio 2026).
  - PRECORE_BOOT_ENVIRONMENT_AUTODETECTION
      - Fase: Rilevamento piattaforma.
      - Azione: Rileva dinamicamente la piattaforma host (Android, macOS, WSL,
        Cygwin, BSD, Linux) ed esporta globalmente i corrispondenti flag
        booleani BASH4LLM_PLAT_... Inizializza l'associativa
        BASH4LLM_MODEL_CACHE.
  - PRECORE_BOOT_SETUP_ENV_CMDS
      - Fase: Validazione dei requisiti software di sistema.
      - Azione: Esegue una scansione di sistema per accertarsi che tutte le
        utilità essenziali richieste siano installate nel percorso di sistema
        (escluso flock per portabilità macOS/BSD/Termux, includendo comm). Se
        una o più utilità risultano assenti, emette un messaggio di errore
        fatale su stderr ed esce con stato 1.
  - PRECORE_BOOT_DIR_PATH
      - Fase: Determinazione della directory di runtime.
      - Azione: Risolve il percorso radice dell'installazione del software
        ($BASH4LLM_DIR). Adotta in ordine di priorità la cartella definita in
        $BASH4LLM_DIR, in alternativa compila il percorso tramite $BASH4LLM_ROOT
        (derivandone la directory bash4llm.d), o calcola la directory
        predefinita puntando a $SCRIPTDIR/bash4llm.d. Crea transazionalmente e
        in sicurezza la gerarchia di directory operative tramite safe_mkdir

  - PRECORE_BOOT_CRYPTOGRAPHIC_VAULT_LOAD
      - Fase: Integrazione OpenSSL Key Vault.
      - Azione: Se BASH4LLM_VAULT_ENABLED=1 e openssl-helper.sh presente (e non
        symlink), esegue il caricamento protetto del modulo ed esporta
        BASH4LLM_OPENSSL_ACTIVE=1.
  - PRECORE_BOOT_CLI_HELPERS
      - Fase: Intercettazione opzioni diagnostiche a riga di comando.
      - Azione: Cattura ed esegue immediatamente diagnostic offline utili
        (--check-config, --explain-error , ispezioni dei percorsi) terminando lo
        script prima del boot complessivo. Imposta umask 077.
  - PRECORE_BOOT_DETECT_B64_OPTS
      - Fase: Configurazione runtime delle dipendenze di codifica.
      - Azione: Invoca la funzione _detect_base64_opts per configurare le
        opzioni corrette di Base64 in base all'architettura di sistema ospite.

2. PRECORE_RUN

VARIABILI

  - BASH4LLM_ROTATE_HISTORY: Flag di configurazione (valore 1 o 0, default 0)
    per attivare la rotazione automatica periodica dello storico delle
    interazioni.
  - BASH4LLM_HISTORY_MAX_FILES: Soglia numerica massima di file memorizzabili
    nello storico delle risposte (default 100). Superata la soglia, i file più
    vecchi vengono rimossi.
  - BASH4LLM_HISTORY_MAX_BYTES: Soglia massima espressa in byte del peso
    cumulativo di tutti i file storici archiviati (default 104857600,
    equivalente a 100MB).
  - BASH4LLM_HISTORY_KEEP_DAYS: Limite di tempo espresso in giorni per la
    conservazione dei log e dello storico (default 90). I file più vecchi della
    soglia vengono eliminati.
  - THREAD_CACHE_DIR: Directory destinata a contenere la cache temporanea dei
    dati e dei messaggi dei thread attivi (localizzata sotto la directory di
    configurazione utente, alla voce thread_cache/).
  - CONTENT: Prompt testuale primario dell'utente (assegnato a CONTENT).
  - PROVIDER: Identificatore del provider API attivo (default "groq").
  - JSON_INPUT: Input strutturato in formato JSON fornito dall'utente
    (sovrascrive o definisce i messaggi).
  - THREAD_ID: Identificatore alfanumerico del thread di chat corrente
    (utilizzato per raggruppare e persistere il contesto dei messaggi in
    interazioni multi-turno).
  - THREAD_WINDOW: Numero massimo di messaggi storici da estrarre e accodare dal
    log di thread per comporre la finestra di contesto della chat.
  - TEMPLATE: Nome identificativo del file di template da applicare per
    formattare o arricchire il prompt dell'utente prima dell'invio.
  - BATCH_FILE: Percorso di un file contenente una serie ordinata di prompt (uno
    per riga) da elaborare in sequenza.
  - CHAT_MODE: Flag booleano (valore 1 o 0, default 0) che abilita l'interfaccia
    interattiva di chat multi-turno da terminale.
  - SET_DEFAULT_MODEL: Nome del modello da memorizzare e configurare come scelta
    predefinita persistente per il provider attivo.
  - REFRESH_MODELS: Flag booleano (valore 1 o 0, default 0) per forzare
    l'allineamento e l'interrogazione delle API esterne al fine di aggiornare la
    lista locale dei modelli supportati.
  - LIST_MODELS: Flag booleano (valore 1 o 0, default 0) per visualizzare a
    schermo l'elenco dei modelli disponibili.
  - FORCE_SAVE_MODE: Regola il salvataggio dell'output testuale generato sul
    disco (save forza il salvataggio, nosave lo disabilita, 0 adotta il
    comportamento standard basato sulla soglia).
  - OUT_PATH: Percorso del file o della directory personalizzata in cui
    archiviare i risultati restituiti dalle API.
  - SYSTEM_PROMPT: Prompt di sistema personalizzato utilizzato per istruire o
    condizionare il comportamento e il tono del modello LLM.
  - TEMPERATURE: Valore numerico decimale che regola la temperatura di
    campionamento delle risposte delle API (eredita ${TEMPERATURE}, ${TURE} o
    adotta 1.0). Sincronizzato con TURE.
  - TURE: Temperatura di campionamento delle risposte delle API (sincronizzato
    con TEMPERATURE).
  - MAX_TOKENS: Limite di token generabili nella risposta (default 4096).
  - MODEL: Nome identificativo del modello LLM richiesto per l'elaborazione
    (ereditato o impostato via CLI o config).
  - AUTO_POLICY: Politica adottata per la selezione automatica del modello di
    fallback in assenza di un modello espresso esplicitamente dall'utente
    (default "preferred").
  - QUIET: Flag booleano (valore 1 o 0, default 0) per inibire i messaggi
    informativi e limitare l'output ai soli risultati o errori bloccanti.
  - DRY_RUN: Flag booleano (valore 1 o 0, default 0) per simulare l'esecuzione
    ed escludere l'invio reale di chiamate di rete esterne.
  - STREAM_MODE: Flag booleano (valore 1 o 0, default 0) per attivare la
    ricezione e la visualizzazione a schermo dei token in tempo reale
    (streaming).
  - OUTPUT_MODE: Configura lo stile di visualizzazione dei risultati restituiti
    dallo script (text, raw, json, pretty).
  - THRESHOLD: Soglia limite espressa in caratteri al di sopra della quale i
    risultati delle API vengono salvati nello storico (default 1000).
  - MAX_RETRIES: Limite di tentativi di connessione da effettuare in caso di
    errori di rete temporanei (default 1 nella versione v2.5.0).
  - SUPPORTED_PROVIDERS: Elenco testuale dei provider supportati (default "groq
    gemini huggingface").
  - CURL_BASE_OPTS: Array contenente i parametri base immutabili passati a curl
    per ogni chiamata di rete (--silent, --show-error, --no-buffer,
    --max-time 120).

FUNZIONI

  - rotate_history
      - Scopo: Gestisce la manutenzione della directory dello storico delle
        risposte per evitare saturazioni del disco. Identifica ed elimina i file
        più vecchi del limite stabilito in giorni ($BASH4LLM_HISTORY_KEEP_DAYS),
        ed esegue riduzioni progressive basate su file e byte complessivi
        scansionando il file-system tramite cicli mtime ed array di mappatura
        dedicati, agendo sotto lock esclusivo di storico.
      - Input: $1 (opzionale: timeout per acquisire il lock di storico, default
        $BASH4LLM_LOCK_TIMEOUT_HISTORY).
      - Output: Purgatura fisica dal file-system dei log e file storici in
        esubero. Ritorna stato 0.
      - Side-effects: Acquisizione temporanea di lock esclusivo su
        $HISTORY_LOCK, eliminazione di file.
      - Dipendenze: find, lock_exec, sh, rm, sort, file_size, _file_mtime.
  - save_to_history
      - Scopo: Archivia in modo sicuro il testo generato dal modello LLM in un
        file cronologico unico contrassegnato da timestamp e PID del processo
        chiamante all'interno della directory storica $BASH4LLM_HISTORY_DIR.
        Esegue la scrittura transazionale tramite file temporanei e lock
        esclusivo, forzando permessi restrittivi a 600. Aggiorna in modo atomico
        il file di stato UI last_history.json registrandone i metadati
        (percorso, nome, timestamp UTC e peso in byte). Se abilitato, attiva il
        processo di rotazione automatica rotate_history.
      - Input: $1 (stringa testuale della risposta da salvare).
      - Output: File di log scritto sul disco. File di stato UI aggiornato.
        Ritorna stato 0 o il relativo codice di fallimento temporaneo.
      - Side-effects: Creazione file temporanei, scrittura atomica con lock
        esclusivo su $HISTORY_LOCK, invocazione asincrona di rotate_history,
        aggiornamenti dello stato JSON.
      - Dipendenze: date, safe_mkdir, _tmpf, printf, rm, lock_exec, chmod,
        file_size, basename, jq, ui_state_write, rotate_history, log_error,
        log_warn.
  - manifest_create
      - Scopo: Inizializza un nuovo file di manifesto JSON (struttura vuota con
        array di parti) in modo sicuro sotto lock esclusivo, generandone
        contemporaneamente una copia speculare codificata in Base64
        (${manifest}.b64) per usi di trasporto veloci.
      - Input: $1 (percorso del file manifesto di destinazione), $2 (opzionale:
        timeout lock, default $BASH4LLM_LOCK_TIMEOUT_MODELS).
      - Output: File manifesto JSON e controparte Base64 inizializzati sul

      - Side-effects: Creazione directory genitrice via safe_mkdir, scrittura
        file, generazione Base64.
      - Dipendenze: safe_mkdir, dirname, lock_exec, mktemp, printf, base64, mv,
        chmod.
  - manifest_add_part
      - Scopo: Inserisce i metadati e il payload di una risorsa esterna (una
        parte o allegato) all'interno del file di manifesto JSON. Codifica in
        Base64 il contenuto del file di input, lo salva in una directory di
        staging protetta, aggiorna transazionalmente il JSON strutturato del
        manifesto inserendo le proprietà della parte (nome, percorso di staging
        Base64, codifica adottata e tipo di risorsa) e rigenera la copia Base64
        complessiva del manifesto sotto lock esclusivo di sicurezza.
      - Input: $1 (percorso file manifesto), $2 (nome della parte), $3 (percorso
        del file payload di input), $4 (MIME type della risorsa), $5 (opzionale:
        timeout lock, default $BASH4LLM_LOCK_TIMEOUT_MODELS).
      - Output: Record inserito nel manifesto JSON, copia Base64 complessiva
        rigenerata, file di staging Base64 salvato. Ritorna stato 0 o 1.
      - Side-effects: Creazione directory via safe_mkdir, codifica Base64,
        generazione e rimozione file temporanei intermedi, aggiornamento del
        JSON tramite jq, lock di file.
      - Dipendenze: log_error, safe_mkdir, dirname, _tmpf, b64encode, mv, chmod,
        lock_exec, mktemp, cp, base64, jq, rm.
  - manifest_read
      - Scopo: Accede ed esegue la lettura decodificata del file di manifesto
        specificato, estraendolo direttamente dal file JSON di origine o, in
        alternativa, decodificando e ripristinando la sua controparte codificata
        .b64.
      - Input: $1 (percorso del file manifesto).
      - Output: Stampa il contenuto JSON risultante su standard output. Ritorna
        stato 0 o 1 in caso di assenza dei file.
      - Dipendenze: cat, b64decode.
  - _get_perm_string
      - Scopo: Restituisce la stringa di permessi di accesso in notazione
        simbolica (es. -rwxr-xr-x) di un file o directory, unificando le
        discrepanze di formato tra distribuzioni Linux e Darwin/macOS tramite
        platform checks.
      - Input: $1 (percorso dell'oggetto del file-system).
      - Output: Stringa di permessi stampata su standard output.
      - Dipendenze: stat, ls, awk, printf.
  - _get_owner
      - Scopo: Identifica e restituisce il nome dell'utente proprietario di un
        file o di una directory in modo portabile tra sistemi macOS e Linux
        tramite platform checks.
      - Input: $1 (percorso dell'oggetto).
      - Output: Nome dell'owner stampato su standard output.
      - Dipendenze: stat, ls, awk, printf.
  - _get_file_signature
      - Scopo: Calcola l'impronta di stato e integrità di un file. Se abilitato
        l'uso degli hash ($BASH4LLM_SIG_HASH attivo ed eseguibile openssl o
        sha256sum presente), calcola l'hash SHA-256 del contenuto; in caso
        contrario o in aggiunta, interroga stat estraendo metadati fisici (ID
        dispositivo, inode, dimensione, ctime, mtime, ID proprietario, ID gruppo
        e permessi ottali) unificandoli in una stringa delimitata da pipe.
        Questo rileva alterazioni del file anche in caso di falsificazione delle
        date di modifica.
      - Input: $1 (percorso del file).
      - Output: Stringa strutturata
        hash|device|inode|size|ctime|mtime|uid|gid|mode stampata su standard
        output.
      - Dipendenze: sha256sum, openssl, awk, stat, read.
  - getfile_signature
      - Scopo: Wrapper diretto della funzione interna _get_file_signature.
      - Input: $1 (percorso del file).
      - Output: Stringa della firma stampata su standard output.
  - _is_world_writable
      - Scopo: Verifica se una directory presenta permessi di accesso non sicuri
        che consentono la scrittura a gruppi esterni o a chiunque
        (world-writable).
      - Input: $1 (percorso della cartella).
      - Output: Ritorna stato 0 (vulnerabile/scrivibile pubblicamente),
        altrimenti 1 (sicura o errore).
      - Dipendenze: _get_perm_string, awk, printf.
  - make_tmpdir
      - Scopo: Crea una directory temporanea univoca e protetta all'interno di
        $BASH4LLM_TMPDIR sotto lock esclusivo $TMP_LOCK, impostando permessi
        blindati a 700 via safe_mkdir.
      - Input: Nessuno.
      - Output: Stampa su standard output il percorso della cartella generata.
        Ritorna stato 0 o il codice di errore BASH4LLM_ERR_TMP.
      - Side-effects: Creazione directory via safe_mkdir, modifica permessi,
        lock esclusivo su $TMP_LOCK.
      - Dipendenze: safe_mkdir, chmod, lock_exec, _tmpf, printf.
  - _tmpf
      - Scopo: Generatore di oggetti temporanei sicuro. Convalida che la
        directory temporanea principale $BASH4LLM_TMPDIR non sia un collegamento
        simbolico, verifica che la cartella di base fornita risieda strettamente
        all'interno della directory temporanea autorizzata (impedendo attacchi
        di tipo Directory Traversal), imposta la maschera di processo umask 077
        e alloca file (permessi 600) o directory (permessi 700) avvalendosi di
        mktemp.
      - Input: $1 (modalità: file o dir), $2 (directory di base), $3 (opzionale:
        prefisso del nome, default "groq").
      - Output: Stampa su standard output il percorso assoluto dell'oggetto
        creato. Ritorna stato 0 o codici d'errore specifici.
      - Side-effects: Scrittura sul disco, modifica dei permessi del
        file-system, tracciamento log di errore.
      - Dipendenze: log_error, chmod, mktemp, printf.
  - thread_validate_id
      - Scopo: Verifica la correttezza sintattica dell'identificatore del thread
        fornito dall'utente per escludere iniezioni di percorsi o l'uso di
        caratteri speciali dannosi.
      - Input: $1 (stringa dell'ID thread).
      - Output: Ritorna stato 0 se l'ID rispetta il formato alfanumerico ammesso
        (comprendente caratteri alfanumerici, punto, trattino e underscore, con
        una lunghezza totale compresa tra 1 e 128 caratteri), altrimenti
        ritorna 1.
      - Dipendenze: grep.
  - thread_now_ts
      - Scopo: Restituisce la data e l'ora correnti espresse in formato ISO 8601
        UTC.
      - Input: Nessuno.
      - Output: Stringa di timestamp inviata a standard output (formato
        AAAA-MM-GGTHH:MM:SSZ).
      - Dipendenze: date.
  - thread_messages_tmp_path
      - Scopo: Calcola e restituisce la posizione del file JSON destinato ad
        accumulare temporaneamente i messaggi del thread attivo nel corso
        dell'elaborazione corrente.
      - Input: $1 (ID del thread).
      - Output: Stampa su standard output il percorso del file provvisorio

        errori nell'allocazione della cartella temporanea.
      - Dipendenze: ensure_run_tmpdir, printf.
  - thread_sanitize_cmd
      - Scopo: Pulisce e sanifica le stringhe dei comandi di esecuzione prima di

        caratteri per ragioni di spazio.
      - Input: $1 (stringa del comando crudo).
      - Output: Stringa sanificata stampata su standard output.
      - Dipendenze: printf, sed, cut.
  - _update_thread_index
      - Scopo: Aggiorna in sicurezza l'indice globale dei thread attivi nel file
        index.json posizionato nella directory threads/ dello stato utente sotto

      - Input: $1 (thread_id).
      - Output: Stato 0 o errore.
      - Dipendenze: mktemp, jq, mv, ui_state_write, rm, lock_exec.
  - _thread_delete_locked
      - Scopo: Esegue la rimozione fisica del database dei thread, dei lock

      - Input: $1 (thread_id), $2 (file_thread), $3 (file_meta), $4
        (file_indice), $5 (tmp_indice).
      - Output: Eliminazione file da disco e aggiornamento indice JSON.
      - Dipendenze: rm, jq, mv, chmod.
  - thread_delete_core
      - Scopo: Rimuove transazionalmente un thread, i relativi lock e file di

      - Input: $1 (thread_id).
      - Output: Stato 0 o errore.

  - _thread_rename_locked

        cambiarne il titolo visualizzato.
      - Input: $1 (file_meta), $2 (tmp_meta), $3 (thread_id), $4 (titolo), $5
        (conteggio_msg), $6 (ultimo_ts).
      - Output: Stato 0 o errore, metadati aggiornati.
      - Dipendenze: jq, mv, chmod.
  - thread_rename_core
      - Scopo: Rinomina transazionalmente il titolo di un thread aggiornandone i
        metadati storici sotto lock esclusivo di storico.
      - Input: $1 (thread_id), $2 (titolo).
      - Output: Stato 0 o errore.
      - Dipendenze: thread_validate_id, _tmpf, wc, tail, jq, lock_exec,

  - acquire_thread_lock
      - Scopo: Acquisisce un lock esclusivo di processo non-bloccante sul thread

      - Input: Nessuno (usa $THREAD_ID).

      - Dipendenze: thread_validate_id, flock, log_error.
  - release_thread_lock
      - Scopo: Rilascia il lock di processo esclusivo del thread attivo

      - Input: Nessuno (usa $THREAD_ID).
      - Dipendenze: thread_validate_id, flock, log_error.
  - _thread_read_window_locked
      - Scopo: Estrae l'ultima porzione del database NDJSON del thread senza
        scansioni di tutto il file.
      - Input: $1 (file_thread), $2 (n_righe), $3 (file_temporaneo).
      - Dipendenze: tail, cp.
  - thread_read_window
      - Scopo: Estrae dal file log NDJSON del thread specificato la finestra
        degli ultimi $n messaggi registrati. Opera sotto lock esclusivo, preleva
        le righe via _thread_read_window_locked ed unifica in array JSON
        ordinato tramite pipeline jq nativa (con Try-Catch resiliente ad
        interruzioni). Aggiorna i metadati del thread in threads/.json via
        ui_state_write.
      - Input: $1 (thread_id), $2 (finestra, default 10), $3 (file JSON
        destinazione).
      - Output: File JSON strutturato scritto nel percorso di destinazione. File

      - Side-effects: Lock esclusivo, allocazione e rimozione file temporanei
        intermedi, lettura del registro di storico, aggiornamento delle
        directory di stato UI.
      - Dipendenze: safe_mkdir, lock_exec, _thread_read_window_locked, jq, mv,
        cp, rm, ui_state_write, log_error, printf.
  - thread_append
      - Scopo: Accoda un messaggio nel registro NDJSON del thread specificato.
        Genera ID univoco (SHA-256 di testo+ts+random), coordina la scrittura
        tramite lock flock e previene conflitti controllando le cartelle marker
        associate al PID. Aggiorna i metadati in threads/.json e l'indice
        globale in threads/index.json sotto lock esclusivo.
      - Input: $1 (thread_id), $2 (ruolo), $3 (testo), $4 (metadati).
      - Output: Messaggio aggiunto in append nel file NDJSON di storico con

      - Side-effects: Creazione directory via safe_mkdir, allocazione e

      - Dipendenze: thread_now_ts, jq, sed, awk, sha256sum, openssl, safe_mkdir,
        _tmpf, flock, grep, wc, tail, ui_state_write, lock_exec, log_error,
        log_warn.
  - _thread_hash
      - Scopo: Genera un hash crittografico sicuro SHA-256 (con fallback su
        codifica Base64 troncata qualora non vi siano binari idonei installati)
        per identificare in modo univoco una stringa o una combinazione di
        parametri.
      - Input: $1 (stringa di input).
      - Output: Stringa di hash stampata su standard output.
      - Dipendenze: sha256sum, awk, openssl, base64, tr, cut, printf.
  - thread_cache_key
      - Scopo: Calcola la chiave identificativa univoca di cache accoppiando
        l'ID thread con l'hash SHA-256 dei parametri di invocazione forniti.
      - Input: $1 (ID thread), $2 (stringa dei parametri).
      - Output: Chiave di cache stampata su standard output (formato
        thread_id|hash_parametri).
      - Dipendenze: _thread_hash, printf.
  - thread_cache_get
      - Scopo: Estrae i risultati archiviati nella cache temporanea per il
        specifico thread e i parametri indicati. Verifica la validità temporale
        del record leggendone la prima riga (timestamp Unix di scadenza) e
        confrontandola con l'ora corrente del sistema; se il record risulta
        scaduto, cancella fisicamente il file dal disco e restituisce un errore.
      - Input: $1 (ID thread), $2 (stringa dei parametri), $3 (opzionale: file
        di destinazione in cui copiare i dati della cache).
      - Output: Riversa i dati su stdout o scrive nel file di output indicato se
        la cache è valida. Ritorna stato 0 in caso di successo, 1 se mancante o
        scaduta.
      - Side-effects: Rimozione fisica dal disco del file di cache se scaduto.
      - Dipendenze: thread_cache_key, date, rm, tail.
  - thread_cache_set
      - Scopo: Persiste i risultati di un thread e dei relativi parametri
        all'interno della cartella di cache, impostando un tempo di validità
        personalizzato (TTL, default 300 secondi). Scrive la scadenza assoluta
        espressa in timestamp Unix all'interno della prima riga del file, vi
        annette in coda i dati (letti da file o stdin) e archivia il file in
        modo atomico con permessi 600.
      - Input: $1 (ID thread), $2 (stringa parametri), $3 (valore TTL in
        secondi, default 300), $4 (opzionale: file sorgente contenente il
        payload; se assente legge da stdin).
      - Output: File di cache generato sotto $THREAD_CACHE_DIR con permessi 600.
        Ritorna stato 0 o 1 in caso di errore.
      - Side-effects: Allocazione file temporanei, scrittura atomica con
        sovrascrittura.
      - Dipendenze: thread_cache_key, date, cat, _tmpf, mv, rm, chmod.
  - thread_cache_invalidate
      - Scopo: Invalida e purga i file di cache associati a una sessione thread.
        Se sono passati parametri specifici, elimina esclusivamente la cache di
        quella combinazione, altrimenti rimuove fisicamente tutti i file di
        cache correlati all'ID della sessione thread.
      - Input: $1 (ID thread), $2 (opzionale: parametri specifici).
      - Output: Eliminazione fisica dei file di cache. Ritorna stato 0, o 1 se
        l'ID sessione thread risulta vuoto.
      - Dipendenze: thread_cache_key, rm.
  - _normalize_bool_env
      - Scopo: Uniforma i parametri di configurazione booleani principali
        presenti nell'ambiente (ALLOW_API_CALLS, DRY_RUN, DEBUG), convertendo
        stringhe permissive (es. "true", "yes") nei valori numerici 1 (attivo)
        o 0 (inattivo) ed esportandoli per la coerenza globale del runtime.
      - Input: Nessuno.
      - Output: Esporta le variabili d'ambiente allineate.
      - Dipendenze: is_truthy, export.

3. PROVIDER

VARIABILI

  - GROQ_API_KEY: Chiave API deputata all'autenticazione delle richieste verso i
    servizi del provider Groq. Eredita ${GROQ_API_KEY}.

FUNZIONI

  - buildpayload_groq
      - Scopo: Compila ed organizza la struttura del file JSON di payload da
        trasmettere alle API di Groq. Valida la correttezza numerica di
        temperatura (TURE, default 1.0) e numero massimo di token (MAX_TOKENS,
        default 4096), convertendoli in valori numerici JSON validi. Estrae i
        messaggi della conversazione con priorità sequenziale: decodifica e
        convalida il JSON utente se presente in $JSON_INPUT, estrae da
        $MESSAGES_JSON, elabora file JSON completi di messaggi indicati in
        $BUILD_MESSAGES_FILE o converte il prompt testuale crudo $CONTENT in un
        record utente standard. Se non rileva alcun messaggio o testo valido,
        genera come fallback un record utente contenente un testo vuoto ed
        emette un avvertimento log. Se non definito, tenta di estrarre il
        modello da utilizzare dal file dei messaggi. Scrive infine il file JSON
        temporaneo risultante e lo converte in formato Base64 (tramite
        stage_b64) per scopi di transazione sicura se la funzione è presente nel
        runtime.
      - Input: Parametri opzionali di configurazione.
      - Output: Assegna il percorso del file di payload finale generato a
        BASH4LLM_TMP_PAYLOAD (nella versione 2.5.0) e $PAYLOAD. Ritorna stato 0,
        o BASH4LLM_ERR_TMP se si verificano errori fisici di scrittura o di
        directory temporanea.
      - Side-effects: Scrittura e cancellazione file temporanei, codifica
        Base64, emissione di log di avvertimento su stderr.
      - Dipendenze: ensure_run_tmpdir, _mktemp_in_dir, _tmpf, is_truthy, grep,
        log_warn, is_valid_json_string, jq, is_valid_json_file, stage_b64, rm,
        wc, head, log_error, log_info.
  - call_api_groq
      - Scopo: Esegue una richiesta API sincrona non-streaming verso l'endpoint
        di Groq. Verifica la politica di rete e accerta la presenza della chiave
        API nell'ambiente. Se il file del payload è codificato in Base64
        (estensione .b64), esegue la decodifica in un file provvisorio prima
        dell'invio. Prepara l'array di argomenti per l'utilità curl (includendo
        i parametri di rete e connessione, gli header HTTP di autenticazione e
        il tipo di contenuto), esegue la chiamata trasmettendo in modalità
        binaria il payload ed estrae lo stato HTTP restituito. Se la risposta
        dell'API risulta corrotta o presenta testo non-JSON accodato in coda al
        blocco strutturato principale, isola e ripristina la sola porzione JSON
        sintatticamente corretta eliminando i dati estranei di trailing per
        garantire la stabilità di esecuzione. Salva il risultato finale in $RESP
        con permessi 600, o vi scrive un JSON diagnostico contenente i dettagli
        dell'errore e i codici di errore in caso di fallimenti di connessione o
        del server.
      - Input: Nessuno.
      - Output: File di risposta scritto in $RESP. Ritorna stato 0 se eseguito
        (anche in presenza di errori HTTP restituiti dalle API), o codici di
        fallimento in caso di errori fisici di curl o di directory non
        scrivibili.
      - Side-effects: Decodifica di flussi, connessioni di rete tramite curl,
        generazione e cancellazione di file temporanei sul disco.
      - Dipendenze: ensure_run_tmpdir, enforce_network_policy,
        resolve_provider_url, _tmpf, b64decode, curl, jq, tail, awk, mv, chmod,
        log_error, log_info, rm, head.
  - call_api_streaming_groq
      - Scopo: Esegue una chiamata API in tempo reale (streaming dei token)
        verso l'endpoint di Groq. Decodifica il payload se codificato Base64,
        configura curl disabilitando il buffering (-N) ed esegue una pipeline di
        lettura riga per riga dell'output conforme allo standard Server-Sent
        Events (SSE). Tramite un singolo pass unbuffered di jq, estrae i delta
        testuali di risposta stampandoli all'istante su stdout, isolando e
        visualizzando in parallelo i messaggi d'errore strutturati formali del
        server. Accumula l'intero flusso crudo ricevuto in un file di log
        temporaneo di staging. Al termine della connessione, convalida ed
        assembla tutti i frammenti JSON estratti per ricostruire l'intero corpo
        di risposta cumulativo, riversa il testo complessivo unificato e salva
        il pacchetto JSON completo all'interno di $RESP con permessi 600.
        Aggiorna infine in modo atomico il file di stato last_api.json con
        metadati diagnostici dettagliati (ID della richiesta API, codice HTTP di
        stato e motivo di arresto finish_reason).
      - Input: Nessuno.
      - Output: Flusso di token stampato istantaneamente su stdout. File JSON
        finale di risposta salvato in $RESP. Ritorna lo stato dell'esecuzione
        della pipeline di connessione.
      - Side-effects: Connessione di rete persistente, decodifica, scrittura in
        tempo reale su stdout, generazione e cancellazione di molteplici file di
        staging di flussi, aggiornamento atomico dello stato dell'interfaccia.
      - Dipendenze: ensure_run_tmpdir, enforce_network_policy,
        resolve_provider_url, _tmpf, b64decode, curl, tee, jq, grep, sed, mv,
        chmod, ui_state_write, log_error, log_info, log_warn, rm, head.
  - refresh_models_groq
      - Scopo: Sincronizza ed aggiorna la lista locale dei modelli supportati
        interrogando l'endpoint /openai/v1/models di Groq. Esegue una chiamata
        GET autenticata tramite chiave API, valida la risposta JSON ricevuta ed
        estrae i nomi dei modelli (.data o piatti), normalizzandoli (rimuove
        spazi, modelli/ o groq/), ordinandoli e limitandoli a $MAX_MODELS. Salva

        parte di processi concorrenti.
      - Input: Nessuno.
      - Output: File $MODELS_FILE aggiornato con l'elenco dei modelli
        normalizzati. Ritorna stato 0 se completato con successo, codici
        d'errore specifici in caso di fallimenti delle API, di parsing o di
        lock.
      - Side-effects: Invocazione di rete (GET HTTP), allocazione directory
        temporanea, elaborazione dati intensiva, scrittura atomica con lock
        esclusivo su $MODELS_FILE.
      - Dipendenze: ensure_api_key_for_provider, ensure_run_tmpdir, mktemp,
        resolve_provider_url, curl, jq, awk, sed, sort, _mktemp_in_dir, _tmpf,
        b64_atomic_write, lock_exec, base64, chmod, rm, log_error, log_info.
  - validate_model_groq
      - Scopo: Convalida la compatibilità del modello richiesto chiamando
        validate_model_core.
      - Input: $1 (nome del modello da validare).
      - Output: Ritorna stato 0 se il modello è valido, 1 in caso d'errore.
      - Dipendenze: validate_model_core.
  - validate_key_groq
      - Scopo: Diagnostica la validità della chiave API del provider inviando

      - Input: $1 (chiave da testare).
      - Output: Codice stato: 0 (connessione ed autenticazione riuscita), 1

      - Dipendenze: _mktemp_in_dir, _tmpf, mktemp, curl, rm.
  - auto_select_model_groq
      - Scopo: Scansiona in ordine sequenziale il file dei modelli locali

      - Input: Nessuno.

  - CHAT_MODE: Flag booleano (valore 1 o 0, default 0) che abilita l'interfaccia

  - FORCE_SAVE_MODE: Regola il salvataggio dei risultati sul disco (eredita

FUNZIONI

  - call_provider
      - Scopo: Esegue in modo dinamico una specifica funzione dichiarata da un
        modulo provider esterno, a patto che la funzione risulti regolarmente
        caricata in memoria.
      - Input: $1 (nome della funzione da richiamare), seguito in coda dai

      - Output: Restituisce lo stato d'uscita della funzione eseguita, o il

      - Dipendenze: type.
  - validate_provider_key_dispatch
      - Scopo: Dispaccia l'esecuzione della validazione formale della chiave API
        delegando al metodo del provider attivo (es. validate_key_groq).
      - Input: $1 (chiave API da convalidare).
      - Output: Ritorna il codice stato del validatore del provider, o 127 se

      - Dipendenze: type.
  - refresh_models_dispatch
      - Scopo: Dispaccia l'esecuzione dell'aggiornamento dei modelli richiamando

      - Input: $1 (opzionale: percorso del file di modelli, default

      - Dipendenze: type, log_error, log_info.
  - validate_model_dispatch
      - Scopo: Dispaccia la validazione sintattica e di compatibilità del

        validate_model_groq). Se il provider non supporta la convalida
        specializzata, emette un avvertimento una tantum ed esegue una convalida
        permissiva di fallback.
      - Input: $1 (nome del modello).
      - Output: Ritorna lo stato di uscita del validatore o 0 se opera in
        convalida permissiva.
      - Dipendenze: type, log_warn.
  - auto_select_model_dispatch
      - Scopo: Dispaccia la selezione automatica del modello chiamando la
        routine nativa del provider attivo (es. auto_select_model_groq).
      - Input: Nessuno.
      - Output: Nome del modello su stdout e stato d'uscita.
      - Dipendenze: type.
  - resolve_model
      - Scopo: Risolve ed assegna il modello LLM finale da adottare seguendo un
        rigoroso ordine di precedenza sequenziale:
        1.  Se impostato esplicitamente via parametro CLI (-m/--model), adotta
            direttamente tale scelta.
        2.  Identifica il provider attivo (analizzando parametri CLI, leggendo
            il file canonico del provider $PROVIDER_FILE o adottando il fallback
            standard "groq").
        3.  Tenta di leggere la prima riga del file di configurazione specifico
            del modello per tale provider (model.). Se presente, valido e
            supportato, lo adotta dopo averlo normalizzato.
        4.  Tenta di richiamare la funzione di auto-selezione nativa del
            provider (auto_select_model_dispatch). Se restituisce un modello
            valido, lo adotta.
        5.  Esegue una scansione riga per riga del file dei modelli locali
            (MODELS_FILE) fino alla soglia limite $MAX_MODELS alla ricerca del
            primo modello valido e supportato.
        6.  Tenta di recuperare il modello inserito all'interno del file di
            configurazione globale (config).
        7.  Cerca all'interno dell'elenco di modelli autorizzati accumulati
            nella whitelist $ALLOWED_MODELS. Se tutte le ricerche falliscono,
            restituisce un errore di selezione.
      - Input: Nessuno.
      - Output: Ritorna stato 0 se risolto (assegnando il nome del modello alla
        variabile globale $MODEL), 1 in caso di totale fallimento di
        risoluzione.
      - Side-effects: Letture multiple sul file-system, invocazione di
        validatori.
      - Dipendenze: canonical_model_file, canonical_provider_file, sed, awk,
        validate_model_core, validate_model_dispatch,
        auto_select_model_dispatch, _normalize_model_name, is_supported_model,
        log_warn, log_error.
  - build_payload_from_vars
      - Scopo: Dispaccia ed orchestra la costruzione del file JSON di richiesta
        API richiamando dinamicamente la routine dedicata del provider attivo
        (buildpayload_).
      - Input: Nessuno.
      - Output: Generazione del file di payload. Ritorna stato 0, o termina lo
        script con codice d'errore BASH4LLM_ERR_API se la funzione non è
        implementata nel modulo.
      - Dipendenze: ensure_run_tmpdir, call_provider, log_error.
  - call_api_once
      - Scopo: Dispaccia l'esecuzione di una singola chiamata API sincrona
        non-streaming richiamando la routine dedicata del provider attivo
        (call_api_). Se è attiva la simulazione dry-run, esclude la chiamata
        reale e visualizza a schermo a scopo diagnostico l'intestazione del
        payload d'invio.
      - Input: Nessuno.
      - Output: Ritorna lo stato dell'esecuzione della chiamata o 0 se operato
        in simulazione.
      - Dipendenze: call_provider, show_payload_head, log_info, log_error.
  - call_api_streaming
      - Scopo: Dispaccia l'esecuzione della chiamata API in streaming
        richiamando la routine dedicata del provider attivo
        (call_api_streaming_) sotto lock di thread esclusivo. Implementa un
        meccanismo di ripristino sicuro del segnale d'interruzione (SIGINT) per
        sbloccare l'fd lock in caso di uscita abrupta. Se è attiva la
        simulazione dry-run, simula la connessione e visualizza l'intestazione
        del payload d'invio.
      - Input: Nessuno.
      - Output: Ritorna lo stato dell'esecuzione o 0 se simulato.
      - Side-effects: Gestione dei segnali e dei lock concorrenziali del
        database.
      - Dipendenze: acquire_thread_lock, release_thread_lock, call_provider,
        show_payload_head, log_info, log_error.
  - extract_api_error
      - Scopo: Ispeziona ed estrae in modo strutturato l'eventuale messaggio
        d'errore formale restituito dal provider all'interno del file di
        risposta $RESP (ricercando campi errore standard nel JSON o leggendo la
        prima riga di testo crudo).
      - Input: Nessuno (analizza $RESP).
      - Output: Messaggio d'errore stampato su standard output.
      - Dipendenze: jq, awk.
  - detect_empty_edge_case
      - Scopo: Algoritmo diagnostico per rilevare ed intercettare "risposte
        vuote" anomale dei provider (situazioni in cui l'API restituisce stato
        HTTP 200 e termina la generazione con motivo "stop" o assenza di token,
        ma senza produrre alcun testo utile). Analizza il file di risposta $RESP
        ed estrae metadati come l'ID richiesta, il finish reason e il conteggio
        dei token generati per classificare l'evento.
      - Input: Nessuno.
      - Output: Assegna il verdetto booleano a
        BASH4LLM_EDGE_EMPTY (1 se rilevato caso vuoto anomalo, 0 altrimenti) e memorizza i dettagli in variabili globali (BASH4LLM_EDGE_REQ_ID,
        $BASH4LLM_EDGE_FINISH_REASON, $BASH4LLM_EDGE_COMPLETION_TOKENS).
      - Side-effects: Duplicazione temporanea del file per l'analisi sicura.
      - Dipendenze: _tmpf, cp, is_valid_json_file, jq, rm.
  - finalize_and_output
      - Scopo: Formatta e visualizza i risultati finali generati in base alla
        modalità selezionata in $OUTPUT_MODE (testo, raw, json o pretty).
        Inoltre, se l'output testuale supera la soglia di caratteri $THRESHOLD o
        se è forzato dall'utente, si occupa di salvare i risultati nel
        file-system (all'interno di $OUT_PATH o nella directory storica standard
        $BASH4LLM_HISTORY_DIR) sotto lock concorrenziale con umask blindata.
      - Input: $1 (formato: json, pretty, raw, text), $2 (stringa di testo del
        risultato).
      - Output: Output stampato a schermo secondo la formattazione richiesta.
        File archiviato fisicamente su disco se abilitato. Ritorna stato 0 o

      - Side-effects: Scrittura a schermo, salvataggio file storico

      - Dipendenze: _tmpf, cp, cat, jq, printf, wc, tr, date, safe_mkdir,

  - perform_request_once
      - Scopo: Orchestra l'esecuzione complessiva di una chiamata API sincrona

        tentativi, applicando un backoff lineare per l'attesa tra i tentativi) e
        ripristino dei segnali. Gestisce l'estrazione dei testi, la diagnostica
        dei casi vuoti anomali (detect_empty_edge_case), la notifica degli
        errori formali inviati dalle API e la scrittura dello stato dell'ultima
        transazione in last_api.json. Formatta i risultati e restituisce lo
        stato d'uscita.
      - Input: Nessuno.
      - Output: Output generato stampato su console e file di log aggiornati.
        Ritorna stato 0 se completata, codice d'errore BASH4LLM_ERR_API se
        fallisce.
      - Side-effects: Tentativi multipli di rete, sospensione temporanea del
        processo (sleep), aggiornamenti dello stato UI, messaggi diagnostici su
        stderr, gestione dei segnali e del file fd lock.
      - Dipendenze: acquire_thread_lock, release_thread_lock, call_api_once,
        extract_text_from_resp, detect_empty_edge_case, jq, ui_state_write,
        extract_api_error, finalize_and_output, sleep, log_info, log_error,

  - collect_input_from_files

      - Input: Array di percorsi di file ($@).
      - Output: Stringa di testo unificata stampata su stdout. Interrompe lo

  - expand_args_to_content
      - Scopo: Converte gli argomenti CLI posizionali in frammenti di prompt. Se

      - Input: Elementi accumulati nell'array $ARGS.
      - Output: Stringa di prompt unificata stampata su stdout.
      - Dipendenze: file_readable, cat.
  - file_readable
      - Scopo: Convalida se il percorso indicato corrisponde a un file regolare

      - Input: $1 (percorso file).
      - Output: Ritorna stato 0 se leggibile, 1 in caso contrario.
  - trim
      - Scopo: Rimuove in modo efficiente gli spazi vuoti, i tabulatori e i

      - Input: $1 (stringa da pulire).

  - is_number

      - Input: $1 (stringa da analizzare).
      - Output: Ritorna stato 0 se numerico, 1 altrimenti.
      - Dipendenze: awk, printf.
  - is_supported_model
      - Scopo: Analizza il nome del modello e ne convalida la compatibilità con

  - list_models_cli

  - validate_model_core

      - Azione: Scorre una lista predefinita di variabili di controllo booleane

5. CORE_PROVIDER

VARIABILI

  - _supported_providers_arr: Array interno per l'indicizzazione dei provider
    rilevati.
  - SUPPORTED_PROVIDERS: Elenco testuale unificato dei provider supportati
    separati da caratteri di spazio.

FUNZIONI

  - validate_provider_interface
      - Scopo: Convalida che il modulo del provider caricato implementi
        correttamente le specifiche dell'interfaccia di Bash4LLM⁺. Verifica la
        presenza fisica in memoria delle funzioni obbligatorie (buildpayload_ e
        call_api_), e traccia a scopo informativo la presenza o assenza di
        metodi opzionali (streaming, validazione modelli, auto-selezione,
        refresh modelli).
      - Input: $1 (nome identificativo del provider da analizzare).
      - Output: Ritorna stato 0 se l'interfaccia obbligatoria è pienamente
        implementata, 1 in caso contrario (emettendo errori critici loggati).
      - Dipendenze: type, log_error, log_info.
  - assemble_content
      - Scopo: Compila ed organizza la stringa del prompt di input principale
        $CONTENT che verrà inviata al modello LLM. Segue una catena di regole di
        priorità esclusive:
        1.  Se è configurato un JSON utente in $JSON_INPUT, pulisce la variabile
            $CONTENT ed esce (in quanto il JSON di input definisce autonomamente
            i messaggi).
        2.  Se l'utente ha inserito dei file via parametro -f ($FILE_INPUTS), ne
            raccoglie ed unifica i contenuti via collect_input_from_files. Se
            sono presenti argomenti posizionali aggiuntivi, li annette in coda
            come frammenti testuali.
        3.  Se è richiesto l'uso di un template ($TEMPLATE), preleva il file del
            modello dalla directory dei template, esegue la sostituzione della
            stringa segnaposto {{CONTENT}} rimpiazzandola con il prompt inserito
            (proveniente da file, da stdin o da argomenti posizionali) usando
            awk con file di staging intermedi temporanei, e assegna il testo
            risultante a $CONTENT.
        4.  Se è presente dell'input catturato dallo standard input
            ($STDIN_CONTENT), lo adotta come prompt principale, allegando in
            coda eventuali argomenti posizionali.
        5.  Se vi sono solo argomenti CLI posizionali in $ARGS, li converte ed
            unisce nel prompt testuale principale via expand_args_to_content.
      - Input: Nessuno.
      - Output: Stringa compilata assegnata alla variabile globale $CONTENT.
      - Side-effects: Lettura file, generazione e distruzione di file di staging
        temporanei sul disco.
      - Dipendenze: collect_input_from_files, expand_args_to_content, cat,
        ensure_run_tmpdir, _mktemp_in_dir, awk, mv, rm, printf.

BLOCCHI DI CODICE

  - CORE_PROVIDER_PRO_LOAD_INITIALIZATION
      - Fase: Selezione e caricamento del modulo provider attivo.
      - Azione: Scansiona la directory dei provider esterni per mappare i moduli
        installati. Determina il provider attivo leggendolo dai parametri CLI o
        dal file di configurazione del provider $PROVIDER_FILE (adottando "groq"
        in caso di assenza).
          - Se configurato un nuovo provider da CLI ($PROVIDER_CLI diverso da
            "list"): convalida che appartenga a quelli supportati, salva
            transazionalmente la scelta nel file $PROVIDER_FILE con permessi 600
            e, se il provider è cambiato, cancella il vecchio file locale dei
            modelli $MODELS_FILE (per forzarne l'allineamento API), elimina il
            vecchio file URL del provider e risolve il nuovo URL.
          - Se richiesta la selezione interattiva (--provider senza argomenti o
            valore "list"): visualizza a schermo il menu numerato di tutti i
            provider installati evidenziando il default, acquisisce la scelta
            dell'utente dallo standard input (accettando l'indice o il nome
            letterale e adottando il default se premuto invio), scrive in modo
            atomico la scelta in $PROVIDER_FILE con permessi 600, rimuove l'URL
            preesistente e invalida il file locale dei modelli. Infine, se il
            provider attivo è "groq" e manca il file URL locale, scrive l'URL
            dell'API predefinito nel file canonico. Invoca la funzione di
            caricamento sicuro load_provider_module per importare i metodi del
            modulo provider ed esegue la risoluzione dell'URL dell'API. Se
            l'interfaccia interattiva viene eseguita senza comandi aggiuntivi,
            stampa la conferma del provider impostato ed esce direttamente con
            stato 0.
  - CORE_PROVIDER_PRO_LOAD_VALIDATION_REFRESH
      - Fase: Validazione del modulo provider ed allineamento modelli.
      - Azione: Verifica che il modulo del provider sia stato correttamente
        importato ed esegue la convalida della conformità della sua interfaccia
        via validate_provider_interface; in caso di errori strutturali termina
        con codice BASH4LLM_ERR_API. Se è stato richiesto esplicitamente il
        refresh dei modelli ($REFRESH_MODELS attivo), richiede la chiave API ed
        avvia l'allineamento dei modelli locali tramite refresh_models_dispatch,
        terminando con stato 0. In subordine, se il file dei modelli locali
        $MODELS_FILE è assente o completamente vuoto, tenta di eseguire un
        allineamento automatico dei modelli in background (richiedendo la chiave
        API ed invocando la routine di refresh, senza interrompere lo script in
        caso di fallimento per permettere l'elaborazione dei flussi in
        background).
  - CORE_PROVIDER_SHOW
      - Fase: Esecuzione delle risposte diagnostiche immediate.
      - Azione: Gestisce e soddisfa le richieste diagnostiche e di ispezione dei
        percorsi dello script, terminando l'esecuzione con stato 0 dopo la
        visualizzazione dei dati:
          - --print-config-dir / PRINT_CONFIG_DIR: Stampa il percorso della
            directory di configurazione canonica.
          - --print-provider-file / PRINT_PROVIDER_FILE: Stampa il percorso del
            file canonico del provider.
          - PRINT_MODEL_FILE: Stampa il percorso del file del modello canonico
            per il provider indicato.
          - SHOW_CONFIG: Visualizza un riepilogo tabulare dettagliato dei
            parametri correnti dell'installazione (versione, date, provider
            attivo, modello attivo, modello di default, percorsi di tutte le
            directory di lavoro di Bash4LLM⁺, stato di debug e simulazione,
            conteggio dei modelli in whitelist).
          - DIAGNOSTICS: Esegue un'analisi di integrità diagnostica completa
            dello stato del sistema. Scansiona e verifica l'esistenza e
            l'accessibilità fisica di tutte le directory operative, attesta che
            le funzioni obbligatorie del provider selezionato siano regolarmente
            mappate in memoria, verifica la conformità del modello attivo
            rispetto alla whitelist locale, controlla l'effettiva presenza della
            chiave API nell'ambiente correlandola con il nome esatto della
            variabile attesa dal provider, lo stato di sblocco della sessione
            vault (_B4L_RT_CTX), l'esito dei test crittografici e del TLS
            Handshake verso l'endpoint remoto attivo.
  - CORE_PROVIDER_MAIN_RESOLVE
      - Fase: Risoluzione del modello e cattura standard input.
      - Azione: Se l'avvio è per solo bootstrap ($BOOTSTRAP_ONLY), esce
        immediatamente con stato 0. Carica la configurazione locale e le
        whitelist dei modelli, garantisce la stabilità della directory
        temporanea $RUN_TMPDIR, risolve il modello LLM finale da adottare via
        resolve_model ed esegue su di esso le convalide sintattiche e di
        interfaccia del provider (terminando con codice BASH4LLM_ERR_BAD_MODEL
        se il modello selezionato fallisce le verifiche). Infine, cattura e
        memorizza l'eventuale flusso di testo in arrivo da una redirezione o
        pipeline nello standard input (cat -) all'interno di STDIN_CONTENT e, in
        presenza di un JSON strutturato, ne estrae transazionalmente la chiave
        API interna ed i prompt d'ingresso.
  - CORE_PROVIDER_MAIN_EXECUTION
      - Fase: Esecuzione principale del prompt (Core Execution loop).
      - Azione: Invoca assemble_content per generare il prompt definitivo. Se lo
        script è stato richiamato in modalità interattiva e non contiene prompt,
        visualizza la notifica del provider attivo ed esce. Esegue controlli di
        sicurezza sull'input bloccando l'esecuzione con codice
        BASH4LLM_ERR_NO_PROMPT in assenza di prompt testuale o JSON d'input
        (tranne se in chat interattiva o elaborazione batch). Valida la
        correttezza numerica dei parametri di temperatura e token massimi.
        Controlla che il modello sia regolarmente supportato o non deprecato
        ispezionando la whitelist e garantisce la presenza della directory
        temporanea $RUN_TMPDIR. Procede quindi con i tre flussi di esecuzione
        principali:
        1.  Elaborazione BATCH ($BATCH_FILE attivo): Scansiona il file batch
            riga per riga, escludendo righe vuote o commenti. Per ciascun
            prompt:
              - Configura l'ambiente di thread operativo se richiesto (validando
                l'ID thread, limitando la finestra dei messaggi, e richiamando
                session_engine_build_window o la legacy thread_read_window per
                compilare il JSON storico $BUILD_MESSAGES_FILE). Se lo storico è
                assente o vuoto, lo inizializza inserendo un record utente con
                il prompt corrente.
              - Dispaccia la costruzione del payload del file JSON via
                build_payload_from_vars.
              - Verifica e richiede la chiave API per il provider e avvia
                l'interrogazione dell'API (eseguendo la chiamata in tempo reale
                se $STREAM_MODE è attivo, oppure richiamando la routine
                perform_request_once).
        2.  Chat Interattiva ($CHAT_MODE attivo): Delega integralmente il ciclo
            di chat interattiva multi-turno ed allocazione del terminale TTY
            all'utility esterna tui-repl.sh per ottimizzare l'isolamento ed
            isolamento di processo.
        3.  Esecuzione Standard di Singolo Prompt (Streaming o Sincrona):
              - Se è definito un thread operativo ($THREAD_ID), esegue la
                validazione dell'ID, determina la finestra di recupero, e
                compila il file storico dei messaggi $BUILD_MESSAGES_FILE
                interrogando il Session Engine o richiamando la routine legacy
                di lettura NDJSON.
              - Compila il payload del file JSON via build_payload_from_vars.
              - Se si opera in esecuzione reale (non simulata), valida la
                presenza della chiave API del provider.
              - Se streaming attivo ($STREAM_MODE == 1): Invoca la funzione di
                streaming dedicatale (call_api_streaming) sotto lock esclusivo
                di thread. Se completato con successo:
                  - Se vi è un thread attivo, sanifica ed accoda il messaggio
                    dell'utente al registro storico (via session_engine_append o
                    la legacy thread_append).
                  - Se la connessione ha generato risposta reale ed il file
                    $RESP contiene dati validi, estrae il testo generato
                    dall'assistente, ne esegue la sanificazione dai ritorni a
                    capo ed accoda in modo sicuro la risposta dell'assistente
                    nel file NDJSON storico del thread (delegandola al Session
                    Engine o alla legacy thread_append).
                  - Stampa un carattere a capo e termina l'esecuzione con
                    stato 0. In caso di errore termina con errore di curl.
              - Se sincrona standard ($STREAM_MODE == 0): Stampa i dettagli
                diagnostici di visualizzazione del prompt e del modello se in
                terminale interattivo verboso. Invoca la chiamata transazionale
                via perform_request_once sotto lock esclusivo di thread. In caso
                di successo:
                  - Se è configurato un thread, sanifica e registra il messaggio
                    dell'utente nel file storico. Se l'esecuzione è reale ed il
                    file di risposta contiene dati, estrae il testo generato
                    dall'assistente, lo ripulisce e registra in modo sicuro la
                    risposta dell'assistente nel file NDJSON di storico. Esce
                    con stato 0. In caso di fallimento della chiamata, accoda
                    comunque il messaggio utente al thread operativo
                    contrassegnandolo come interazione fallita ed interrompe
                    l'esecuzione restituendo lo stato d'uscita della
                    perform_request_once.
