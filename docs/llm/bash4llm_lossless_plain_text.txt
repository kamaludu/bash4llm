SPECIFICA DELL'ARCHITETTURA E DELLA LOGICA DI BASH4LLM⁺ (v2.5.0)

Questo documento descrive in modo completo e senza perdita di informazioni la
logica, le variabili, le funzioni e i blocchi strutturali del sistema Bash4LLM⁺,
suddivisi secondo le cinque macro-sezioni reali.

1. PRECORE_BOOT

VARIABILI

  - SCRIPT_NAME: Nome identificativo del programma (impostato a "bash4llm").
  - SCRIPT_VERSION: Versione corrente del software (impostata a "2.5.0").
  - SCRIPT_DATE: Data di rilascio dello script (impostata a "2026-07-18").
  - BASH4LLM_PLAT_ANDROID: Flag booleano (valore 1 o 0) impostato su bootstrap
    indicante se il runtime è eseguito sotto ambiente Termux/Android.
  - BASH4LLM_PLAT_MACOS: Flag booleano (valore 1 o 0) impostato su bootstrap
    indicante se il runtime è eseguito sotto sistema macOS/Darwin.
  - BASH4LLM_PLAT_WSL: Flag booleano (valore 1 o 0) impostato su bootstrap
    indicante se il runtime è eseguito sotto Windows Subsystem for Linux (WSL).
  - BASH4LLM_PLAT_CYGWIN: Flag booleano (valore 1 o 0) impostato su bootstrap
    indicante se l'esecuzione avviene sotto ambienti Windows compatibili Cygwin,
    MSYS2 o MinGW.
  - BASH4LLM_PLAT_BSD: Flag booleano (valore 1 o 0) impostato su bootstrap
    indicante se l'esecuzione avviene sotto sistemi FreeBSD, OpenBSD, NetBSD o
    DragonFly.
  - BASH4LLM_PLAT_LINUX: Flag booleano (valore 1 o 0) impostato su bootstrap
    indicante se l'esecuzione avviene sotto ambiente Linux generico o WSL.
  - BASH4LLM_MODEL_CACHE: Array associativo adibito ad archiviare in memoria i
    nomi dei modelli normalizzati per evitare ricalcoli regex costosi in tempo
    di esecuzione.
  - BASH4LLM_ERR_NO_API_KEY: Costante numerica di errore (valore 10) associata
    alla mancanza di una chiave API.
  - BASH4LLM_ERR_BAD_MODEL: Costante numerica di errore (valore 11) associata
    alla richiesta di un modello non valido o non supportato.
  - BASH4LLM_ERR_CURL_FAILED: Costante numerica di errore (valore 12) associata
    al fallimento della chiamata di rete curl.
  - BASH4LLM_ERR_NO_PROMPT: Costante numerica di errore (valore 14) associata
    alla mancanza di un prompt di input.
  - BASH4LLM_ERR_TMP: Costante numerica di errore (valore 15) associata a
    problemi del file-system temporaneo o dei file di lock.
  - BASH4LLM_ERR_API: Costante numerica di errore (valore 16) associata a
    risposte d'errore o codici HTTP non validi restituiti dal provider API.
  - BASH4LLM_ERR_SEC: Costante numerica di errore (valore 17) associata a
    vulnerabilità sul file-system o violazioni della policy di sicurezza
    statica.
  - BASH4LLMERR_NO_API_KEY: Mappatura alias diretta del valore di
    BASH4LLM_ERR_NO_API_KEY con notazione ad underscore.
  - BASH4LLMERR_BAD_MODEL: Mappatura alias diretta del valore di
    BASH4LLM_ERR_BAD_MODEL con notazione ad underscore.
  - BASH4LLMERR_CURL_FAILED: Mappatura alias diretta del valore di
    BASH4LLM_ERR_CURL_FAILED con notazione ad underscore.
  - BASH4LLMERR_NO_PROMPT: Mappatura alias diretta del valore di
    BASH4LLM_ERR_NO_PROMPT con notazione ad underscore.
  - BASH4LLMERR_TMP: Mappatura alias diretta del valore di BASH4LLM_ERR_TMP con
    notazione ad underscore.
  - BASH4LLMERR_API: Mappatura alias diretta del valore di BASH4LLM_ERR_API con
    notazione ad underscore.
  - BASH4LLMERR_SEC: Mappatura alias diretta del valore di BASH4LLM_ERR_SEC con
    notazione ad underscore.
  - DEBUG: Flag di configurazione per l'output di tracciamento dello sviluppo.
    Eredita il valore di ambiente ${DEBUG} o ${BASH4LLM_DEBUG}, o viene
    impostato su 0 (disattivo).
  - BASH4LLM_LOG: Percorso assoluto o relativo del file in cui registrare i log
    di tracciamento. Eredita ${BASH4LLM_LOG} (vuoto se disattivato).
  - SCRIPTDIR: Percorso assoluto della directory contenente lo script corrente,
    risolto dinamicamente tramite la funzione resolve_script_dir.
  - BASH4LLM_DIR: Directory principale di installazione dei dati e delle
    configurazioni di Bash4LLM⁺. Dichiarata dall'ambiente o derivata da
    SCRIPTDIR/bash4llm.d.
  - BASH4LLM_CONFIG_DIR: Directory dei file di configurazione utente
    persistenti. Impostata a BASH4LLM_CONFIG_DIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/config.
  - BASH4LLM_MODELS_DIR: Directory dei file relativi ai modelli supportati.
    Impostata a BASH4LLM_MODELS_DIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/models.
  - BASH4LLM_TEMPLATES_DIR: Directory contenente i file dei modelli di template.
    Impostata a BASH4LLM_TEMPLATES_DIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/templates.
  - BASH4LLM_HISTORY_DIR: Directory contenente lo storico e i log delle
    interazioni dello script. Impostata a BASH4LLM_HISTORY_DIR di ambiente o
    posizionata sotto $BASH4LLM_DIR/history.
  - BASH4LLM_TMPDIR: Directory radice per l'allocazione dei dati temporanei di
    Bash4LLM⁺. Impostata a BASH4LLM_TMPDIR di ambiente o posizionata sotto
    $BASH4LLM_DIR/tmp.
  - CANONICAL_EXTRAS_DIR: Percorso della directory canonica delle estensioni,
    calcolato rimuovendo la barra finale da BASH4LLM_DIR e annettendo /extras.
  - LEGACY_EXTRAS_DIR: Percorso storico di fallback delle estensioni, relativo
    alla cartella dello script (SCRIPTDIR/extras).
  - BASH4LLM_EXTRAS_DIR: Directory attiva per l'accesso alle risorse aggiuntive
    dello script (assegnata a CANONICAL_EXTRAS_DIR).
  - PROVIDERS_DIR: Percorso assoluto o relativo dei moduli esterni dei provider
    (impostato su PROVIDERS_DIR di ambiente o, in alternativa, localizzato sotto
    la directory delle estensioni alla voce /providers).
  - THREAD_DIR: Directory dedicata alla registrazione dei log storici NDJSON dei
    thread operativi (posizionata in $BASH4LLM_HISTORY_DIR/threads).
  - BASH4LLM_VAULT_ENABLED: Variabile di controllo per l'abilitazione (1) o
    disattivazione (0) del Key Vault OpenSSL. Prelevata dal file config o
    impostata di default a 1.
  - BASH4LLM_OPENSSL_ACTIVE: Flag booleano interno indicante se
    openssl-helper.sh è caricato con successo nel runtime (valore 1 o 0).
  - MODELS_FILE: File di testo locale in cui memorizzare la lista dei modelli
    recuperati, sincronizzato tramite sync_models_file_path con pattern
    $BASH4LLM_MODELS_DIR/.txt.
  - MAX_MODELS: Limite numerico massimo di modelli importabili o gestibili
    localmente. Eredita ${MAX_MODELS} o adotta il valore di default di 200.
  - PROVIDER_FILE: Percorso del file che persiste l'ultimo provider configurato
    dall'utente, determinato tramite canonical_provider_file.
  - MODELS_LOCK: Percorso del file di lock di sincronizzazione per l'accesso ai
    modelli, impostato su MODELS_LOCK di ambiente o localizzato sotto
    $BASH4LLM_MODELS_DIR/models.lock.
  - HISTORY_LOCK: Percorso del file di lock per l'accesso concorrente ai log di
    storico e sessione, impostato su HISTORY_LOCK di ambiente o localizzato
    sotto $BASH4LLM_HISTORY_DIR/history.lock.
  - TMP_LOCK: Percorso del file di lock esclusivo per la gestione sicura della
    directory temporanea dello script, impostato su TMP_LOCK di ambiente o
    localizzato sotto $BASH4LLM_TMPDIR/tmp.lock.
  - BASH4LLM_LOCK_TIMEOUT_TMP: Timeout massimo di attesa espresso in secondi per
    acquisire il lock temporaneo (eredita BASH4LLM_LOCK_TIMEOUT_TMP o
    imposta 10).
  - BASH4LLM_LOCK_TIMEOUT_MODELS: Timeout in secondi per acquisire il lock sui
    modelli (eredita BASH4LLM_LOCK_TIMEOUT_MODELS o imposta 10).
  - BASH4LLM_LOCK_TIMEOUT_HISTORY: Timeout in secondi per acquisire il lock
    sullo storico dei messaggi e dei thread (eredita
    BASH4LLM_LOCK_TIMEOUT_HISTORY o imposta 10).

FUNZIONI

  - resolve_script_dir: Identifica la cartella reale in cui risiede lo script in
    esecuzione, risolvendo i collegamenti simbolici sul file-system. Input:
    Nessuno (analizza la variabile speciale ${BASH_SOURCE[0]}). Output:
    Restituisce su stdout il percorso assoluto normalizzato della directory.
    Dipendenze: readlink, dirname, cd, pwd, printf.
  - safe_mkdir: Crea in sicurezza una directory con i permessi specificati,
    verificando che non coincida con un collegamento simbolico (mitigazione
    symlink attack). Input: $1 (directory), $2 (opzionale: permessi ottali,
    default 700). Output: Stato 0 o arresto immediato con codice d'uscita 15
    (BASH4LLM_ERR_TMP) se fallisce. Dipendenze: mkdir, chmod, printf.
  - check_required_arg: Convalida la presenza del parametro CLI obbligatorio
    richiesto da un'opzione, verificando la prossimità del limite posizionale
    degli argomenti. Input: $1 (opzione indagata), $2 (conteggio argomenti
    residui). Output: Stato 0 o arresto con codice 15. Dipendenze: log_error.
  - canonical_config_dir: Normalizza il percorso della directory di
    configurazione rimuovendo slash finali. Input: Nessuno
    ($BASH4LLM_CONFIG_DIR). Output: Restituisce il percorso su stdout.
    Dipendenze: printf.
  - canonical_provider_file: Genera il percorso canonico del file di persistenza
    del provider. Input: Nessuno. Output: Percorso del file /provider su stdout.
    Dipendenze: printf, canonical_config_dir.
  - canonical_model_file: Genera il percorso del file in cui memorizzare il
    modello di default per il provider attivo. Input: $1 (provider). Output:
    Percorso del file /model.<provider> su stdout. Dipendenze: printf,
    canonical_config_dir.
  - canonical_provider_url_file: Genera il percorso del file contenente l'URL
    API del provider attivo. Input: Nessuno. Output: Percorso del file
    /provider-url su stdout. Dipendenze: canonical_config_dir, printf.
  - trim_space: Rimuove spazi e tabulazioni iniziali e finali tramite costrutti
    di scomposizione nativi della shell (senza fork). Input: $1 (stringa).
    Output: Stringa pulita su stdout. Dipendenze: printf.
  - sync_models_file_path: Sincronizza il percorso globale di MODELS_FILE
    basandosi sul provider attivo, forzando una sanitizzazione protettiva contro
    directory traversal che esclude caratteri non alfanumerici (eccetto trattino
    e trattino basso). Input: $1 (opzionale: provider). Output: Esporta la
    variabile MODELS_FILE in ambiente. Dipendenze: export.
  - validate_file_input: Esegue la convalida di sicurezza di un file di input
    per scopi transazionali, verificando l'assenza di byte nulli, DEL o
    controlli C0 incompatibili (tranne tab, CR, LF). Input: $1 (percorso file).
    Output: Codici stato: 0 (valido), 1 (percorso vuoto), 2 (non leggibile), 3
    (0 byte), 4 (violazioni binarie). Dipendenze: tr, head, file_size.
  - _normalize_model_name: Esegue la normalizzazione dei nomi dei modelli
    rimuovendo spazi, ritorni a capo ed i prefissi models/ e <provider>. Per
    ottimizzare il runtime, conserva i risultati in un array associativo di
    cache in memoria. Input: $1 (nome modello). Output: Nome normalizzato su
    stdout. Dipendenze: normalize_model_.
  - ensure_api_key_for_provider: Valida e acquisisce la chiave API per il
    provider selezionato. Tenta di recuperare la chiave dal Key Vault
    crittografico OpenSSL se attivo ($BASH4LLM_OPENSSL_ACTIVE=1). In ambienti
    non interattivi esce con errore se assente. In ambienti interattivi (TTY),
    richiede l'inserimento allo standard input, sanifica la stringa (rimuovendo
    spazi, export, assegnazioni) e la esporta visualizzando le istruzioni di
    salvataggio. Input: $1 (provider). Output: Stato 0 o BASH4LLM_ERR_NO_API_KEY
    (10). Esporta la chiave API. Dipendenze: provider_api_env_var_name,
    vault_exists, vault_load_keys, jq, printf, export.
  - enforce_network_policy: Determina se le chiamate di rete sono abilitate
    analizzando $DRY_RUN, $BASH4LLM_SKIP_NETWORK,
    $BASH4LLM_ENFORCE_NO_NETWORK_IF_QUIET e $QUIET. Input: Nessuno. Output:
    Stato 1 (bloccate) o 0 (autorizzate). Dipendenze: is_truthy, log_info.
  - _extract_notes_section: Estrae sezioni testuali letterali (senza esecuzione
    di codice) dal file di documentazione core-notes.sh. Input: $1 (titolo
    sezione target). Output: Righe estratte su stdout o stato 1 se assente.
    Dipendenze: awk.
  - log_prefix: Restituisce il prefisso di log "bash4llm: ". Input: Nessuno.
    Output: Prefisso su stdout. Dipendenze: printf.
  - log_info: Registra messaggi informativi. Se $DEBUG=1 scrive su stderr; se
    $BASH4LLM_LOG è definito, appende il record con timestamp UTC. Input: $1
    (categoria, default INFO), $2 (messaggio). Dipendenze: log_prefix, date,
    printf.
  - log_warn: Registra messaggi di avviso non critici su stderr e file
    $BASH4LLM_LOG. Input: $1 (modulo), $2 (messaggio). Dipendenze: log_prefix,
    date, printf.
  - log_error: Registra errori bloccanti su stderr e file $BASH4LLM_LOG. Input:
    $1 (modulo), $2 (messaggio). Dipendenze: log_prefix, date, printf.
  - log_info_user: Registra messaggi informativi di esecuzione destinati
    all'utente (non silenziati da QUIET) su stderr e file log. Input: $1
    (modulo), $2 (messaggio). Dipendenze: log_prefix, date, printf.
  - dbg: Stampa diagnostica su stderr se $DEBUG=1. Input: $* (argomenti).
    Dipendenze: printf.
  - ensure_config_dir: Assicura la presenza e l'accessibilità fisica della
    cartella config. La crea via safe_mkdir, imposta permessi 700 e valida la
    scrivibilità tramite scrittura e rimozione immediata di un file di test.
    Input: Nessuno. Output: Stato 0 o 1. Dipendenze: safe_mkdir, chmod, rm,
    log_error.
  - write_provider_url_if_missing: Registra l'URL API del provider in modo
    transazionale tramite un file temporaneo di staging, applicandovi
    permessi 600. Input: $1 (provider), $2 (URL). Output: Stato 0 o 1.
    Dipendenze: canonical_provider_url_file, dirname, safe_mkdir, mktemp,
    printf, mv, cp, rm, chmod.
  - resolve_provider_url: Identifica ed esporta l'URL API per il provider
    selezionato. Controlla prioritariamente BASH4LLM_API_URL o
    BASH4LLM_PROVIDER_URL. In assenza, estrae la prima riga dal file
    /provider-url. Se mancante ed attivo è "groq", esporta l'endpoint nativo
    Groq. Input: $1 (provider, default $PROVIDER). Output: Stato 0 o 1. Esporta
    BASH4LLM_PROVIDER_URL. Dipendenze: canonical_provider_url_file, sed, awk,
    export.
  - provider_api_env_var_name: Calcola il nome standardizzato della variabile
    d'ambiente deputata a contenere la chiave API del provider (es: "gemini" ->
    "GEMINI_API_KEY") sfruttando l'espansione nativa della shell. Input: $1
    (provider). Output: Nome normalizzato su stdout. Dipendenze: printf.
  - is_valid_json_string: Valida la conformità sintattica JSON di una stringa.
    Input: $1 (stringa). Output: Stato 0 o 1. Dipendenze: printf, jq.
  - b64encode: Esegue la codifica Base64 a riga singola in modo portabile,
    tentando prioritariamente openssl enc. Input: stdin. Output: Stringa
    codificata su stdout. Dipendenze: openssl, base64, tr.
  - b64decode: Esegue la decodifica Base64 isolando flussi tramite awk (folding
    righe a 64 caratteri) per prevenire troncamenti anomali con LibreSSL/BSD.
    Input: stdin. Output: Flusso decodificato su stdout. Dipendenze: awk,
    openssl, base64.
  - is_truthy: Valuta se la stringa è vera (1, true, TRUE, True, yes, YES, Yes).
    Input: $1 (stringa). Output: Stato 0 (vero) o 1 (falso).
  - file_size: Restituisce la dimensione del file in byte sfruttando le
    definizioni di piattaforma. Input: $1 (percorso). Output: Byte su stdout.
    Dipendenze: stat, printf.
  - is_valid_json_file: Valida l'esistenza e conformità JSON di un file. Input:
    $1 (percorso). Output: Stato 0 o 1. Dipendenze: jq.
  - stage_b64: Codifica Base64 transazionale di un payload testuale. Controlla
    che non ecceda $MAX_STAGE_BYTES (default 10MB), scrive in temp, acquisisce
    lock esclusivo e sposta a destinazione finale applicando chmod 600. Input:
    $1 (sorgente; se vuoto legge stdin), $2 (destinazione). Dipendenze: dirname,
    safe_mkdir, _tmpf, cat, rm, file_size, base64, tr, chmod, lock_exec, mv,
    log_error, log_info.
  - lock_exec: Esegue un comando sotto flock esclusivo o directory lock
    provvisorio in base a piattaforma, umask ed mtime orfani, bloccando attacchi
    basati su symlink. Input: $1 (file lock), $2 (timeout, default 10), seguiti
    da -- e dal comando. Output: Stato comando, 124 timeout, 15 symlink.
    Dipendenze: dirname, safe_mkdir, flock, printf, mkdir, rmdir, log_error.
  - _mktemp_in_dir: Alloca un file temporaneo univoco all'interno della cartella
    di base. Input: $1 (cartella), $2 (prefisso, default groq). Output: Percorso
    completo su stdout o BASH4LLM_ERR_TMP. Dipendenze: _tmpf, log_error.
  - show_payload_head: Visualizza a scopo diagnostico su stderr le prime n righe
    del payload di richiesta (decodificando al volo se .b64). Input: $1
    (payload, default $PAYLOAD), $2 (righe, default 200). Dipendenze: printf,
    grep, b64decode, head.
  - atomic_write: Scrittura atomica e sicura da stdin. Scrive su file temporaneo
    nella directory finale, applica permessi 600, acquisisce lock esclusivo ed
    esegue lo spostamento atomico definitivo (mv). Input: $1 (destinazione), $2
    (timeout, default 10). Output: Stato 0 o errore. Dipendenze: dirname,
    safe_mkdir, mktemp, cat, rm, chmod, lock_exec, mv, log_error.
  - extract_text_from_resp: Estrae con resilienza il testo dalle strutture JSON
    dei provider (completions, delta, output_text, scalari stringa). Esegue una
    copia provvisoria isolata per stabilità di I/O. Input: Nessuno ($RESP).
    Output: Stato 0 (testo su stdout), 1 (errore/vuoto), 2 (diagnostico).
    Dipendenze: _tmpf, cp, is_valid_json_file, jq, log_warn, log_error, rm.
  - cleanup_run_tmp_on_exit: Ripristina lo stato del terminale (stty echo),
    rilascia i file lock del thread attivo ed esegue la rimozione ricorsiva
    sicura di RUN_TMPDIR (tranne se $DEBUG_PRESERVE=1). Input: Nessuno.
    Dipendenze: stty, release_thread_lock, rm, log_info.
  - ensure_run_tmpdir: Configura l'ambiente temporaneo istanza $RUN_TMPDIR sotto
    $BASH4LLM_TMPDIR. Inizializza $PAYLOAD, $RESP, $ERRF (700 dir, 600 file) e
    imposta la trap su cleanup_run_tmp_on_exit. Input: $1 (opzionale --print).
    Output: Stato 0 o BASH4LLM_ERR_TMP. Esporta RUN_TMPDIR, PAYLOAD, RESP, ERRF.
    Dipendenze: safe_mkdir, chmod, _tmpf, rm, trap, log_warn, log_error, printf.
  - b64_atomic_write: Codifica Base64 stdin e scrive in modo atomico a
    destinazione sotto lock (chmod 600). Input: $1 (destinazione), $2 (timeout,
    default 10). Output: Stato 0 o BASH4LLM_ERR_TMP. Dipendenze: dirname, mkdir,
    _tmpf, b64encode, chmod, lock_exec, mv, rm, log_error.
  - b64_atomic_read: Legge in sicurezza, decodifica ed invia a stdout un file
    codificato in Base64. Input: $1 (percorso). Dipendenze: b64decode.
  - ui_state_write: Scrive in modo atomico lo stato UI sotto
    $BASH4LLM_CONFIG_DIR/ui_state o RUN_TMPDIR con permessi 600 e lock
    esclusivo. Input: $1 (nome file), $2 (JSON). Dipendenze: safe_mkdir, chmod,
    atomic_write, wc, log_info, log_warn.
  - run_static_config_check: Linter di conformità statica della configurazione.
    Convalida che il file config non sia scrivibile da terzi (group/world
    writable) lanciando errore 17 in caso di violazione, ed esegue l'analisi
    sintattica dei parametri locali contro typo o definizioni scorrette tramite
    le definizioni in core-notes.sh. Input: Nessuno. Output: Stato 0 o esce con
    codice 17 (BASH4LLM_ERR_SEC). Dipendenze: stat, awk, sed, tr, grep, printf.
  - explain_error_code: Associa un codice d'errore o alias (es. 15 o
    BASH4LLM_ERR_TMP) alla sua definizione ufficiale e alle indicazioni
    operative di risoluzione prelevate in modalità zero-eval da core-notes.sh.
    Input: $1 (codice o alias). Output: Definizione strutturata stampata su
    stderr. Dipendenze: printf, _extract_notes_section, grep.
  - load_provider_module: Importa in sicurezza moduli dei provider esterni.
    Controlla group/world writable (salvo WSL o IGNORE_SEC), convalida owner,
    esegue dry-run (bash -n) e convalida la firma del file pre-post source
    contro Race Condition. Verifica la conformità delle funzioni buildpayload e
    call_api in sotto-shell isolata prima di importarle nel runtime, gestendo
    l'opzione nounset. Scrive provider_capabilities.json. Input: $1 (provider).
    Output: Stato 0 o 1. Dipendenze: safe_mkdir, _is_world_writable, _get_owner,
    _get_perm_string, getfile_signature, bash, _mktemp_in_dir, awk, sort, comm,
    grep, type, declare, rm, ui_state_write, jq.
  - _detect_base64_opts: Rileva le opzioni Base64 native ed esporta
    $B64_WRAP_OPT e $B64_DECODE_OPT. Input: Nessuno. Dipendenze: printf, base64,
    grep.
  - list_files_sorted_by_mtime: Elenca file ordinati per modifica mtime
    crescente (portabile Linux/Darwin). Input: $1 (cartella). Dipendenze: find,
    stat, sort.
  - tac_fallback: Inverte l'ordine delle righe di un file tramite tac o fallback
    awk. Input: $1 (file). Dipendenze: command, tac, awk.
  - _file_mtime: Ottiene data ultima modifica file portabile (Unix timestamp, 0
    se assente). Input: $1 (file). Dipendenze: stat.
  - jq_safe: Filtro jq sicuro. Se fallisce scrive i log diagnostici in $ERRF
    senza interrompere lo script. Input: $1 (filtro), $2 (file). Dipendenze: jq.

BLOCCHI DI CODICE

  - PRECORE_BOOT_SETUP_SHELL - Fase: Inizializzazione primaria. - Azione:
    Imposta set -euo pipefail solo se eseguito direttamente per evitare shell
    pollution. Disabilita core dumps (ulimit -c 0), esporta le proprietà di
    identità del software (v2.5.0 e data 18 Luglio 2026).
  - PRECORE_BOOT_ENVIRONMENT_AUTODETECTION - Fase: Rilevamento piattaforma. -
    Azione: Rileva dinamicamente la piattaforma host (Android, macOS, WSL,
    Cygwin, BSD, Linux) ed esporta globalmente i corrispondenti flag booleani
    BASH4LLM_PLAT_... Inizializza l'associativa BASH4LLM_MODEL_CACHE.
  - PRECORE_BOOT_SETUP_ENV_CMDS - Fase: Validazione requisiti. - Azione:
    Controlla la presenza di sistema di tutte le utilità necessarie (escluso
    flock per portabilità macOS/BSD/Termux, includendo comm). Esce con stato 1
    in caso di errore.

  - FILE_INPUTS: Array percorsi file input passati con -f da unire al contesto.
  - ARGS: Array argomenti posizionali CLI (trattati come prompt o file).
  - OUTPUT_MODE: Formato visualizzazione output (eredita ${OUTPUT_MODE}, default
    "text").
  - MODEL_CLI_SET: Flag interno modello esplicito da CLI (-m/--model)
    (default 0).
  - INSTALL_EXTRAS_SRC: Directory sorgente estensioni da installare.
  - BOOTSTRAP_ONLY: Flag arresto post bootstrap (sola importazione moduli)
    (default 0).
  - INIT_THREAD: Flag che richiede l'inizializzazione transazionale di un thread
    (default 0).
  - DELETE_THREAD: Flag che richiede la rimozione fisica ed atomica di un thread
    (default 0).
  - DELETE_THREAD_ID: Identificatore alfanumerico del thread da eliminare.
  - RENAME_THREAD: Flag che richiede la ridenominazione dei metadati di un
    thread (default 0).
  - RENAME_THREAD_ID: Identificatore alfanumerico del thread da rinominare.
  - RENAME_TITLE: Titolo testuale da applicare al thread rinominato.
  - _engine_path: Percorso script Session Engine
    ($BASH4LLM_EXTRAS_DIR/session/session-engine.sh).
  - _engine_available: Flag disponibilità operativa Session Engine (valore 1
    o 0, default 0).
  - _supported_providers_arr: Array interno provider installati rilevati.

FUNZIONI

  - validate_provider_interface: Convalida che il modulo del provider caricato
    implementi correttamente le specifiche dell'interfaccia. Verifica la
    presenza delle funzioni obbligatorie (buildpayload_ e call_api_), e traccia
    a fini diagnostici la presenza/assenza dei metodi opzionali. Input: $1
    (provider). Output: Stato 0 (conforme) o 1 (errore loggato). Dipendenze:
    type, log_error, log_info.
  - call_provider: Esegue funzione dinamica modulo provider caricato in memoria.
    Input: $1 (funzione), argomenti. Output: Stato uscita o 127 se non definita.
    Dipendenze: type.
  - validate_provider_key_dispatch: Dispaccia la validazione della chiave API
    delegando alla funzione dedicata del provider attivo (es.
    validate_key_groq). Input: $1 (chiave_api). Output: Codice stato del
    validatore, o 127 se non implementata. Dipendenze: type.
  - refresh_models_dispatch: Dispaccia aggiornamento modelli a provider attivo
    (es. refresh_models_groq). Tenta con percorso file dest, fallback senza
    argomenti se firma incompatibile. Input: $1 (file_dest, default
    $MODELS_FILE). Output: Stato 0, errore o 127. Dipendenze: type, log_error,
    log_info.
  - validate_model_dispatch: Dispaccia validazione modello a provider attivo. Se
    non supportata, log warn una-tantum e convalida permissiva di fallback.
    Input: $1 (modello). Output: Stato uscita validatore o 0. Dipendenze: type,
    log_warn.
  - auto_select_model_dispatch: Dispaccia l'auto-selezione del modello alla
    funzione nativa del provider attivo (es. auto_select_model_groq). Input:
    Nessuno. Output: Nome del modello su stdout e stato d'uscita. Dipendenze:
    type.
  - resolve_model: Risolve/assegna modello finale con precedenza: 1. CLI
    (-m/--model), 2. file canonico $PROVIDER_FILE (fallback 'groq'), 3. prima
    riga model., 4. auto-selezione auto_select_model_provider, 5. scansione riga
    per riga di MODELS_FILE, 6. config globale, 7. whitelist $ALLOWED_MODELS.
    Errore se falliscono tutti. Input: Nessuno. Output: Stato 0 (assegna $MODEL)
    o 1. Side-effects: Scansione file-system, validazione modelli. Dipendenze:
    canonical_model_file, canonical_provider_file, sed, awk,
    validate_model_core, validate_model_dispatch, auto_select_model_dispatch,
    _normalize_model_name, is_supported_model, log_warn, log_error.
  - build_payload_from_vars: Orchestra costruzione JSON payload delegando a
    provider attivo (buildpayload_provider). Assicura RUN_TMPDIR. Input:
    Nessuno. Output: Generazione file payload o arresto errore API. Dipendenze:
    ensure_run_tmpdir, call_provider, log_error.
  - call_api_once: Chiamata sincrona non-streaming via provider. Se dry-run,
    esclude rete e print head payload. Input: Nessuno. Output: Stato uscita o 0.
    Dipendenze: call_provider, show_payload_head, log_info, log_error.
  - call_api_streaming: Chiamata streaming via provider sotto lock di
    concorrenza del thread attivo ($THREAD_ID). Implementa un meccanismo di
    ripristino sicuro del segnale d'interruzione (SIGINT) per sbloccare l'fd
    lock del thread orfano in caso di uscita abrupta. Se dry-run, simula e print
    head payload. Input: Nessuno. Output: Stato uscita o 0. Side-effects:
    Gestione dei segnali e dei lock concorrenziali del database. Dipendenze:
    acquire_thread_lock, release_thread_lock, call_provider, show_payload_head,
    log_info, log_error.
  - extract_api_error: Estrae errore formale da risposta RESP (ispeziona campi
    errore JSON o prima riga testo crudo). Input: Nessuno (usa $RESP). Output:
    Messaggio errore su stdout. Dipendenze: jq, awk.
  - detect_empty_edge_case: Rileva risposte vuote anomale API (HTTP 200,
    finish_reason 'stop', 0 token, no testo). Estrae request ID, finish reason,
    tokens. Copia sicura temp. Input: Nessuno. Output: Assegna
    BASH4LLM_EDGE_EMPTY (0/1), BASH4LLM_EDGE_REQ_ID,
    BASH4LLM_EDGE_FINISH_REASON, BASH4LLM_EDGE_COMPLETION_TOKENS. Dipendenze:
    _tmpf, cp, is_valid_json_file, jq, rm.
  - finalize_and_output: Formatta, visualizza (text, raw, json, pretty) e salva
    output. Salva su disco (in OUT_PATH o history standard) via save_to_history
    sotto lock concorrenziale con umask blindata. Input: $1 (formato), $2
    (testo). Output: Output stampato, stato 0 o errore. Side-effects: Scrittura,
    salvataggio file storico transazionale, creazione directory. Dipendenze:
    _tmpf, cp, cat, jq, printf, wc, tr, date, safe_mkdir, dirname, lock_exec,
    log_error, log_warn.
  - perform_request_once: Esegue richiesta sincrona sotto lock concorrenziale di
    thread. Gestisce interruzioni ripristinando in sicurezza l'fd lock del
    thread. Retry automatico fino a $MAX_RETRIES in caso di errori rete
    temporanei, con backoff lineare. Estrae testo, diagnostica casi vuoti,
    gestisce errori API, scrive stato in last_api.json, formatta output. Input:
    Nessuno. Output: Stato 0 o errore BASH4LLM_ERR_API. Side-effects: Chiamate
    rete, sleep, aggiornamento stato UI, log stderr, gestione dei segnali e del
    file fd lock. Dipendenze: acquire_thread_lock, release_thread_lock,
    call_api_once, extract_text_from_resp, detect_empty_edge_case, jq,
    ui_state_write, extract_api_error, finalize_and_output, sleep, log_info,
    log_error, log_warn.
  - collect_input_from_files: Concatena file input con delimitatori strutturati
    visivi. Esce se un file non è leggibile (con deviazione testuale d'ambiente
    del loop ereditata). Input: Array percorsi. Output: Testo unificato su
    stdout. Dipendenze: file_readable, cat, log_error.
  - expand_args_to_content: Converte argomenti CLI in prompt. Se argomento è
    file esistente e leggibile ne riversa il contenuto formattato, altrimenti lo
    accoda come testo. Input: Array ARGS. Output: Prompt unificato su stdout.
    Dipendenze: file_readable, cat.
  - file_readable: Verifica se file esiste ed è regolare e leggibile da utente
    corrente. Input: $1 (percorso). Output: Stato 0 o 1.
  - trim: Rimuove spazi/tab/a capo iniziali e finali. Input: $1 (stringa).
    Output: Stringa pulita su stdout. Dipendenze: awk, printf.
  - is_number: Verifica se stringa è valore numerico valido (intero o decimale).
    Input: $1 (stringa). Output: Stato 0 o 1. Dipendenze: awk, printf.
  - is_supported_model: Filtra/convalida compatibilità modello. Esclude
    (ritorna 1) formati non-testuali: immagini, visione/VQA,
    audio/speech/whisper, tts, embedding, vettori, multimodali. Ammette solo
    testo. Input: $1 (modello). Output: Stato 0 (testo) o 1 (escluso).
    Dipendenze: tr.
  - list_models_cli: Visualizza modelli locali in stderr normalizzandoli con
    _normalize_model_name ed evidenziando i non supportati (non-testuali) per
    evitarne la scelta. Input: Nessuno. Output: Stato 0 o 1. Dipendenze:
    _normalize_model_name, is_supported_model, printf.
  - validate_model_core: Algoritmo centrale validazione modelli. Normalizza nome
    tramite _normalize_model_name, controlla presenza in file modelli locali,
    verifica is_supported_model. Input: $1 (modello). Output: Stato 0 o 1.
    Dipendenze: _normalize_model_name, grep, awk, is_supported_model, printf.
  - load_local_config: Carica parametri config locale (config sotto
    $BASH4LLM_CONFIG_DIR) in memoria (interpreta key=val, ignora
    commenti/vuote), assegna globali (MODEL, TEMPERATURE/TURE, MAX_TOKENS,
    OUTPUT_MODE/FORMAT, THRESHOLD). Input: Nessuno. Output: Assegna variabili
    globali.
  - load_whitelist: Popola whitelist modelli autorizzati scansionando in
    sicurezza MODELS_FILE tramite normalizzazione riga per riga di
    _normalize_model_name. Assegna a ALLOWED_MODELS separati da newline. Input:
    Nessuno. Output: Assegna ALLOWED_MODELS. Dipendenze: _normalize_model_name.
  - is_tty_out: Verifica se stdout è terminale interattivo (TTY) reale. Input:
    Nessuno. Output: Stato 0 (TTY) o 1.

BLOCCHI DI CODICE

5. CORE_PROVIDER

VARIABILI

  - _supported_providers_arr: Array interno per l'indicizzazione dei provider
    rilevati.
  - SUPPORTED_PROVIDERS: Elenco testuale unificato dei provider supportati
    separati da caratteri di spazio.

FUNZIONI

  - validate_provider_interface: Convalida che il modulo del provider caricato
    in memoria implementi correttamente le specifiche dell'interfaccia. Verifica
    la presenza fisica delle funzioni obbligatorie (buildpayload_ e call_api_),
    e traccia a fini diagnostici la presenza/assenza dei metodi opzionali.
    Input: $1 (provider). Output: Stato 0 (conforme) o 1 (errore loggato).
    Dipendenze: type, log_error, log_info.
  - assemble_content: Compila ed organizza la stringa del prompt di input
    principale $CONTENT che verrà inviata al modello LLM. Segue una catena di
    regole di priorità esclusive: 1. Se è configurato un JSON utente in
    $JSON_INPUT, pulisce la variabile
    CONTENT ed esce (in quanto il JSON di input definisce autonomamente i messaggi). 2. Se l'utente ha inserito dei file via parametro -f (FILE_INPUTS),
    ne raccoglie ed unifica i contenuti via collect_input_from_files. Se sono
    presenti argomenti posizionali aggiuntivi, li annette in coda come frammenti
    testuali. 3. Se è richiesto l'uso di un template ($TEMPLATE), preleva il
    file del modello dalla directory dei template, esegue la sostituzione della
    stringa segnaposto {{CONTENT}} rimpiazzandola con il prompt inserito
    (proveniente da file, da stdin o da argomenti posizionali) usando awk con
    file di staging intermedi temporanei, e assegna il testo risultante a
    CONTENT. 4. Se è presente dell'input catturato dallo standard input (STDIN_CONTENT),
    lo adotta come prompt principale, allegando in coda eventuali argomenti
    posizionali. 5. Se vi sono solo argomenti CLI posizionali in $ARGS, li
    converte ed unisce nel prompt testuale principale via
    expand_args_to_content. - Input: Nessuno. - Output: Stringa compilata
    assegnata alla variabile globale $CONTENT. - Side-effects: Lettura file,
    generazione e distruzione di file di staging temporanei sul disco. -
    Dipendenze: collect_input_from_files, expand_args_to_content, cat,
    ensure_run_tmpdir, _mktemp_in_dir, awk, mv, rm, printf.

BLOCCHI DI CODICE

  - CORE_PROVIDER_PRO_LOAD_INITIALIZATION - Fase: Selezione e caricamento del
    modulo provider attivo. - Azione: Scansiona la directory dei provider
    esterni per mappare i moduli installati. Determina il provider attivo
    leggendolo dai parametri CLI o dal file canonico del provider
    PROVIDER_FILE (adottando "groq" in caso di assenza). Se configurato un nuovo provider da CLI (PROVIDER_CLI
    diverso da "list"): Convalida che appartenga a quelli supportati, salva
    transazionalmente la scelta nel file $PROVIDER_FILE con permessi 600 e, se
    il provider è cambiato, cancella il vecchio file locale dei modelli
    $MODELS_FILE (per forzarne l'allineamento API), elimina il vecchio file URL
    del provider e risolve il nuovo URL. Se richiesta la selezione interattiva
    (--provider senza argomenti o valore "list"): Visualizza a schermo il menu
    numerato di tutti i provider installati evidenziando il default, acquisisce
    la scelta dell'utente dallo standard input (accettando l'indice o il nome
    letterale e adottando il default se premuto invio), scrive in modo atomico
    la scelta in $PROVIDER_FILE con permessi 600, rimuove l'URL preesistente e
    invalida il file locale dei modelli. Infine, se il provider attivo è "groq"
    e manca il file URL locale, scrive l'URL dell'API predefinito nel file
    canonico. Invoca la funzione di caricamento sicuro load_provider_module per
    importare i metodi del modulo provider ed esegue la risoluzione dell'URL
    dell'API. Se l'interfaccia interattiva viene eseguita senza comandi
    aggiuntivi, stampa la conferma del provider impostato ed esce direttamente
    con stato 0.
  - CORE_PROVIDER_PRO_LOAD_VALIDATION_REFRESH - Fase: Validazione del modulo
    provider ed allineamento modelli. - Azione: Verifica che il modulo del
    provider sia stato correttamente importato ed esegue la convalida della
    conformità della sua interfaccia via validate_provider_interface; in caso di
    errori strutturali termina con codice BASH4LLM_ERR_API. Se è stato richiesto
    esplicitamente il refresh dei modelli ($REFRESH_MODELS attivo), richiede la
    chiave API ed avvia l'allineamento dei modelli locali tramite
    refresh_models_dispatch, terminando con stato 0. In subordine, se il file
    dei modelli locali $MODELS_FILE è assente o completamente vuoto, tenta di
    eseguire un allineamento automatico dei modelli in background (richiedendo
    la chiave API ed invocando la routine di refresh, senza interrompere lo
    script in caso di fallimento per permettere l'elaborazione dei flussi in
    background).
  - CORE_PROVIDER_SHOW - Fase: Esecuzione delle risposte diagnostiche immediate.
    - Azione: Gestisce e soddisfa le richieste diagnostiche e di ispezione dei
    percorsi dello script, terminando l'esecuzione con stato 0 dopo la
    visualizzazione dei dati: --print-config-dir / PRINT_CONFIG_DIR: Stampa il
    percorso della directory di configurazione canonica. --print-provider-file /
    PRINT_PROVIDER_FILE: Stampa il percorso del file canonico del provider.
    PRINT_MODEL_FILE: Stampa il percorso del file del modello canonico per il
    provider indicato. SHOW_CONFIG: Visualizza un riepilogo tabulare dettagliato
    dei parametri correnti dell'installazione (versione, date, provider attivo,
    modello attivo, modello di default, percorsi di tutte le directory di lavoro
    di Bash4LLM⁺, stato di debug e simulazione, conteggio dei modelli in
    whitelist). DIAGNOSTICS: Esegue un'analisi di integrità diagnostica completa
    dello stato del sistema. Scansiona e verifica l'esistenza e l'accessibilità
    fisica di tutte le directory operative, attesta che le funzioni obbligatorie
    del provider selezionato siano regolarmente mappate in memoria, verifica la
    conformità del modello attivo rispetto alla whitelist locale, controlla
    l'effettiva presenza della chiave API nell'ambiente correlandola con il nome
    esatto della variabile attesa dal provider, lo stato di sblocco della
    sessione vault (_B4L_RT_CTX), l'esito dei test crittografici e del TLS
    Handshake verso l'endpoint remoto attivo.
  - CORE_PROVIDER_MAIN_RESOLVE - Fase: Risoluzione del modello e cattura
    standard input. - Azione: Se l'avvio è per solo bootstrap ($BOOTSTRAP_ONLY),
    esce immediatamente con stato 0. Carica la configurazione locale e le
    whitelist dei modelli, garantisce la stabilità della directory temporanea
    $RUN_TMPDIR, risolve il modello LLM finale da adottare via resolve_model ed
    esegue su di esso le convalide sintattiche e di interfaccia del provider
    (terminando con codice BASH4LLM_ERR_BAD_MODEL se il modello selezionato
    fallisce le verifiche). Infine, cattura e memorizza l'eventuale flusso di
    testo in arrivo da una redirezione o pipeline nello standard input (cat -)
    all'interno di STDIN_CONTENT e, in presenza di un JSON strutturato, ne
    estrae transazionalmente la chiave API interna ed i prompt d'ingresso.
  - CORE_PROVIDER_MAIN_EXECUTION - Fase: Esecuzione principale del prompt (Core
    Execution loop). - Azione: Invoca assemble_content per generare il prompt
    definitivo. Se lo script è stato richiamato in modalità interattiva e non
    contiene prompt, visualizza la notifica del provider attivo ed esce. Esegue
    controlli di sicurezza sull'input bloccando l'esecuzione con codice
    BASH4LLM_ERR_NO_PROMPT in assenza di prompt testuale o JSON d'input (tranne
    se in chat interattiva o elaborazione batch). Valida la correttezza numerica
    dei parametri di temperatura e token massimi. Controlla che il modello sia
    regolarmente supportato o non deprecato ispezionando la whitelist e
    garantisce la presenza della directory temporanea
    RUN_TMPDIR. Procede quindi con i tre flussi di esecuzione principali: 1. Elaborazione BATCH (BATCH_FILE
    attivo): Scansiona il file batch riga per riga, escludendo righe vuote o
    commenti. Per ciascun prompt: Configura l'ambiente di thread operativo se
    richiesto (validando l'ID thread, limitando la finestra dei messaggi, e
    richiamando session_engine_build_window o la legacy thread_read_window per
    compilare il JSON storico $BUILD_MESSAGES_FILE). Se lo storico è assente o
    vuoto, lo inizializza inserendo un record utente con il prompt corrente.
    Dispaccia la costruzione del payload del file JSON via
    build_payload_from_vars. Verifica e richiede la chiave API per il provider e
    avvia l'interrogazione dell'API (eseguendo la chiamata in tempo reale se
    STREAM_MODE è attivo, oppure richiamando la routine perform_request_once). 2. Chat Interattiva (CHAT_MODE
    attivo): Delega integralmente l'esecuzione interattiva all'utility esterna
    tui-repl.sh per ottimizzare l'isolamento ed allocazione del processo TTY. 3.
    Esecuzione Standard di Singolo Prompt (Streaming o Sincrona): Se è definito
    un thread operativo ($THREAD_ID), esegue la validazione dell'ID, determina
    la finestra di recupero, e compila il file storico dei messaggi
    BUILD_MESSAGES_FILE interrogando il Session Engine o richiamando la routine legacy di lettura NDJSON. Compila il payload del file JSON via build_payload_from_vars. Se si opera in esecuzione reale (non simulata), valida la presenza della chiave API del provider. Se streaming attivo (STREAM_MODE
    == 1): Invoca la funzione di streaming dedicatale (call_api_streaming) sotto
    lock esclusivo di thread. Se completato con successo: Se vi è un thread
    attivo, sanifica ed accoda il messaggio dell'utente al registro storico (via
    session_engine_append o la legacy thread_append). Se la connessione ha
    generato risposta reale ed il file
    RESP contiene dati validi, estrae il testo generato dall'assistente, ne esegue la sanificazione dai ritorni a capo ed accoda in modo sicuro la risposta dell'assistente nel file NDJSON storico del thread (delegandola al Session Engine o alla legacy thread_append). Stampa un carattere a capo e termina l'esecuzione con stato 0. In caso di errore termina con errore di curl. Se sincrona standard (STREAM_MODE
    == 0): Stampa i dettagli diagnostici di visualizzazione del prompt e del
    modello se in terminale interattivo verboso. Invoca la chiamata
    transazionale via perform_request_once sotto lock esclusivo di thread. In
    caso di successo: Se è configurato un thread, sanifica e registra il
    messaggio dell'utente nel file storico. Se l'esecuzione è reale ed il file
    di risposta contiene dati, estrae il testo generato dall'assistente, lo
    ripulisce e registra in modo sicuro la risposta dell'assistente nel file
    NDJSON di storico. Esce con stato 0. In caso di fallimento della chiamata,
    accoda comunque il messaggio utente al thread operativo contrassegnandolo
    come interazione fallita ed interrompe l'esecuzione restituendo lo stato
    d'uscita della perform_request_once.
