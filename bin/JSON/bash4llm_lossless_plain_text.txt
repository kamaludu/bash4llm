SPECIFICA DELL'ARCHITETTURA E DELLA LOGICA DI BASH4LLM (v2.0.0)

Questo documento descrive in modo completo e senza perdita di informazioni la logica, le variabili, le funzioni e i blocchi strutturali del sistema Bash4LLM, suddivisi secondo le cinque macro-sezioni originarie.

1. PRECORE_BOOT

VARIABILI
SCRIPT_NAME: Nome identificativo del programma (impostato a "bash4llm").
SCRIPT_VERSION: Versione corrente del software (impostata a "2.0.0").
SCRIPT_DATE: Data di rilascio dello script (impostata a "2026-06-09").
BASH4LLM_ERR_NO_API_KEY: Costante numerica di errore (valore 10) associata alla mancanza di una chiave API.
BASH4LLM_ERR_BAD_MODEL: Costante numerica di errore (valore 11) associata alla richiesta di un modello non valido o non supportato.
BASH4LLM_ERR_CURL_FAILED: Costante numerica di errore (valore 12) associata al fallimento della chiamata di rete curl.
BASH4LLM_ERR_INVALID_JSON: Costante numerica di errore (valore 13) associata alla ricezione o generazione di un JSON non valido.
BASH4LLM_ERR_NO_PROMPT: Costante numerica di errore (valore 14) associata alla mancanza di un prompt di input.
BASH4LLM_ERR_TMP: Costante numerica di errore (valore 15) associata a problemi del file-system temporaneo o dei file di lock.
BASH4LLM_ERR_API: Costante numerica di errore (valore 16) associata a risposte d'errore o codici HTTP non validi restituiti dal provider API.
BASH4LLMERRNOAPIKEY: Mappatura alias diretta del valore di BASH4LLM_ERR_NO_API_KEY.
BASH4LLMERRBAD_MODEL: Mappatura alias diretta del valore di BASH4LLM_ERR_BAD_MODEL.
BASH4LLMERRCURL_FAILED: Mappatura alias diretta del valore di BASH4LLM_ERR_CURL_FAILED.
BASH4LLMERRINVALID_JSON: Mappatura alias diretta del valore di BASH4LLM_ERR_INVALID_JSON.
BASH4LLMERRNO_PROMPT: Mappatura alias diretta del valore di BASH4LLM_ERR_NO_PROMPT.
BASH4LLMERRTMP: Mappatura alias diretta del valore di BASH4LLM_ERR_TMP.
BASH4LLMERRAPI: Mappatura alias diretta del valore di BASH4LLM_ERR_API.
DEBUG: Flag di configurazione per l'output di tracciamento dello sviluppo. Eredita il valore di ambiente ${DEBUG} o viene impostato su 0 (disattivo).
BASH4LLM_LOG: Percorso assoluto o relativo del file in cui registrare i log di tracciamento. Eredita ${BASH4LLM_LOG} (vuoto se disattivato).
SCRIPTDIR: Percorso assoluto della directory contenente lo script corrente, risolto dinamicamente tramite la funzione resolve_script_dir.
CANONICAL_EXTRAS_DIR: Percorso della directory canonica delle estensioni, calcolato rimuovendo la barra finale da BASH4LLM_DIR e annettendo /extras.
LEGACY_EXTRAS_DIR: Percorso storico di fallback delle estensioni, relativo alla cartella dello script (SCRIPTDIR/extras).
BASH4LLM_EXTRAS_DIR: Directory attiva per l'accesso alle risorse aggiuntive dello script (assegnata a CANONICAL_EXTRAS_DIR).
PROVIDERS_DIR: Percorso assoluto o relativo dei moduli esterni dei provider (impostato su PROVIDERS_DIR di ambiente o, in alternativa, localizzato sotto la directory delle estensioni alla voce /providers).
BASH4LLM_CONFIG_DIR: Directory dei file di configurazione utente persistenti. Impostata a BASH4LLM_CONFIG_DIR di ambiente o posizionata sotto $BASH4LLM_DIR/config.
BASH4LLM_MODELS_DIR: Directory dei file relativi ai modelli supportati. Impostata a BASH4LLM_MODELS_DIR di ambiente o posizionata sotto $BASH4LLM_DIR/models.
BASH4LLM_TEMPLATES_DIR: Directory contenente i file dei modelli di template. Impostata a BASH4LLM_TEMPLATES_DIR di ambiente o posizionata sotto $BASH4LLM_DIR/templates.
BASH4LLM_HISTORY_DIR: Directory contenente lo storico e i log delle interazioni dello script. Impostata a BASH4LLM_HISTORY_DIR di ambiente o posizionata sotto $BASH4LLM_DIR/history.
BASH4LLM_TMPDIR: Directory radice per l'allocazione dei dati temporanei di Bash4LLM. Impostata a BASH4LLM_TMPDIR di ambiente o posizionata sotto $BASH4LLM_DIR/tmp.
MODELS_FILE: File di testo locale in cui memorizzare la lista dei modelli recuperati. Impostato su MODELS_FILE di ambiente o posizionato sotto $BASH4LLM_MODELS_DIR/models.txt.
MAX_MODELS: Limite numerico massimo di modelli importabili o gestibili localmente. Eredita ${MAX_MODELS} o adotta il valore di default di 200.
PROVIDER_FILE: Percorso del file che persiste l'ultimo provider configurato dall'utente, determinato tramite canonical_provider_file.
SESSION_DIR: Directory dedicata alla registrazione dei log storici delle chat multi-turno (posizionata in $BASH4LLM_HISTORY_DIR/sessions).
MODELS_LOCK: Percorso del file di lock di sincronizzazione per l'accesso ai modelli, impostato su MODELS_LOCK di ambiente o localizzato sotto $BASH4LLM_MODELS_DIR/models.lock.
HISTORY_LOCK: Percorso del file di lock per l'accesso concorrente ai log di storico e sessione, impostato su HISTORY_LOCK di ambiente o localizzato sotto $BASH4LLM_HISTORY_DIR/history.lock.
TMP_LOCK: Percorso del file di lock esclusivo per la gestione sicura della directory temporanea dello script, impostato su TMP_LOCK di ambiente o localizzato sotto $BASH4LLM_TMPDIR/tmp.lock.
BASH4LLM_LOCK_TIMEOUT_TMP: Timeout massimo di attesa espresso in secondi per acquisire il lock temporaneo (eredita BASH4LLM_LOCK_TIMEOUT_TMP o imposta 10).
BASH4LLM_LOCK_TIMEOUT_MODELS: Timeout in secondi per acquisire il lock sui modelli (eredita BASH4LLM_LOCK_TIMEOUT_MODELS o imposta 10).
BASH4LLM_LOCK_TIMEOUT_HISTORY: Timeout in secondi per acquisire il lock sullo storico dei messaggi e delle sessioni (eredita BASH4LLM_LOCK_TIMEOUT_HISTORY o imposta 10).

FUNZIONI
resolve_script_dir - Scopo: Identifica la cartella reale in cui risiede lo script in esecuzione, tracciando in modo sicuro i collegamenti simbolici sul file-system. - Input: Nessuno (analizza la variabile speciale $0 del processo). - Output: Restituisce su standard output il percorso assoluto normalizzato della directory. - Side-effects: Nessuno. - Dipendenze: Comandi readlink, dirname, cd, pwd, printf.
canonical_config_dir - Scopo: Normalizza il percorso della directory di configurazione rimuovendo eventuali barre finali ridondanti. - Input: Nessuno (analizza $BASH4LLM_CONFIG_DIR). - Output: Restituisce il percorso canonico su standard output. - Side-effects: Nessuno. - Dipendenze: printf.
canonical_provider_file - Scopo: Genera il percorso canonico del file di persistenza del provider selezionato. - Input: Nessuno. - Output: Restituisce su standard output il percorso del file /provider sotto la directory canonica di configurazione. - Side-effects: Richiama internamente canonical_config_dir. - Dipendenze: printf, canonical_config_dir.
canonical_model_file - Scopo: Genera il percorso del file in cui memorizzare il modello predefinito associato a un determinato provider. - Input: $1 (nome identificativo del provider). - Output: Restituisce su standard output il percorso del file formato /model.<provider> posizionato nella directory canonica di configurazione. - Side-effects: Richiama canonical_config_dir. - Dipendenze: printf, canonical_config_dir.
canonical_provider_url_file - Scopo: Genera il percorso canonico del file contenente l'endpoint o URL API specifico del provider attivo. - Input: Nessuno. - Output: Restituisce su standard output il percorso del file /provider-url posizionato nella directory canonica di configurazione. - Side-effects: Richiama o simula la logica di canonical_config_dir. - Dipendenze: type, canonical_config_dir, printf.
ensure_api_key_for_provider - Scopo: Convalida la presenza di una chiave API valida in ambiente per il provider selezionato. Se non disponibile in una sessione non interattiva (es. script o cron), genera un errore critico. In sessioni interattive (TTY), richiede l'inserimento della chiave all'utente dallo standard input, ripulisce la stringa rimuovendo comandi o intestazioni ridondanti (como stringhe di assegnamento export o spazi superflui) e la esporta nell'ambiente. Visualizza a schermo istruzioni di salvataggio permanente nel profilo della shell dell'utente per le esecuzioni future. - Input: $1 (nome identificativo del provider). - Output: Ritorna stato 0 se la chiave è presente o acquisita correttamente, altrimenti restituisce il codice di errore BASH4LLMERRNOAPIKEY. Esporta la chiave API nella variabile d'ambiente corretta (es. GROQ_API_KEY o variabile personalizzata indiretta). - Side-effects: Esporta variabili d'ambiente globali; effettua letture dallo standard input e scritture su stderr. - Dipendenze: provider_api_env_var_name, printf, tr, sed, grep, export.
enforce_network_policy - Scopo: Valuta se lo stato corrente del runtime permette l'esecuzione di richieste di rete reali o se queste devono essere bloccate. - Input: Nessuno (analizza i flag $DRY_RUN, $BASH4LLM_SKIP_NETWORK, $BASH4LLM_ENFORCE_NO_NETWORK_IF_QUIET, $QUIET). - Output: Ritorna lo stato 1 se le connessioni esterne sono bloccate (o se si è in simulazione dry-run), 0 se le chiamate di rete sono autorizzate. - Side-effects: Scrittura di log di debug se attiva la modalità di tracciamento. - Dipendenze: is_truthy, log_info.
log_prefix - Scopo: Restituisce il prefisso di intestazione unificato per i log dello script. - Input: Nessuno. - Output: Restituisce su standard output la stringa formattata bash4llm: <SCRIPT_NAME>: . - Dipendenze: printf.
log_info - Scopo: Traccia e visualizza i messaggi informativi. - Input: $1 (codice categoria/modulo, default "INFO"), $2 (corpo del messaggio). - Output: Scrive il messaggio formattato su stderr se $DEBUG è attivo. Se $BASH4LLM_LOG è valorizzato, appende il record con timestamp UTC e categoria nel file di destinazione. - Side-effects: Scrittura su console e file-system. - Dipendenze: log_prefix, date, printf.
log_warn - Scopo: Traccia e visualizza a schermo i messaggi di avvertimento non critici. - Input: $1 (codice modulo, default "WARN"), $2 (corpo del messaggio). - Output: Scrive il messaggio su stderr. Se abilitato il log, appende il tracciamento nel file $BASH4LLM_LOG. - Side-effects: Scrittura su stderr e file-system. - Dipendenze: log_prefix, date, printf.
log_error - Scopo: Registra ed evidenzia gli errori critici. - Input: $1 (codice modulo, default "ERROR"), $2 (corpo del messaggio). - Output: Scrive l'errore su stderr. Se abilitato il log, appende il tracciamento nel file $BASH4LLM_LOG. - Side-effects: Scrittura su stderr e file-system. - Dipendenze: log_prefix, date, printf.
dbg - Scopo: Visualizza messaggi di diagnostica rapida su stderr se il debug è attivo. - Input: Argomenti testuali liberi da stampare ($*). - Output: Righe di testo inviate a stderr. - Dipendenze: printf.
ensure_config_dir - Scopo: Assicura la presenza e la sicurezza della cartella di configurazione utente. Crea la cartella se mancante, imposta i permessi restrittivi 700 e convalida l'effettiva scrivibilità scrivendo e rimuovendo immediatamente un file di test. - Input: Nessuno. - Output: Ritorna stato 0 se la directory è scrivibile e pronta, 1 in caso di errore. - Side-effects: Creazione directory, test di scrittura sul disco, alterazione permessi. - Dipendenze: mkdir, chmod, rm, log_error.
write_provider_url_if_missing - Scopo: Memorizza in modo permanente l'URL delle API per un determinato provider all'interno del file dedicato, a patto che il file non sia già popolato. Per evitare scritture corrotte, la scrittura avviene in modo transazionale su un file temporaneo, poi spostato o copiato nella destinazione finale ed impostato con permessi 600. - Input: $1 (nome identificativo del provider), $2 (stringa dell'URL). - Output: Ritorna stato 0 in caso di successo, 1 se falliscono le operazioni su file-system. - Side-effects: Creazione directory, file temporanei, scrittura atomica, modifica permessi del file. - Dipendenze: canonical_provider_url_file, dirname, mkdir, mktemp, printf, mv, cp, rm, chmod.
resolve_provider_url - Scopo: Identifica ed esporta l'URL di connessione per le API. Controlla prioritariamente se l'utente ha impostato variabili d'ambiente esplicite (BASH4LLM_API_URL o BASH4LLM_PROVIDER_URL). In assenza di esse, legge la prima riga del file canonico dell'URL del provider. Se il file non esiste e il provider attivo è "groq", esporta l'endpoint nativo di default di Groq. - Input: $1 (nome provider, default $PROVIDER). - Output: Ritorna stato 0 se l'URL viene risolto ed esportato nella variabile globale $BASH4LLM_PROVIDER_URL, 1 in caso di fallimento. - Side-effects: Esporta la variabile d'ambiente $BASH4LLM_PROVIDER_URL. - Dipendenze: canonical_provider_url_file, sed, awk, export.
provider_api_env_var_name - Scopo: Calcola il nome standardizzato della variabile d'ambiente deputata a contenere la chiave API di un determinato provider (es: "gemini" -> "GEMINI_API_KEY"). - Input: $1 (nome del provider). - Output: Restituisce su standard output il nome normalizzato (tutto in maiuscolo, convertendo i caratteri speciali non alfanumerici in trattini bassi _ e annettendo il suffisso _API_KEY). - Dipendenze: printf, tr.
is_valid_json_string - Scopo: Valida se la stringa in input contiene una struttura sintatticamente corretta in formato JSON. - Input: $1 (stringa da analizzare). - Output: Ritorna stato 0 se il JSON è valido, 1 altrimenti. - Dipendenze: printf, jq.
b64encode - Scopo: Esegue la codifica di flussi in formato Base64 leggendo da standard input ed unificando l'output su un'unica riga priva di caratteri di a capo. - Input: Flusso dati in ingresso su stdin. - Output: Stringa codificata inviata a stdout. Ritorna stato 0 o 1 in caso di errore. - Dipendenze: base64, tr.
b64decode - Scopo: Esegue la decodifica in formato Base64 di flussi di dati letti da standard input, avvalendosi dei flag ottimali rilevati all'avvio. - Input: Stringa codificata Base64 da stdin. - Output: Flusso decodificato su stdout. - Dipendenze: base64 (configurato con $B64_DECODE_OPT).
is_truthy - Scopo: Valuta se il parametro testuale in ingresso rappresenta un valore di veridicità affermativo logico. - Input: $1 (stringa da valutare). - Output: Ritorna stato 0 (vero) se il parametro corrisponde a 1, true, TRUE, True, yes, YES o Yes. In tutti gli altri casi ritorna 1 (falso).
file_size - Scopo: Restituisce la dimensione esatta in byte di un file in modo compatibile sia con sistemi Linux (che usano stat -c %s) sia macOS/BSD (che usano stat -f %z). - Input: $1 (percorso del file). - Output: Restituisce su standard output il valore numerico dei byte (o 0 se il file è mancante o vuoto). - Dipendenze: uname, stat, printf.
is_valid_json_file - Scopo: Valida se il file indicato esiste, contiene dati ed è strutturato come JSON conforme. - Input: $1 (percorso del file). - Output: Ritorna stato 0 se valido, 1 altrimenti. - Dipendenze: jq.
stage_b64 - Scopo: Acquisisce un payload testuale (da file o standard input), verifica che la sua dimensione non ecceda la soglia massima consentita ($MAX_STAGE_BYTES, default 10MB), ne esegue in sicurezza la codifica Base64 scrivendola in un file temporaneo protetto, e infine sposta in modo atomico (tramite lock della directory) il file temporaneo nel percorso finale di staging con permessi 600. - Input: $1 (opzionale: percorso del file payload di origine; se omesso legge da stdin), $2 (percorso del file di destinazione finale codificato in Base64). Se viene passato un singolo argomento, questo viene trattato come destinazione e l'input viene letto da stdin. - Output: Ritorna stato 0 se il processo transazionale si conclude correttamente, 1 in caso di fallimento o di superamento dei limiti di dimensione. - Side-effects: Scrittura e cancellazione di file temporanei intermedi sul disco, modifica permessi, acquisizione concorrente di lock. - Dipendenze: dirname, mkdir, _tmpf, cat, rm, file_size, base64, tr, chmod, lock_exec, mv, log_error, log_info.
lock_exec - Scopo: Esegue un comando arbitrario garantendo mutua esclusione e protezione concorrente su file-system tramite lock gestito con flock. Se non riesce ad acquisire il lock entro il limite di tempo impostato, fallisce in modo controllato senza eseguire il comando. - Input: $1 (percorso del file di lock), $2 (valore numerico del timeout in secondi, default 10), seguiti dal delimitatore -- e dal comando con i relativi argomenti da avviare. - Output: Restituisce lo stato di uscita del comando eseguito, 124 in caso di scadenza del timeout senza acquisizione del lock, o 2 in caso di problemi di sintassi o directory. - Side-effects: Creazione directory del lock, blocco concorrente del file-system. - Dipendenze: dirname, mkdir, command, flock, printf, log_error.
_mktemp_in_dir - Scopo: Crea in sicurezza un file temporaneo univoco all'interno della cartella specificata. - Input: $1 (cartella radice), $2 (opzionale: prefisso del nome, default "groq"). - Output: Restituisce su standard output il percorso completo del file temporaneo creato, altrimenti restituisce il codice BASH4LLMERRTMP se fallisce. - Dipendenze: _tmpf, log_error.
show_payload_head - Scopo: Estrae e visualizza a schermo le prime righe del file di payload inviato alle API a scopo diagnostico di debug. Se il file di origine risulta codificato in Base64 (estensione .b64), esegue la decodifica istantanea prima dell'estrazione delle righe. - Input: $1 (percorso file payload, default $PAYLOAD), $2 (numero massimo di righe, default 200). - Output: Stampa le righe decodificate su stderr. - Side-effects: Lettura, eventuale decodifica di flussi. - Dipendenze: printf, grep, b64decode, head.
atomic_write - Scopo: Scrive o sovrascrive un file in modo atomico, transazionale e sicuro. Riceve un flusso di dati da stdin, lo scrive in un file temporaneo posizionato all'interno della stessa directory di destinazione, imposta i permessi di sicurezza a 600, acquisisce un lock esclusivo di protezione concorrente (se lock_exec è disponibile) e sposta in modo definitivo il file temporaneo sovrascrivendo l'originale. Questa sequenza impedisce letture di file parziali o corrotti in caso di interruzioni impreviste o accessi paralleli. - Input: $1 (percorso del file di destinazione definitivo), $2 (timeout del lock, default 10). - Output: Ritorna stato 0 se completato, altrimenti restituisce il relativo codice di errore di fallimento o di lock. - Side-effects: Creazione directory genitrice se mancante, allocazione e rimozione di file temporanei intermedi, lock, sovrascrittura, alterazione permessi. - Dipendenze: dirname, mkdir, mktemp, cat, rm, chmod, lock_exec, mv, log_error.
extract_text_from_resp - Scopo: Estrae in modo flessibile e resiliente il testo di risposta principale dai JSON restituiti dalle API del provider LLM. Esegue una copia isolata del file di risposta, verifica che il file sia un JSON strutturato valido e non sia marcato come file diagnostico. Tenta l'estrazione scansionando diverse varianti di percorsi e formati standardizzati: strutture chat completions OpenAI (delta o messaggi in .choices[].message.content o .choices[].delta.content), formati di completamento testuale piatti (.choices[].text), formati personalizzati (.output_text o .data[].text) o, in ultima istanza, estrae e concatena ricorsivamente tutti i valori scalari di tipo stringa presenti nel JSON. - Input: Nessuno (analizza il file puntato da $RESP). - Output: Restituisce su standard output il testo decodificato se individuato (ritornando stato 0). Ritorna 1 se non viene trovato testo, 2 se il file analizzato contiene dati diagnostici o è corrotto. - Side-effects: Allocazione e rimozione di file di copia temporanei sul disco per isolamento. - Dipendenze: _tmpf, cp, is_valid_json_file, jq, log_warn, log_error, rm.
ensure_run_tmpdir - Scopo: Prepara, isola ed esporta l'ambiente operativo temporaneo ($RUN_TMPDIR) dedicato in modo esclusivo alla singola istanza o sotto-shell dello script in esecuzione, posizionandolo all'interno della directory temporanea principale sicura $BASH4LLM_TMPDIR. Inizializza i file di payload, risposta e log di errore associati a questa istanza ($PAYLOAD, $RESP, $ERRF) forzando permessi di massima sicurezza (700 per la directory, 600 per i file). Registra una trap di uscita (EXIT, INT, TERM) incaricata della pulizia ricorsiva automatica dei dati temporanei al termine dello script (salvo nei casi in cui sia attivo il flag di debug e conservazione dei file $DEBUG_PRESERVE). - Input: $1 (opzionale: --print per stampare esclusivamente il percorso su stdout ed uscire). - Output: Ritorna stato 0 ed esporta le variabili $RUN_TMPDIR, $PAYLOAD, $RESP e $ERRF. Ritorna il codice BASH4LLMERRTMP se fallisce. - Side-effects: Creazione fisica di cartelle e file di testo vuoti, modifica permessi di accesso, impostazione di trap di sistema. - Dipendenze: mkdir, chmod, _tmpf, rm, trap, log_warn, log_error, printf.
cleanup_tmp - Scopo: Rimuove in modo ricorsivo la directory provvisoria associata all'istanza $RUN_TMPDIR, a patto che risieda all'interno della directory temporanea autorizzata dello script. - Input: Nessuno. - Output: Rimozione dei file dal disco. - Dipendenze: rm.
b64_atomic_write - Scopo: Esegue la codifica Base64 del flusso di dati letto da standard input e lo scrive nel file di destinazione in modo transazionale e atomico protetto da lock, applicandovi permessi 600. - Input: $1 (file di destinazione), $2 (timeout del lock, default 10). - Output: Ritorna stato 0 o il codice di errore BASH4LLMERRTMP. - Side-effects: Creazione directory, allocazione file temporaneo, scrittura e spostamento atomico. - Dipendenze: dirname, mkdir, _tmpf, b64encode, chmod, lock_exec, mv, rm, log_error.
b64_atomic_read - Scopo: Legge un file cifrato o codificato in Base64 in modo sicuro, ne esegue la decodifica al volo e ne restituisce il contenuto decodificato. - Input: $1 (percorso del file codificato Base64). - Output: Restituisce il contenuto decodificato su standard output. - Dipendenze: b64decode.
ui_state_write - Scopo: Scrive in modo atomico lo stato corrente dell'interfaccia utente (frammenti di informazioni e metadati di runtime) in un file JSON dedicato all'interno della cartella degli stati dell'interfaccia (posizionata sotto $BASH4LLM_CONFIG_DIR/ui_state o sotto $RUN_TMPDIR), blindando i permessi a 600. - Input: $1 (nome del file di stato), $2 (stringa del contenuto JSON). - Output: File di stato creato o aggiornato. Ritorna stato 0 se completato, 1 se si verificano errori. - Side-effects: Creazione della directory degli stati se non presente, scrittura atomica con lock concorrente. - Dipendenze: mkdir, chmod, atomic_write, wc, log_info, log_warn.
load_provider_module - Scopo: Carica ed importa in modo sicuro e isolato i moduli dei provider esterni (plugin). Esegue rigidi controlli di sicurezza: convalida che la directory dei moduli e il file non siano scrivibili da terzi (group/world writable), verifica che il file appartenga all'utente corrente, esegue un dry-run di analisi sintattica (bash -n) e calcola la firma crittografica del file prima e dopo l'inclusione per prevenire attacchi di tipo "Race Condition" o "File Tampering". Per isolare l'importazione, esegue il caricamento provvisorio del modulo all'interno di una sotto-shell, cattura le funzioni esportate e verifica la conformità dell'interfaccia accertandosi che le due routine obbligatorie richieste (buildpayload_<provider> e call_api_<provider>) siano effettivamente dichiarate; in caso positivo, importa le funzioni nell'ambiente di esecuzione principale dello script, memorizzandone le funzionalità nel file di stato provider_capabilities.json. - Input: $1 (nome identificativo del provider). - Output: Ritorna stato 0 se il caricamento ed i controlli di sicurezza si concludono correttamente (o se si tratta del provider predefinito integrato "groq", che non richiede file esterni). Ritorna 1 in caso di violazioni di sicurezza, sintassi corrotta, o interfacce incomplete. - Side-effects: Scrittura log di errore/sicurezza, generazione file temporanei di dump, source di codice esterno via ., disabilitazione e ripristino temporaneo dell'opzione shell nounset (set -u), scrittura dello stato UI delle capacità. - Dipendenze: mkdir, _is_world_writable, _get_owner, _get_perm_string, getfile_signature, bash, _mktemp_in_dir, awk, sort, comm, grep, type, declare, rm, ui_state_write, jq.
_detect_base64_opts - Scopo: Rileva a runtime le opzioni compatibili con l'eseguibile base64 locale per la formattazione a riga singola (senza a capo) e per la decodifica, differenziando in modo sicuro tra distribuzioni GNU/Linux e macOS/BSD. Esporta i flag rilevati in variabili globali. - Input: Nessuno. - Output: Esporta le variabili $B64_WRAP_OPT (es. -w0 o vuoto) e $B64_DECODE_OPT (es. -d o -D). - Dipendenze: printf, base64, grep.

VARIABILI
BASH4LLM_ROTATE_HISTORY: Flag di configurazione (valore 1 o 0, default 0) per attivare la rotazione automatica periodica dello storico delle interazioni.
BASH4LLM_HISTORY_MAX_FILES: Soglia numerica massima di file memorizzabili nello storico delle risposte (default 100). Superata la soglia, i file più vecchi vengono rimossi.
BASH4LLM_HISTORY_MAX_BYTES: Soglia massima espressa in byte del peso cumulativo di tutti i file storici archiviati (default 104857600, equivalente a 100MB).
BASH4LLM_HISTORY_KEEP_DAYS: Limite di tempo espresso in giorni per la conservazione dei log e dello storico (default 90). I file più vecchi della soglia vengono eliminati.
SESSION_CACHE_DIR: Directory destinata a contenere la cache temporanea dei dati e dei messaggi delle sessioni attive (localizzata sotto la directory di configurazione utente).
CONTENT: Prompt testuale primario dell'utente (assegnato a CONTENT).
JSON_INPUT: Input strutturato in formato JSON fornito dall'utente (sovrascrive o definisce i messaggi).
SESSION_ID: Identificatore alfanumerico della sessione di chat corrente (utilizzato per raggruppare e persistere il contesto dei messaggi in interazioni multi-turno).
SESSION_WINDOW: Numero massimo di messaggi storici da estrarre e accodare dal log di sessione per comporre la finestra di contesto della chat.
TEMPLATE: Nome identificativo del file di template da applicare per formattare o arricchire il prompt dell'utente prima dell'invio.
BATCH_FILE: Percorso di un file contenente una serie ordinata di prompt (uno per riga) da elaborare in sequenza.
CHAT_MODE: Flag booleano (valore 1 o 0, default 0) che abilita l'interfaccia interattiva di chat multi-turno da terminale.
SET_DEFAULT_MODEL: Nome del modello da memorizzare e configurare come scelta predefinita persistente per il provider attivo.
REFRESH_MODELS: Flag booleano (valore 1 o 0, default 0) per forzare l'allineamento e l'interrogazione delle API esterne al fine di aggiornare la lista locale dei modelli supportati.
LIST_MODELS: Flag booleano (valore 1 o 0, default 0) per visualizzare a schermo l'elenco dei modelli disponibili.
FORCE_SAVE_MODE: Regola il salvataggio dell'output testuale generato sul disco (save forza il salvataggio, nosave lo disabilita, 0 adotta il comportamento standard basato sulla soglia).
OUT_PATH: Percorso del file o della directory personalizzata in cui archiviare i risultati restituiti dalle API.
SYSTEM_PROMPT: Prompt di sistema personalizzato utilizzato per istruire o condizionare il comportamento e il tono del modello LLM.
TURE (Temperatura): Valore numerico decimale che regola la temperatura di campionamento delle risposte delle API (eredita ${TEMPERATURE}, ${TURE} o adotta 1.0). Sincronizzato bidirezionalmente con TEMPERATURE.
MAX_TOKENS: Limite di token generabili nella risposta (default 4096).
MODEL: Nome identificativo del modello LLM richiesto per l'elaborazione.
AUTO_POLICY: Politica adottata per la selezione automatica del modello di fallback in assenza di un modello espresso esplicitamente dall'utente (default "preferred").
QUIET: Flag booleano (valore 1 o 0, default 0) per inibire i messaggi informativi e limitare l'output ai soli risultati o errori bloccanti.
DRY_RUN: Flag booleano (valore 1 o 0, default 0) per simulare l'esecuzione ed escludere l'invio reale di chiamate di rete esterne.
STREAM_MODE: Flag booleano (valore 1 o 0, default 0) per attivare la ricezione e la visualizzazione a schermo dei token in tempo reale (streaming).
OUTPUT_MODE: Configura lo stile di visualizzazione dei risultati restituiti dallo script (text, raw, json, pretty).
THRESHOLD: Soglia limite espressa in caratteri al di sopra della quale i risultati delle API vengono salvati nello storico (default 1000).
MAX_RETRIES: Limite di tentativi di connessione da effettuare in caso di errori di rete temporanei (default 3).
SUPPORTED_PROVIDERS: Elenco testuale dei provider supportati (default "groq gemini huggingface").
PROVIDER: Identificatore del provider API attivo (default "groq").

load_local_config - Scopo: Carica in memoria i parametri di configurazione definiti dall'utente nel file di configurazione locale (config all'interno della cartella $BASH4LLM_CONFIG_DIR), interpretando le righe chiave/valore e impostando le variabili globali corrispondenti (MODEL, TEMPERATURE/TURE, MAX_TOKENS, OUTPUT_MODE/FORMAT, THRESHOLD). Ignora commenti e righe vuote. - Input: Nessuno. - Output: Assegnazione di variabili globali di ambiente.
load_whitelist - Scopo: Carica in memoria e normalizza l'elenco di tutti i modelli autorizzati o presenti nel file dei modelli locali per comporre la stringa globale di whitelist $ALLOWED_MODELS usata per i filtri di convalida dei modelli. - Input: Nessuno. - Output: Popola la variabile globale $ALLOWED_MODELS con i nomi normalizzati separati da a capo. - Dipendenze: awk.
is_tty_out - Scopo: Verifica se lo standard output (stdout) dello script è collegato a un terminale interattivo (TTY) reale anziché essere ridiretto a un file o pipeline. - Input: Nessuno. - Output: Ritorna stato 0 se TTY, 1 in caso contrario.

BLOCCHI DI CODICE
parse_cli_arguments - Fase: Parsing iniziale dei parametri a riga di comando. - Azione: Analizza in modo iterativo tutti gli argomenti passati da CLI. Registra e configura le opzioni booleane e i parametri complessi (file di input -f, JSON d'ingresso, file batch, sessioni, scadenze, modelli, formati di layout, log di tracciamento e parametri dei provider). Raggruppa i parametri posizionali o i prompt liberi all'interno dell'array $ARGS. Gestisce la terminazione immediata dello script in caso di parametri di aiuto (-h/--help) o visualizzazione versione (--version).
source_session_engine - Fase: Integrazione ed attivazione dei moduli estesi. - Azione: Rileva e tenta di importare lo script del Session Engine esterno (session-engine.sh). Esegue un dry-run di analisi sintattica (bash -n) e verifica la presenza in memoria delle funzioni chiave di interfaccia. Se i controlli hanno successo e l'engine non è disabilitato da configurazione utente, attiva il flag $SE_AVAILABLE=1 (abilitando la gestione estesa della cronologia della chat), altrimenti imposta il fallback automatico sui metodi storici legacy integrati.
verify_api_calls_and_rebuild_providers - Fase: Convalida delle chiamate esterne e mappatura dei plug-in. - Azione: Blocca l'esecuzione dello script se è configurata la chiave API ma le chiamate esterne sono inibite senza essere in modalità di simulazione dry-run. Esegue una scansione dei file all'interno della directory dei provider (providers/*.sh) per raccogliere dinamicamente tutti i moduli dei provider installati e comporre la stringa globale aggiornata $SUPPORTED_PROVIDERS.
raw_listings - Fase: Elaborazione delle interrogazioni rapide. - Azione: Intercetta le richieste di visualizzazione cruda dei provider (--list-providers-raw) o dei modelli locali (--list-models-raw), le stampa riga per riga su stdout ed esce immediatamente con stato 0 (o codici di errore in caso di elenchi vuoti o assenti) senza procedere oltre con l'esecuzione dello script.
immediate_actions - Fase: Esecuzione comandi immediati ed installazione estensioni. - Azione: Elabora ed esegue le azioni che non richiedono l'avvio delle chiamate API dei modelli, terminando lo script con stato 0: --list-providers: Visualizza a schermo l'elenco formattato dei provider installati ed esce. --list-models: Visualizza l'elenco strutturato dei modelli supportati ed esce. --set-default: Imposta in modo persistente il modello predefinito per il provider attivo scrivendolo in modo atomico nel file dedicato ed esce. --install-extras: Installa le estensioni sul sistema. Risolve i percorsi sorgente/destinazione, impedisce l'installazione in presenza di directory legacy non conformi, convalida l'owner e l'integrità dei file per mitigare attacchi di sicurezza e copia i file (usando atomic_write se disponibile) suddivisi per directory di destinazione (providers, librerie, documentazione, script di sicurezza, test). Infine valida i file dei provider copiatisi e termina con stato 0. Blocca l'avvio se la modalità streaming è accoppiata con output strutturato JSON.
normalize_boolean_flags - Fase: Normalizzazione e allineamento logico dei flag booleani. - Azione: Scorre una lista predefinita di variabili di controllo booleane dello script, converte i loro valori in stati numerici puri 0 o 1 tramite is_truthy, uniformandone l'elaborazione interna.

5. CORE_PROVIDER

VARIABILI
_supported_providers_arr: Array interno per l'indicizzazione dei provider rilevati.
SUPPORTED_PROVIDERS: Elenco testuale unificato dei provider supportati separati da caratteri di spazio.

FUNZIONI
validate_provider_interface - Scopo: Convalida che il modulo del provider caricato implementi correttamente le specifiche dell'interfaccia di Bash4LLM. Verifica la presenza fisica in memoria delle funzioni obbligatorie (buildpayload_<provider> e call_api_<provider>), e traccia a scopo informativo la presenza o assenza di metodi opzionali (streaming, validazione modelli, auto-selezione, refresh modelli). - Input: $1 (nome identificativo del provider da analizzare). - Output: Ritorna stato 0 se l'interfaccia obbligatoria è pienamente implementata, 1 in caso contrario (emettendo errori critici loggati). - Dipendenze: type, log_error, log_info.
assemble_content - Scopo: Compila ed organizza la stringa del prompt di input principale $CONTENT che verrà inviata al modello LLM. Segue una catena di regole di priorità esclusive: 1. Se è configurato un JSON utente in $JSON_INPUT, pulisce la variabile $CONTENT ed esce (in quanto il JSON di input definisce autonomamente i messaggi). 2. Se l'utente ha inserito dei file via parametro -f ($FILE_INPUTS), ne raccoglie ed unifica i contenuti via collect_input_from_files. Se sono presenti argomenti posizionali aggiuntivi, li annette in coda come frammenti testuali. 3. Se è richiesto l'uso di un template ($TEMPLATE), preleva il file del modello dalla directory dei template, esegue la sostituzione della stringa segnaposto {{CONTENT}} rimpiazzandola con il prompt inserito (proveniente da file, da stdin o da argomenti posizionali) usando awk con file di staging intermedi temporanei, e assegna il testo risultante a $CONTENT. 4. Se è presente dell'input catturato dallo standard input ($STDIN_CONTENT), lo adotta come prompt principale, allegando in coda eventuali argomenti posizionali. 5. Se vi sono solo argomenti CLI posizionali in $ARGS, li converte ed unisce nel prompt testuale principale via expand_args_to_content. - Input: Nessuno. - Output: Stringa compilata assegnata alla variabile globale $CONTENT. - Side-effects: Lettura file, generazione e distruzione di file di staging temporanei sul disco. - Dipendenze: collect_input_from_files, expand_args_to_content, cat, ensure_run_tmpdir, _mktemp_in_dir, awk, mv, rm, printf.

BLOCCHI DI CODICE
CORE_PROVIDER_PRO_LOAD_INITIALIZATION - Fase: Selezione e caricamento del modulo provider attivo. - Azione: Scansiona la directory dei provider esterni per mappare i moduli installati. Determina il provider attivo leggendolo dai parametri CLI o dal file di configurazione del provider $PROVIDER_FILE (adottando "groq" in caso di assenza). Se configurato un nuovo provider da CLI ($PROVIDER_CLI diverso da "list"): convalida che appartenga a quelli supportati, salva transazionalmente la scelta nel file $PROVIDER_FILE con permessi 600 e, se il provider è cambiato, cancella il vecchio file locale dei modelli $MODELS_FILE (per forzarne l'allineamento API), elimina il vecchio file URL del provider e risolve il nuovo URL. Se richiesta la selezione interattiva (--provider senza argomenti o valore "list"): visualizza a schermo il menu numerato di tutti i provider installati evidenziando il default, acquisisce la scelta dell'utente dallo standard input (accettando l'indice o il nome letterale e adottando il default se premuto invio), scrive in modo atomico la scelta in $PROVIDER_FILE con permessi 600, rimuove l'URL preesistente e invalida il file locale dei modelli. Infine, se il provider attivo è "groq" e manca il file URL locale, scrive l'URL dell'API predefinito nel file canonico. Invoca la funzione di caricamento sicuro load_provider_module per importare i metodi del modulo provider ed esegue la risoluzione dell'URL dell'API. Se l'interfaccia interattiva viene eseguita senza comandi aggiuntivi, stampa la conferma del provider impostato ed esce direttamente con stato 0.
CORE_PROVIDER_PRO_LOAD_VALIDATION_REFRESH - Fase: Validazione del modulo provider ed allineamento modelli. - Azione: Verifica che il modulo del provider sia stato correttamente importato ed esegue la convalida della conformità della sua interfaccia via validate_provider_interface; in caso di errori strutturali termina con codice BASH4LLMERRAPI. Se è stato richiesto esplicitamente il refresh dei modelli ($REFRESH_MODELS attivo), richiede la chiave API ed avvia l'allineamento dei modelli locali tramite refresh_models_dispatch, terminando con stato 0. In subordine, se il file dei modelli locali $MODELS_FILE is assente o completamente vuoto, tenta di eseguire un allineamento automatico dei modelli in background (richiedendo la chiave API ed invocando la routine di refresh, senza interrompere lo script in caso di fallimento per permettere l'elaborazione dei flussi in background).
CORE_PROVIDER_SHOW - Fase: Esecuzione delle risposte diagnostiche immediate. - Azione: Gestisce e soddisfa le richieste diagnostiche e di ispezione dei percorsi dello script, terminando l'esecuzione con stato 0 dopo la visualizzazione dei dati: --print-config-dir / PRINT_CONFIG_DIR: Stampa il percorso della directory di configurazione canonica. --print-provider-file / PRINT_PROVIDER_FILE: Stampa il percorso del file canonico del provider. PRINT_MODEL_FILE: Stampa il percorso del file del modello canonico per il provider indicato. SHOW_CONFIG: Visualizza un riepilogo tabulare dettagliato dei parametri correnti dell'installazione (versione, date, provider attivo, modello attivo, modello di default, percorsi di tutte le directory di lavoro di Bash4LLM, stato di debug e simulazione, conteggio dei modelli in whitelist). DIAGNOSTICS: Esegue un'analisi di integrità diagnostica completa dello stato del sistema. Scansiona e verifica l'esistenza e l'accessibilità fisica di tutte le directory operative, attesta che le funzioni obbligatorie del provider selezionato siano regolarmente mappate in memoria, verifica la conformità del modello attivo rispetto alla whitelist locale e controlla l'effettiva presenza della chiave API nell'ambiente correlandola con il nome esatto della variabile attesa dal provider.
CORE_PROVIDER_MAIN_RESOLVE - Fase: Risoluzione del modello e cattura standard input. - Azione: Se l'avvio è per solo bootstrap ($BOOTSTRAP_ONLY), esce immediatamente con stato 0. Carica la configurazione locale e le whitelist dei modelli, garantisce la stabilità della directory temporanea $RUN_TMPDIR, risolve il modello LLM finale da adottare via resolve_model ed esegue su di esso le convalide sintattiche e di interfaccia del provider (terminando con codice BASH4LLM_ERR_BAD_MODEL se il modello selezionato fallisce le verifiche). Infine, cattura e memorizza l'eventuale flusso di testo in arrivo da una redirezione o pipeline nello standard input (cat -) all'interno di $STDIN_CONTENT.
CORE_PROVIDER_MAIN_EXECUTION - Fase: Esecuzione principale del prompt (Core Execution loop). - Azione: Invoca assemble_content per generare il prompt definitivo. Se lo script è stato richiamato in modalità interattiva e non contiene prompt, visualizza la notifica del provider attivo ed esce. Esegue controlli di sicurezza sull'input bloccando l'esecuzione con codice BASH4LLMERRNO_PROMPT in assenza di prompt testuale o JSON d'input (tranne se in chat interattiva o elaborazione batch). Valida la correttezza numerica dei parametri di temperatura e token massimi. Controlla che il modello sia regolarmente supportato o non deprecato ispezionando la whitelist e garantisce la presenza della directory temporanea $RUN_TMPDIR. Procede quindi con i tre flussi di esecuzione principali: 1. Elaborazione BATCH ($BATCH_FILE attivo): Scansiona il file batch riga per riga, escludendo righe vuote o commenti. Per ciascun prompt: Configura l'ambiente di sessione se richiesto (validando l'ID sessione, limitando la finestra dei messaggi, e richiamando session_engine_build_window o la legacy session_read_window per compilare il JSON storico $BUILD_MESSAGES_FILE). Se lo storico è assente o vuoto, lo inizializza inserendo un record utente con il prompt corrente. Dispaccia la costruzione del payload del file JSON via build_payload_from_vars. Verifica e richiede la chiave API per il provider e avvia l'interrogazione dell'API (eseguendo la chiamata in tempo reale se $STREAM_MODE è attivo, oppure richiamando la routine perform_request_once). 2. Chat Interattiva ($CHAT_MODE attivo): Previene l'avvio se l'input non è collegato a un terminale reale (TTY). Stampa l'intestazione di avvio della chat e apre un ciclo continuo di lettura: Acquisisce la riga inserita dall'utente, la pulisce e la accumula nella memoria storica della conversazione corrente. Se è configurata una sessione attiva, convalida l'ID, inizializza la directory e la finestra dei messaggi, e compila il file JSON di messaggi storici $BUILD_MESSAGES_FILE tramite l'estensione del Session Engine o la routine legacy. Compila il payload per il provider tramite build_payload_from_vars. Controlla la chiave API e invoca la chiamata in tempo reale streaming o sincrona. Il ciclo si ripete fino alla cattura del segnale di fine trasmissione (Ctrl+D). 3. Esecuzione Standard di Singolo Prompt (Streaming o Sincrona): Se è definita una sessione ($SESSION_ID), esegue la validazione rigorosa dell'ID, determina la finestra di recupero, e compila il file storico dei messaggi $BUILD_MESSAGES_FILE interrogando il Session Engine o richiamando la routine legacy di lettura NDJSON. Compila il payload del file JSON via build_payload_from_vars. Se si opera in esecuzione reale (non simulata), valida la presenza della chiave API del provider. Se streaming attivo ($STREAM_MODE == 1): Invoca la funzione di streaming dedicatale (call_api_streaming). Se completato con successo: Se vi è una sessione attiva, sanifica ed accoda il messaggio dell'utente al registro storico (via session_engine_append o la legacy session_append). Se la connessione ha generato risposta reale ed il file $RESP contiene dati validi, estrae il testo generato dall'assistente, ne esegue la sanificazione dai ritorni a capo ed accoda in modo sicuro la risposta dell'assistente nel file NDJSON storico della sessione (delegandola al Session Engine o alla legacy session_append). Stampa un carattere a capo e termina l'esecuzione con stato 0. In caso di errore termina con errore di curl. Se sincrona standard ($STREAM_MODE == 0): Stampa i dettagli diagnostici di visualizzazione del prompt e del modello se in terminale interattivo verboso. Invoca la chiamata transazionale via perform_request_once. In caso di successo: Se è configurata una sessione, sanifica e registra il messaggio dell'utente nel file storico della sessione. Se l'esecuzione è reale ed il file di risposta contiene dati, estrae il testo generato dall'assistente, lo ripulisce e registra in modo sicuro la risposta dell'assistente nel file NDJSON di storico della sessione. Esce con stato 0. In caso di fallimento della chiamata, accoda comunque il messaggio utente alla sessione contrassegnandolo come interazione fallita ed interrompe l'esecuzione con codice BASH4LLMERRAPI.
